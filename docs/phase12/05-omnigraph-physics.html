PHASE 12 — NVIDIA OMNIVERSE

Phase 12 · Station 05

고급 — OmniGraph · 물리 · 실시간 데이터 바인딩

확장(04)이 파이썬 코드로 스테이지를 만졌다면, 이 스테이션은 코드를 노드로 대신하는 길을 본다 — OmniGraph다. 노드를 잇는 비주얼 스크립팅으로 계산과 데이터 흐름을 짜고, PhysX로 물리를 얹고, 그 위에 팹 텔레메트리를 USD 속성으로 흘려 넣어 화면을 실시간으로 살아 있게 만든다. Phase 8의 실시간 브리지와 Phase 5의 시뮬레이션이 Omniverse에서 만나는 지점이다. 정확한 개념을 얻는 게 목표이므로, 클릭 단위 튜토리얼이 아니라 무엇이 무엇인지에 집중한다.

OmniGraph — 노드로 짜는 계산·데이터 흐름

OmniGraph는 Omniverse의 비주얼 스크립팅·계산 프레임워크다. 각 노드는 입력·출력 속성을 가진 작은 계산 단위이고, 노드의 출력을 다른 노드의 입력에 선으로 잇는 것이 데이터 흐름이다. Phase 3에서 만난 셰이더 그래프가 "표면의 색"을 노드로 조립했듯, OmniGraph는 "씬의 동작과 데이터 처리"를 노드로 조립한다. 코드로도 할 수 있는 일을 그래프로 여는 이유는 둘이다 — 비개발자도 읽고 고칠 수 있고, 엔진이 어떤 노드를 언제 계산할지를 스스로 최적화할 수 있다. 특히 뒤의 이유가 대규모 씬에서 값을 한다: 수천 개 노드가 얽힌 그래프에서 엔진은 값이 실제로 바뀐 노드와 그 하류만 골라 다시 계산하므로, "매 프레임 전부 다시 도는" 순진한 코드보다 유리해질 수 있다.

두 갈래 — 푸시 그래프 vs 액션 그래프

OmniGraph에는 언제 노드가 계산되는가가 다른 두 갈래가 있다. 이 구분이 핵심이다.

  • 푸시 그래프(push graph) — 데이터가 바뀌면 계산된다. 실행 순서를 잇는 별도의 선 없이, 입력이 더러워지면(dirty) 그에 딸린 노드들이 갱신되는 순수한 데이터 흐름이다. 매 프레임 값을 이어 흐르게 하는 연속적 계산에 맞는다.
  • 액션 그래프(action graph) — 사건이 나면 계산된다. "매 틱"·"키 입력"·"충돌" 같은 이벤트 노드에서 시작해, 데이터 선과 별개로 실행 선(execution)이 노드의 실행 순서를 정한다. 조건 분기·트리거처럼 언제·어떤 순서로가 중요한 로직에 맞는다.

비유하면, 푸시 그래프는 스프레드시트다 — 셀 값을 바꾸면 그에 딸린 수식이 알아서 다시 계산된다. 액션 그래프는 순서도다 — "이 사건이 나면 → 이걸 하고 → 그다음 저걸 한다"는 실행의 줄기가 그림에 그려져 있다.

PhysX — 물리를 얹는다

Omniverse의 물리는 NVIDIA PhysX 엔진이 맡는다. 강체·충돌·조인트부터 차량·연성체·유체까지 시뮬레이션하는데, 핵심 발상은 물리도 USD로 기술한다는 것이다. 어느 프림에 UsdPhysics 스키마(강체·콜라이더·질량 등)를 적용해 두면 — Phase 11에서 본 그 "스키마를 프림에 얹는다" 그대로 — PhysX가 그 선언을 읽어 매 스텝 자세를 계산하고, 결과를 다시 프림의 트랜스폼 속성에 되돌려 쓴다. 그래서 물리 설정은 별도 포맷이 아니라 스테이지 위의 속성이고, 저장·공유·컴포지션이 씬의 나머지와 똑같이 된다.

우리 팹 맥락에서 PhysX가 전면에 나설 일은 많지 않다 — 반송 차량의 움직임은 대개 시뮬레이션이 정한 경로 보간이지 물리 낙하가 아니기 때문이다. 다만 로봇 팔의 관절, 적재물의 안착, 충돌 감지처럼 접촉·역학이 실제로 중요한 국소 장면에서 PhysX가 값을 한다. 개념으로만 쥐고, 필요할 때 스키마를 얹는 도구로 기억해 두면 된다.

"물리도 USD로 기술한다"가 왜 중요한 설계인지 한 번 더 짚자. 만약 물리 설정이 별도의 독점 포맷에 들어 있었다면, 팹 씬을 다른 팀에 넘기거나 레이어로 합성할 때 물리만 따로 챙겨야 했을 것이다. 그러나 UsdPhysics 스키마로 프림에 얹힌 강체·질량·조인트는 씬의 나머지 속성과 똑같이 저장·레퍼런스·오버라이드된다 — Phase 11에서 배운 컴포지션이 물리에도 그대로 적용된다는 뜻이다. 예컨대 기본 팹 레이어에는 콜라이더만 두고, "물리 검증용" 레이어에서만 중력·강체를 켜는 식의 비파괴 실험이 가능하다.

실시간 데이터 바인딩 — 텔레메트리 → USD 속성 → 화면

이 스테이션의 종착점이자 우리 커리큘럼이 겨눠 온 그림이다. 외부에서 들어오는 팹 텔레메트리(시뮬레이션 출력이든 실물 피드든)를 받아 대응하는 USD 속성에 쓰면, RTX 뷰포트가 그 변화를 즉시 렌더한다 — 이것이 데이터 바인딩이다. Phase 8에서 다듬은 것들이 그대로 여기서 값을 한다: NDJSON 이벤트를 받고, 플레이아웃 지연으로 지터를 흡수하고, 스냅샷으로 재동기화하는 그 브리지의 출력을, 이번엔 Unity 뷰가 아니라 USD 스테이지에 꽂는다.

쓰는 경로는 두 갈래다 — (1) 04에서 본 것처럼 Kit 확장이 파이썬으로 매 틱 속성을 갱신하거나, (2) OmniGraph 노드가 소스를 읽어 속성에 흘려 넣는다. 어느 쪽이든 "상태를 USD 속성에 쓴다 → 엔진이 그린다"는 원리는 같다. 확장의 심장(04)과 OmniGraph의 데이터 흐름(이 스테이션)이 같은 문 앞에서 만난다.

두 경로의 선택 기준을 실무 감각으로 정리하면 이렇다. Kit 확장(파이썬)은 복잡한 조건 분기, 외부 라이브러리 호출, Phase 8식 재동기화 로직처럼 제대로 된 코드가 필요한 곳에 강하다. OmniGraph 노드는 "이 텔레메트리 값을 저 프림 색에 잇는다" 같은 단순·시각적 바인딩을, 비개발자도 그래프를 열어 선을 다시 이어 고칠 수 있게 하는 데 강하다. 실제 팹 트윈에서는 둘을 섞는다 — 무거운 수신·버퍼링은 확장이 맡고, 마지막 "값 → 화면 속성" 매핑은 OmniGraph로 노출해 운영자가 조정하게 두는 식이다.

DIGITAL TWIN 관점

Phase 5는 시뮬레이션이 시간을 만들었고, Phase 8은 실물 피드가 시간을 만드는 트윈으로 그 소스 자리를 바꿨다. 이 스테이션은 그 출력이 도착하는 화면 쪽을 Omniverse로 갈아 끼운다 — 소스가 만든 상태를 USD 속성에 바인딩하면 RTX가 실시간으로 그린다. "누가 시간을 만드는가"는 소스가 정하고, "얼마나 실감 나게 보이는가"는 이 렌더 쪽이 정한다. 트윈의 두 반쪽이 여기서 맞물린다.

핵심 확인 질문

푸시 그래프와 액션 그래프를 가르는 한 가지 질문은?

"노드는 언제 계산되는가"다. 푸시 그래프는 입력 데이터가 바뀌면 계산되는 순수 데이터 흐름이고, 액션 그래프는 이벤트가 나면 실행 선을 따라 순서대로 계산된다. 값이 흐르느냐, 실행이 흐르느냐의 차이다.

액션 그래프에만 있는 "실행 선"은 무엇을 정하나?

노드가 어떤 순서로 실행되는지를 정한다. 데이터 선(값의 연결)과 별개로, 이벤트 노드에서 시작하는 실행 선이 "이걸 한 뒤 저걸 한다"는 제어 흐름을 그린다. 조건 분기·트리거가 여기서 표현된다.

Omniverse에서 물리 설정이 별도 파일이 아니라 스테이지 속성인 이유는?

물리를 UsdPhysics 스키마로 프림에 적용해 기술하기 때문이다. 강체·콜라이더·질량이 프림의 속성으로 얹히고, PhysX가 그 선언을 읽어 시뮬레이션한 결과를 다시 트랜스폼 속성에 되돌려 쓴다. 그래서 저장·공유·컴포지션이 씬의 나머지와 똑같이 동작한다.

외부 텔레메트리가 화면에 반영되는 경로를 한 줄로?

텔레메트리 → USD 속성에 쓰기 → RTX가 렌더. 확장(파이썬)이든 OmniGraph 노드든 상태를 대응하는 프림 속성에 쓰기만 하면, 스테이지 변화를 엔진이 다시 그린다. 이것이 실시간 데이터 바인딩이다.

Phase 8 브리지의 어떤 조각들이 이 바인딩에서 그대로 쓰이나?

NDJSON 이벤트 수신, 플레이아웃 지연·지터 버퍼로 네트워크 흔들림 흡수, 스냅샷+델타로 재동기화 — 트윈 브리지의 출력이 도착하는 곳만 Unity 뷰에서 USD 스테이지로 바뀔 뿐, 실시간 처리의 뼈대는 그대로다.

개념을 더 파려면 NVIDIA 공식 문서를 본다 — OmniGraph 문서, Omniverse 물리(PhysX) 문서, USD 물리 스키마는 OpenUSD의 UsdPhysics를 참고한다.
