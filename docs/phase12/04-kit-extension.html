PHASE 12 — NVIDIA OMNIVERSE

Phase 12 · Station 04

Kit 확장 개발 — Python · omni.ui · 스테이지 조작

Omniverse의 앱들(USD Composer 등)은 통짜 프로그램이 아니라 확장(extension)을 레고처럼 끼워 조립한 것이다. Kit은 그 조립의 런타임이자 SDK다. 이 스테이션은 가장 작은 확장 하나 — extension.toml 한 장과 파이썬 진입 클래스 하나 — 를 읽고, 그것이 어떻게 창을 띄우고 USD 스테이지를 만지는지를 짚는다. 여기까지가 팹 상태를 화면에 그리는 우리 확장의 뼈대다.

Kit = 확장을 조립하는 프레임워크

Kit의 발상은 하나다 — 기능은 전부 확장이고, 앱은 확장 목록이다. 뷰포트, 콘텐츠 브라우저, 레이어 창, 물리 엔진까지 모두 각각의 확장이며, 어떤 앱(.kit 파일)은 "이 확장들을 켜라"는 목록에 지나지 않는다. 그래서 새 기능을 넣는 일은 앱을 고치는 게 아니라 확장 하나를 더 만들어 켜는 일이 된다. 확장은 파이썬만으로도, C++로도, 둘을 섞어도 쓸 수 있는데, 우리는 파이썬 확장만 다룬다.

확장 하나는 폴더 하나다. 그 안에 무엇인지 선언하는 파일(extension.toml)과 무엇을 하는지 담은 파이썬 모듈이 들어간다. 이 둘의 관계가 확장의 전부다. 규모가 커져도 이 골격은 변하지 않는다 — 큰 확장은 파이썬 모듈이 여러 파일로 나뉘고 의존이 늘어날 뿐, "선언 파일 + 진입 클래스"라는 뼈대는 그대로다. 그래서 가장 작은 확장을 완전히 이해하면 큰 확장도 같은 눈으로 읽힌다.

extension.toml — 확장을 선언한다

config/extension.toml은 이 확장이 무엇이고, 무엇에 의존하며, Kit이 어느 파이썬 모듈을 로드해야 하는지를 적은 명세다. 핵심 세 부분만 보면 된다 — [package](정체성), [dependencies](다른 확장 의존), [[python.module]](로드할 모듈 이름). 

[package]
version = "0.1.0"
title = "DT1 FabSim Viz"
description = "팹 상태를 USD 스테이지에 반영해 시각화하는 확장"
category = "Simulation"

# 이 확장이 켜질 때 함께 켜져야 하는 다른 확장들
[dependencies]
"omni.usd" = {}
"omni.ui" = {}

# Kit이 로드할 파이썬 모듈 — 이 모듈 안의 IExt를 찾아 진입한다
[[python.module]]
name = "dt1.fabsim.viz"

[dependencies]가 중요하다. 확장은 홀로 서지 않고 다른 확장(여기선 USD 접근용 omni.usd, UI용 omni.ui) 위에 선다. Kit은 이 선언을 읽어 의존 확장을 먼저 켠 뒤 내 확장을 켠다. 의존을 빠뜨리면 import는 되더라도 런타임에 필요한 서비스가 없어 무너진다.

진입점 — omni.ext.IExt

파이썬 모듈 안에는 omni.ext.IExt를 상속한 클래스 하나가 있으면 된다. Kit은 확장을 켤 때 on_startup(ext_id)를, 끌 때 on_shutdown()을 부른다. 이 두 콜백이 확장의 생명 주기다 — 시작에서 자원을 만들고, 종료에서 반드시 되돌린다. Kit은 개발 중 확장을 껐다 켜는 핫 리로드를 지원하므로, on_shutdown에서 창·구독·핸들을 확실히 놓아 주지 않으면 리로드마다 유령 창과 누수가 쌓인다. 

import omni.ext
import omni.ui as ui


class FabSimVizExtension(omni.ext.IExt):
    def on_startup(self, ext_id: str):
        self._window = ui.Window("DT1 FabSim", width=320, height=180)
        with self._window.frame:
            with ui.VStack(spacing=6):
                self._status = ui.Label("스테이지 대기 중…")
                ui.Button("팹 상태 새로고침", clicked_fn=self._refresh)

    def on_shutdown(self):
        # 시작에서 만든 것은 종료에서 되돌린다 — 핫 리로드 누수 방지
        self._window = None

omni.ui — 창과 위젯

omni.ui는 Kit의 즉시 모드(immediate-mode) UI 프레임워크다. ui.Window로 도킹 가능한 창을 열고, ui.VStack/ui.HStack으로 세로·가로 레이아웃을 잡은 뒤 ui.Label·ui.Button 같은 위젯을 파이썬 with 블록으로 중첩해 쌓는다. 버튼의 clicked_fn에 함수를 넘기면 그게 콜백이다. Phase 8에서 만든 UI Toolkit 대시보드가 UXML·USS로 구조와 스타일을 갈랐다면, omni.ui는 그 구조를 파이썬 코드로 직접 쌓는다는 점이 다르다.

"즉시 모드"라는 말이 낯설 수 있는데, 감각적으로는 레이아웃을 데이터가 아니라 코드 실행 순서로 그린다는 뜻이다. with ui.VStack(): 블록 안에서 위젯을 만드는 순서가 곧 화면에 쌓이는 순서가 된다. 그래서 조건에 따라 다른 위젯을 그리는 UI를 if 문 하나로 자연스럽게 짤 수 있다 — 팹 뷰어라면 "차량이 선택됐을 때만 상세 패널을 그린다" 같은 로직이 그대로 파이썬 흐름으로 표현된다. 값이 바뀔 때마다 화면을 갱신하려면 ui.Label 같은 위젯의 핸들을 self에 보관해 두었다가 .text를 다시 써 주면 된다 — 아래 _refresh가 정확히 그 패턴이다.

omni.usd — 스테이지를 만진다

확장이 팹을 "그리는" 방법은 화면에 직접 픽셀을 찍는 게 아니라 USD 스테이지를 고치는 것이다 — 스테이지를 고치면 RTX 뷰포트가 알아서 다시 그린다. 현재 열린 스테이지는 omni.usd.get_context().get_stage()로 얻는다. 여기서부터는 Phase 11에서 배운 그 Prim·Property·Path 세계다. 프림을 경로로 찾아 속성 값을 바꾸면 그게 화면에 뜬다. omni.usd.get_context()는 앱이 지금 열어 둔 USD 컨텍스트를 돌려주고, 그로부터 get_stage()로 현재 스테이지를 얻는다 — Phase 11에서 파이썬 스크립트가 Usd.Stage.Open()으로 파일을 직접 열던 것과 달리, 확장은 이미 앱이 화면에 띄워 둔 스테이지에 올라타는 셈이다. 그래서 스테이지가 None일 수 있다(아직 아무 파일도 안 연 상태). 실전 확장은 아래처럼 스테이지 존재부터 확인하고 시작한다. 

import omni.usd
from pxr import UsdGeom, Gf


def _refresh(self):
    stage = omni.usd.get_context().get_stage()
    if stage is None:
        self._status.text = "열린 스테이지가 없습니다"
        return

    # 팹 레이아웃 안의 OHT 비히클 프림을 경로로 집어 위치를 갱신
    prim = stage.GetPrimAtPath("/World/Fab/OHT/vehicle_007")
    if prim.IsValid():
        xform = UsdGeom.XformCommonAPI(prim)
        xform.SetTranslate(Gf.Vec3d(12.5, 0.0, 4.0))
        self._status.text = "vehicle_007 위치 갱신"

이 한 조각이 우리 확장의 심장이다. 시뮬레이션(Phase 5)이나 실물 피드(Phase 8)가 "vehicle_007이 (12.5, 0, 4)로 갔다"고 알려 오면, 확장은 그 값을 대응하는 프림 속성에 쓰기만 하면 된다. 나머지 — 조명, 반사, 그림자 — 는 RTX가 맡는다.

예제 프로젝트 — omniverse-kit-ext

이 스테이션의 코드는 예제 examples/omniverse-kit-ext/에 완성된 형태로 있다. 확장은 exts/dt1.fabsim.viz/ 아래에 config/extension.toml과 dt1/fabsim/viz/extension.py(omni.ui 창 + 스테이지 조작)로 놓여 있고, Phase 11에서 조립한 팹 USD 씬을 Omniverse에서 열어 이 확장으로 상태를 반영하는 흐름을 담았다. 설치·실행은 다음 실습 스테이션(06)에서 직접 밟는다.

DIGITAL TWIN 관점

Phase 8에서 우리는 "같은 뷰를 두 소스가 구동한다"를 IFleetSource 뒤에서 증명했다. Kit 확장은 그 뷰 자리에 Omniverse 스테이지를 꽂는 일이다 — 소스(시뮬/실물)가 만든 상태를 USD 속성에 쓰는 어댑터가 곧 이 확장이다. 뷰가 Unity에서 Omniverse로 바뀌었을 뿐, "상태를 받아 그린다"는 트윈의 골격은 그대로다.

핵심 확인 질문

Kit에서 "앱"과 "확장"의 관계를 한 문장으로?

앱은 확장 목록이다. 기능은 전부 확장으로 존재하고, 앱(.kit)은 "어떤 확장을 켤지"를 고른 조립일 뿐이다. 그래서 기능 추가 = 새 확장을 만들어 켜기가 된다.

extension.toml의 [dependencies]를 빠뜨리면 무슨 일이?

Kit이 의존 확장(예: omni.usd·omni.ui)을 먼저 켜 주지 못한다. import는 통과하더라도 런타임에 그 확장이 제공하는 서비스가 없어 스테이지 접근이나 창 생성이 무너진다. 의존은 "먼저 켜져 있어야 할 것"의 선언이다.

on_shutdown에서 자원을 놓아 주지 않으면 왜 문제가 되나?

Kit은 개발 중 확장을 껐다 켜는 핫 리로드를 한다. on_startup에서 만든 창·구독·핸들을 on_shutdown에서 되돌리지 않으면 리로드마다 유령 창과 누수가 누적된다. 생명 주기는 시작–종료가 짝을 이뤄야 한다.

확장이 팹을 "그리는" 실제 메커니즘은?

화면에 직접 픽셀을 찍는 게 아니라 omni.usd.get_context().get_stage()로 얻은 USD 스테이지의 프림 속성을 바꾼다. 속성이 바뀌면 RTX 뷰포트가 그 변화를 다시 렌더한다. 확장은 상태를 스테이지에 쓰기만 하고 렌더는 엔진이 맡는다.

omni.ui의 창 구성이 Phase 8의 UI Toolkit과 다른 점은?

UI Toolkit은 구조(UXML)와 스타일(USS)을 파일로 갈라 선언한다. omni.ui는 with 블록으로 VStack·Button 같은 위젯을 파이썬 코드에서 직접 쌓는 즉시 모드 방식이다. 목적(운영자 UI)은 같지만 조립하는 손놀림이 다르다.

더 깊은 내용은 NVIDIA 공식 문서를 본다 — Kit 매뉴얼(확장 개발), omni.ui 문서.
