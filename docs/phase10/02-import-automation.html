PHASE 10 — 실자산 파이프라인

Phase 10 · Station 02

임포트 자동화 — AssetPostprocessor

스테이션 1에서 손으로 하던 교정(스케일·재질·콜라이더·플래그)을, 자산이 수백 개가 되면 사람이 못 한다. 임포트 순간에 규칙을 자동 적용하는 AssetPostprocessor로 옮긴다.

임포트 순간에 끼어들기 — AssetPostprocessor란

AssetPostprocessor는 자산 임포트 때 Unity가 호출하는 훅 클래스다. 상속해 OnPreprocess*/OnPostprocess*를 정의하면 등록 없이 (리플렉션 발견) 파이프라인에 참여한다. "임포트는 이벤트다" — 그 이벤트에 규칙을 건다.

훅 라이프사이클 — 언제 무엇이 불리나

모델 임포트는 순서가 있다: OnPreprocessModel(메시 전 — ModelImporter 설정 교정) → 재질 배정 → OnPostprocessModel(root)(생성된 GameObject 트리 후처리) → 전역 OnPostprocessAllAssets. 텍스처는 OnPreprocessTexture.

훅시점넘어오는 것대표 용도
OnPreprocessModel메시 생성 전assetImporter as ModelImporter스케일·Read/Write·노멀 설정
OnPostprocessModel트리 생성 후GameObject root콜라이더·정적 플래그·재질 리맵
OnPreprocessTexture텍스처 임포트 전TextureImporter압축·sRGB·mipmap
OnPostprocessAllAssets임포트 배치 후경로 목록전역 후처리·인덱싱

폴더 규칙으로 대상 좁히기

자동화는 벤더 자산에만 걸어야 한다 — 프로젝트 고유 에셋까지 건드리면 재앙이다. assetPath.StartsWith("Assets/Vendor/")로 스코프를 좁힌다. 이 한 줄이 자동화의 안전벨트다. 

public sealed class CadImportPostprocessor : AssetPostprocessor
{
    private const string VendorRoot = "Assets/Vendor/";

    private void OnPreprocessModel()
    {
        if (!assetPath.StartsWith(VendorRoot)) { return; }  // 벤더 자산만
        var importer = assetImporter as ModelImporter;
        importer.globalScale = 0.01f;      // mm → m
        importer.isReadable = false;       // 메모리 2배 방지
        // 노멀·용접·탄젠트 설정...
    }

    private void OnPostprocessModel(GameObject root)
    {
        if (!assetPath.StartsWith(VendorRoot)) { return; }
        // MeshCollider 부착, SetStaticEditorFlags, sharedMaterial 리맵...
    }
}

무엇을 자동화하나

  • 스케일 — ModelImporter.globalScale/useFileScale.
  • 재질 리맵 — 이름 규칙으로 프로젝트 공유 머티리얼에 연결 (sharedMaterial — 복제하면 배칭이 깨지므로).
  • MeshCollider — OnPostprocessModel에서 정적 지오메트리에.
  • 레이어·스태틱 플래그 — GameObjectUtility.SetStaticEditorFlags.

임포트 리포트 — EditorWindow로 품질 보기

EditorWindow로 만든 ImportReportWindow가 선택 모델의 정점 수 / 서브메시 수 / 재질 수 / 텍스처 메모리 추정을 요약한다. 통합 효과를 숫자로 확인하는 대시보드 — Phase 6의 "측정으로 증명한다" 태도가 자산 파이프라인에서 되풀이된다.

자동화의 함정과 예의

무한 재임포트를 만들지 말고, 규칙이 넓으면 남의 자산까지 망가뜨린다(폴더 스코프 필수). 팀 공유 시 결정성(같은 입력 → 같은 결과)을 지키고, .meta를 버전 관리에 포함한다 — 임포트 설정은 코드만큼 중요한 자산이다.

핵심 확인 질문

AssetPostprocessor는 어떻게 "등록" 없이 임포트에 참여하나?

Unity가 리플렉션으로 AssetPostprocessor를 상속한 클래스를 찾아 자동으로 훅을 호출한다. 명시적 등록·구독이 필요 없다 — 상속하고 훅 메서드를 정의하면 파이프라인에 끼어든다.

스케일·Read/Write 교정은 어느 훅, 콜라이더 부착은 어느 훅인가?

스케일·Read/Write 같은 임포트 설정은 OnPreprocessModel에서 (ModelImporter를 통해) 메시 생성 전에 바꾼다. 콜라이더 부착·정적 플래그는 생성된 트리를 만지는 OnPostprocessModel(root)에서 한다.

재질 리맵을 material이 아니라 sharedMaterial로 하는 이유는?

renderer.material에 접근하면 머티리얼이 복제돼 인스턴스가 생긴다. 복제된 개별 머티리얼은 서로 배칭되지 못해 드로우콜이 늘어난다(Phase 4). 공유를 유지하려면 sharedMaterial로 리맵해야 배칭이 산다.

자동화 규칙을 Assets/Vendor/로 스코프하는 이유는?

규칙이 프로젝트 전체에 걸리면 고유 에셋까지 오염시킨다(엉뚱한 스케일·재질로 망가짐). 벤더 폴더로 좁히면 자동화 대상이 명확하고, 같은 입력에 같은 결과를 내는 결정성도 지킨다.

임포트가 잘 됐는지 정점·서브메시·재질 수로 확인하려면 무엇을 만드나?

EditorWindow 리포트 창(ImportReportWindow)이다. 선택 모델의 메시·렌더러를 순회해 정점 수·서브메시 수·재질 수·텍스처 메모리를 요약하면, 통합 전/후를 숫자로 대조할 수 있다.
