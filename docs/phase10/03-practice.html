PHASE 10 — 실자산 파이프라인

Phase 10 · Station 03 · 실습

실습 — 임포트 자동화 직접 만들기

결함 FBX를 배칭되는 상태로 길들이고, 그 교정을 AssetPostprocessor로 자동화한다. 연장선 전체의 종점 — 읽어서 이해한 파이프라인을 자기 손으로 재구축하면, 실전 트윈에 도착한 것이다.

실습 무대 — FabSim에 벤더 자산이 도착했다

  • 무대: 사용자의 FabSim(Unity 6000.3.8f1 + URP 17.3.0).
  • 결함 FBX: 완성본이 생성해 둔 FabSim-P10-Complete/Assets/Vendor/oht_station_raw.fbx를 FabSim/Assets/Vendor/로 복사해 시작한다(FBX 생성 자체는 완성본의 몫 — 결함 파일만 받아 실습).
  • 정답지: FabSim-P10-Complete의 CadImportPostprocessor· ImportReportWindow.

과정 상세 — 클릭 단위로 따라 만들기

위 마일스톤은 무엇을의 지도다. 여기서는 어떻게를 클릭·입력·코드 단위로 편다. 먼저 스스로 M1~M5를 밟아 보고, 막히거나 결과가 어긋날 때 이 절차를 펼쳐 대조하라 — 값은 완성본 FabSim-P10-Complete의 DefectFbxGenerator.cs(결함 주입)· oht_station_raw.fbx.meta(교정된 임포트 설정)·CadImportPostprocessor.cs(자동화 훅)에서 그대로 뽑은 재현값이라, 따라 하면 정답 상태가 다시 만들어진다. 임포터와 스크립트는 전부 직접 손으로 만진다.

0단계 — 결함 FBX 받아오기

  1. FabSim Assets에서 우클릭 → Create → Folder, 이름 Vendor. 자동화 스코프가 정확히 이 이름(Assets/Vendor/)이므로 철자·대소문자를 그대로 맞춘다.
  2. 완성본의 FabSim-P10-Complete/Assets/Vendor/oht_station_raw.fbx를 탐색기에서 복사해 FabSim의 Assets/Vendor/에 붙여넣는다. Unity로 돌아오면 자동 임포트된다. (FBX 생성은 완성본의 DefectFbxGenerator 몫 — 결함 파일만 받아 길들인다.)

1단계(M1) — 결함을 눈으로 확인

  1. 임포트된 oht_station_raw 프리팹을 Hierarchy로 드래그해 씬에 놓는다. Scene 뷰에서 거대하게 잡히면(F로 프레이밍하면 카메라가 한참 물러난다) 스케일 결함이 눈에 보인 것이다.
  2. 계층을 펼쳐 구조를 확인한다: 의미 없는 빈 부모 두 겹(export_group → node_0) 밑에 부품 10개(part_00…part_09, Cube/Cylinder 교대)가 흩어져 있다.
  3. 메뉴 FabSim → Import Report를 열고 이 모델을 선택해 정점 수·서브메시 수·고유 재질 수를 읽는다.
  4. Window → Analysis → Frame Debugger를 Enable하고, Game 뷰 Stats에서 Batches·SetPass calls를 읽어 완주 체크표의 "전(前)" 칸에 적는다.

2단계(M2) — 임포트 설정 수동 교정 (Model 탭)

  1. Project에서 oht_station_raw.fbx를 선택 → Inspector Model 탭.
  2. Scale Factor를 0.01로. 루트 100배를 상쇄해 정상 크기(1 유닛 = 1 m)로 돌아온다.
  3. Read/Write Enabled 끔 — 런타임 메시 사본을 없애 메모리 2배를 막는다.
  4. Mesh Compression을 Medium, Weld Vertices 켬, Optimize Mesh는 정점·폴리곤 둘 다 켠다.
  5. Tangents를 Calculate Mikktspace로. 값을 다 맞췄으면 Apply.

3단계(M3) — 재질 통합

  1. 같은 FBX의 Inspector Materials 탭으로 이동한다.
  2. 부품별 개별 재질 10개(vendor_mat_00 … 09)를 프로젝트 공유 머티리얼 1~3개로 리맵한다 — Remap 목록의 각 이름 슬롯에 공유 머티리얼을 지정하고 Apply. 여러 이름을 한 머티리얼로 몰면 통합된다.
  3. 다시 FabSim → Import Report로 고유 재질 수가 10 → 1~3으로 준 것을 확인한다.
material 말고 sharedMaterial

리맵은 반드시 공유 머티리얼로 연결한다. 코드에서 renderer.material에 손대면 머티리얼이 복제돼 인스턴스가 생기고, 복제본끼리는 배칭되지 못해 드로우콜이 되레 는다(Phase 4). 재질 수 = 서브메시 = 드로우콜 하한이므로, 통합이 곧 드로우콜 하한을 낮추는 일이다.

4단계(M4) — AssetPostprocessor 훅 채우기

페이지 위쪽 스켈레톤의 두 TODO를 완성본의 훅 본문으로 채운다. 파일은 Assets/Editor/CadImportPostprocessor.cs이고, 저장하면 Unity가 리플렉션으로 발견해 등록 없이 임포트에 끼어든다. OnPostprocessModel이 쓰는 도우미 두 개(UnifyMaterials·AttachCollider)도 같은 클래스에 추가한다. 

private const string VendorRoot = "Assets/Vendor/";
private const float MillimeterToMeter = 0.01f;

private void OnPreprocessModel()
{
    if (!assetPath.StartsWith(VendorRoot)) { return; }
    var importer = assetImporter as ModelImporter;
    if (importer == null) { return; }

    importer.useFileScale = false;
    importer.globalScale = MillimeterToMeter;   // 0.01f — mm→m, 100배 상쇄
    importer.isReadable = false;                // Read/Write off (메모리 2배 방지)
    importer.meshCompression = ModelImporterMeshCompression.Medium;
    importer.weldVertices = true;
    importer.optimizeMeshPolygons = true;
    importer.optimizeMeshVertices = true;
    importer.importTangents = ModelImporterTangents.CalculateMikk;
}

private void OnPostprocessModel(GameObject root)
{
    if (!assetPath.StartsWith(VendorRoot)) { return; }

    foreach (MeshRenderer renderer in root.GetComponentsInChildren<MeshRenderer>())
    {
        UnifyMaterials(renderer);            // 한 렌더러의 서브메시 재질을 [0]으로 통일
        AttachCollider(renderer.gameObject); // MeshCollider 부착
    }

    GameObjectUtility.SetStaticEditorFlags(root,
        StaticEditorFlags.BatchingStatic | StaticEditorFlags.OccluderStatic
            | StaticEditorFlags.OccludeeStatic);
}

private static void UnifyMaterials(MeshRenderer renderer)
{
    Material[] materials = renderer.sharedMaterials;
    if (materials.Length <= 1) { return; }
    for (int i = 1; i < materials.Length; i++) { materials[i] = materials[0]; }
    renderer.sharedMaterials = materials;
}

private static void AttachCollider(GameObject target)
{
    if (target.GetComponent<MeshCollider>() == null) { target.AddComponent<MeshCollider>(); }
}
정답 값 — 스코프·후처리 (CadImportPostprocessor)

스코프 VendorRoot = "Assets/Vendor/" · 스케일 상수 MillimeterToMeter = 0.01f.
정적 플래그 BatchingStatic | OccluderStatic | OccludeeStatic을 루트에 일괄 지정 → 정적 배칭·오클루전 대상.
각 렌더러에 MeshCollider 부착. 재질은 sharedMaterials로만 다뤄 배칭을 지킨다.

이 코드만으로 10→1이 되진 않는다 — 정직한 경계

UnifyMaterials는 한 렌더러 안의 다중 서브메시 재질을 sharedMaterials[0]으로 통일한다. 이 결함 FBX처럼 부품이 저마다 단일 재질 렌더러로 쪼개져 있으면 materials.Length <= 1이라 이 훅은 그냥 지나간다 — 부품 간 재질 통합(10 → 1~3)은 M3의 Materials 탭 리맵이 담당한다. 포스트프로세서가 매 임포트마다 보장하는 것은 스케일·Read/Write·압축·콜라이더·정적 플래그다. 재임포트 검증: Assets/Vendor/의 FBX를 지우고 다시 복사하면 손 안 대도 이 값들이 교정된 채로 들어온다.

5단계(M5) — 배칭 회복을 수치로

  1. 교정·통합·자동화가 끝난 모델을 씬에 놓고, Frame Debugger와 Game 뷰 Stats로 Batches·SetPass calls를 다시 읽어 체크표의 "후(後)" 칸에 적는다.
  2. FabSim → Import Report로 고유 재질 수·서브메시 수가 전/후로 어떻게 줄었는지 대조한다.
  3. 드로우콜이 유의하게 줄었으면 회복이 확인된 것이다 — 근거는 재질 통합으로 서브메시가 줄고, 정적 배칭·SRP Batcher가 남은 드로우를 묶었기 때문이다.
Batches·SetPass 절대값은 지어내지 말 것

완성본에는 Batches·SetPass calls의 절대 수치가 기록돼 있지 않다(체크표가 빈칸인 이유). 이 두 값은 화면 해상도·카메라 구도·SRP Batcher 상태에 따라 달라지므로 자기 환경에서 측정해 채운다. 정답으로 고정된 것은 재질 수(10 → 1~3)와 스케일(100배 → 정상)뿐이다.

연장선 완주 — 실전 트윈 도착

Phase 6의 테스트·신뢰구간에서 시작해 교통 제어·데드락·라우팅(7), 트윈 브리지(8), 대규모 성능(9), 실자산 파이프라인(10)까지 왔다. 이제 시뮬레이션 결과를 책임지고, 차량이 막히고 멈추는 것을 재현하고, 외부 스트림으로 트윈을 구동하고, 수천 대를 60fps로 굴리고, 고객 CAD를 배칭되는 씬으로 길들일 수 있다. DT개발팀 신입의 첫 과제들이 이 다섯 페이즈에 다 있었다.

핵심 확인 질문

자동화를 붙였는지 확인하려면 결함 FBX를 어떻게 다시 넣어 보나?

FBX를 지우고 다시 복사해 재임포트한다. 자동화가 제대로 걸려 있으면 손 안 대도 스케일·재질·플래그가 교정된 상태로 들어온다 — 이것이 "임포트 순간에 규칙이 적용됐다"의 증거다.

OnPreprocessModel과 OnPostprocessModel의 경계는?

OnPreprocessModel은 메시 생성 전에 ModelImporter 설정(스케일·Read/Write·노멀)을 바꾼다. OnPostprocessModel(root)은 생성된 GameObject 트리를 받아 콜라이더·정적 플래그·재질 리맵 같은 후처리를 한다.

재질 리맵 후 드로우콜이 줄어드는 메커니즘을 Phase 4 용어로?

재질 수 = 서브메시 수 = 드로우콜 하한이다. 개별 재질을 공유 머티리얼로 통합하면 서브메시가 줄어 드로우콜이 줄고, 같은 머티리얼끼리 인스턴싱/SRP Batch로 묶여 CPU 상태 준비 비용이 준다.

자동화 규칙을 Assets/Vendor/에 가두지 않으면 무슨 일이?

프로젝트 고유 자산까지 오염된다 — 엉뚱한 스케일·재질·플래그로 내가 만든 에셋이 망가진다. 폴더 스코프가 자동화의 안전벨트다.

전/후 수치를 기록하는 이유는?

최적화는 측정으로 증명하기 때문이다. "빨라진 것 같다"는 증발하지만 "드로우콜 40→3"은 남는다. Phase 4·6에서 세운 이 태도가 자산 파이프라인에서도 그대로다.
