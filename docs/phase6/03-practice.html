PHASE 6 — 신뢰성·검증

Phase 6 · Station 03 · 실습

실습: 테스트와 신뢰구간 직접 붙이기

Station 01에서 테스트와 asmdef를, 02에서 반복·신뢰구간을 읽었다. 이번엔 그 둘을 직접 자기 손으로 붙인다 — 실습 프로젝트 FabSim에 어셈블리를 나눠 테스트를 얹고, 채워지지 않은 스켈레톤을 green으로 만들고, 배차 랩에 신뢰구간을 달고, 이론값이 CI 안에 드는지 확인한다. 읽는 페이지가 아니라 만드는 페이지다.

이 프로젝트의 다른 실습과 같은 구조다 — 직접 만드는 무대가 있고, 막혔을 때 대조할 정답지가 따로 있다. 정답지는 베끼는 곳이 아니라 대조하는 곳이다.

실습 체크리스트

각 단계는 완료 기준(끝났다고 말할 수 있는 조건)과 막히면 볼 곳을 함께 적었다. 순서대로 하나씩 green으로 만든다.

  1. FabSim에 asmdef 직접 구성 — FabSim/Assets/에 Runtime/Editor/Tests 어셈블리를 나눈다(Station 01의 4어셈블리 그대로). DesCore/와 Scripts/가 한 Runtime 어셈블리에 들어가도록 Assets 루트에 Runtime asmdef를 두고, 하위의 Editor/·Tests/ asmdef가 중첩 override로 떨어져 나가게 한다.
    완료 기준: 스크립트를 저장하고 에디터로 돌아가면 자동 재컴파일되어 Console 창에 오류가 없다 + Test Runner 창에 FabSim.Tests.* 어셈블리가 나타난다.
    막히면: 완성본의 Assets/FabSim.Runtime.asmdef, Assets/Editor/FabSim.Editor.asmdef, Assets/Tests/EditMode/FabSim.Tests.EditMode.asmdef, Assets/Tests/PlayMode/FabSim.Tests.PlayMode.asmdef를 그대로 대조.

    막히면: 완성본 examples/phase2-descore/DesCore.Tests/의 같은 이름 테스트를 대조. 완성 예시로 남겨 둔 EventQueueTests.PopOnEmptyThrows가 형식 견본이다.
  3. fleet-dispatch 반복·신뢰구간 확장 — 정책×부하 조합을 시드 교체로 20회 반복해 평균 ± 95% CI를 붙이고, 혼잡 부하에서 정책 쌍 대응 t-검정을 출력하고, warm-up(첫 30분 제외) 전후 차이를 관찰한다.

막힘 대응

테스트가 빨간색이면 순서를 의심하라 — 컴파일이 안 되면 asmdef 참조(references)를, 참조가 맞는데 실패면 Assert의 기대값을, CI가 이론값을 못 감싸면 반복 수·warm-up·시드를 본다. 완성본은 베끼는 곳이 아니라 대조하는 곳이다. 답을 옮겨 적으면 다음 페이즈에서 다시 막힌다.

과정 상세 — 클릭 단위로 어셈블리 나누기

위 체크리스트는 무엇을 하는지의 지도다. 여기서는 어떻게를 클릭·입력 단위로 편다 — 이 절차는 Unity 축(asmdef 분리 + EditMode/PlayMode 테스트를 Test Runner로 돌리기)만 다룬다. 콘솔 과제(DesCore.Tests.Practice · fleet-dispatch · M/M/1)는 위 체크리스트대로 dotnet test로 별도 진행한다. 먼저 스스로 나눠 보고, 막히거나 컴파일이 깨질 때 이 절차를 펼쳐 대조하라 — 값은 완성본 FabSim-P6-Complete에서 그대로 뽑은 재현값이라, 따라 하면 정답 어셈블리 구성이 다시 만들어진다.

전제는 하나: FabSim은 Phase 5까지 끝난 상태다 — Assets/DesCore/(DES 코어), Assets/Scripts/(Sim/ 포함, 런타임 로직), Assets/Editor/(에디터 셋업 스크립트)에 스크립트가 있고 asmdef는 아직 0개라 전부 Unity 기본 어셈블리(Assembly-CSharp · Assembly-CSharp-Editor) 하나로 뭉쳐 컴파일된다. 목표는 이걸 4개 어셈블리로 쪼개, 테스트가 코어만 콕 참조하게 만드는 것이다.

어셈블리 지도 — 무엇을 어디에

핵심 원리 하나면 나머지가 따라온다: Assets 루트에 asmdef를 하나 두면 그 아래 모든 스크립트가 그 어셈블리로 빨려 들어간다. 단, 하위 폴더에 자기 asmdef가 있으면 그 폴더는 부모에서 떨어져 나온다(중첩 override). 그래서 루트 FabSim.Runtime 하나가 DesCore/ · Scripts/를 삼키고, Editor/ · Tests/EditMode/ · Tests/PlayMode/는 각자 asmdef로 카브아웃된다.

1단계 — 루트 Runtime 어셈블리

  1. Project 창에서 Assets 폴더를 연 채(루트) 우클릭 → Create → Scripting → Assembly Definition. 생성된 파일 이름을 FabSim.Runtime으로 한다.
  2. Inspector에서 Name이 FabSim.Runtime인지 확인한다. 나머지는 기본값 그대로 둔다 — Auto Referenced ✓, Override References ✗, Any Platform ✓, References 비움. 하단 Apply.
  3. Apply 순간 DesCore/와 Scripts/의 모든 스크립트가 FabSim.Runtime으로 재컴파일된다. 이 시점엔 Editor/도 아직 여기 딸려 들어가 컴파일 에러가 난다 — 다음 단계에서 떼어낸다.
지금 뜨는 빨간 에러는 정상이다

루트에 Runtime asmdef를 놓는 순간 Editor/FabSimSetup.cs가 using UnityEditor;를 쓰는데 런타임 어셈블리엔 UnityEditor가 없어 The type or namespace name 'UnityEditor' could not be found가 뜬다. 이건 실패가 아니라 2단계가 필요하다는 신호다. asmdef가 생기면 "Editor" 폴더의 자동 에디터 전용 취급이 사라지므로, 에디터 코드는 반드시 별도 Editor asmdef로 빼야 한다.

2단계 — Editor 어셈블리 카브아웃

  1. Assets/Editor 폴더를 연 채 우클릭 → Create → Scripting → Assembly Definition, 이름 FabSim.Editor.
  2. Inspector Platforms에서 Any Platform 체크를 끄고 목록에서 Editor 하나만 켠다.
  3. Assembly Definition References에 +로 셋을 추가한다: FabSim.Runtime, Unity.RenderPipelines.Universal.Runtime, Unity.RenderPipelines.Core.Runtime. URP 둘은 FabSimSetup · P5SceneBuilder가 UniversalRenderPipelineAsset 같은 URP 타입을 쓰기 때문이다.
  4. Apply. 1단계의 빨간 에러가 사라지고 컴파일이 클린해진다.

3단계 — EditMode 테스트 어셈블리

  1. Project에서 Assets/Tests → 그 아래 EditMode 폴더를 만든다(우클릭 → Create → Folder).
  2. Tests/EditMode를 연 채 우클릭 → Create → Scripting → Assembly Definition, 이름 FabSim.Tests.EditMode.
  3. Inspector에서: Auto Referenced 체크를 끈다. Override References를 켠다(그래야 아래 Assembly References 칸이 열린다). Platforms → Any Platform 끄고 Editor만 켠다.
  4. Define Constraints에 UNITY_INCLUDE_TESTS를 추가한다 — 테스트 코드가 일반 빌드에 섞이지 않게 막는 방어벽이다.
  5. Assembly Definition References: FabSim.Runtime, UnityEngine.TestRunner, UnityEditor.TestRunner.
  6. Assembly References(Override References를 켜야 보임): 드롭다운에서 nunit.framework.dll을 고른다. Apply.
TestRunner와 nunit은 들어가는 칸이 다르다

UnityEngine.TestRunner · UnityEditor.TestRunner는 asmdef라 Assembly Definition References에, nunit.framework.dll은 precompiled DLL이라 Assembly References에 들어간다. nunit을 위쪽 asmdef 칸에서 찾으면 목록에 안 나온다 — 칸을 헷갈리면 [Test] · Assert가 "찾을 수 없음"으로 빨갛게 뜬다.

4단계 — PlayMode 테스트 어셈블리

3단계와 거의 같되 딱 하나가 다르다 — 플랫폼. Assets/Tests/PlayMode 폴더를 만들고 FabSim.Tests.PlayMode asmdef를 만든 뒤, Auto Referenced 끄기 · Override References 켜기 · Define Constraints UNITY_INCLUDE_TESTS · References (FabSim.Runtime · UnityEngine.TestRunner · UnityEditor.TestRunner) · Assembly References nunit.framework.dll까지 EditMode와 똑같이 채운다.

  • Platforms는 Any Platform을 켠 채로 둔다 — Editor로 제한하지 않는다.
정답 값 — 네 asmdef의 필드 (완성본 그대로)

FabSim.Runtime · Auto Referenced ✓ · Override References ✗ · Any Platform ✓ · References 없음
FabSim.Editor · Auto Referenced ✓ · Override References ✗ · Platform Editor · Refs FabSim.Runtime + URP 둘
FabSim.Tests.EditMode · Auto Referenced ✗ · Override References ✓ · Platform Editor · Define UNITY_INCLUDE_TESTS · Refs FabSim.Runtime + TestRunner 둘 · Assembly Ref nunit.framework.dll
FabSim.Tests.PlayMode · 위와 동일하되 Platform Any Platform ✓

왜 PlayMode만 Any Platform인가

EditMode 테스트는 에디터에서만 도는 논리 검증이라 Editor 플랫폼이면 충분하다. PlayMode 테스트는 프레임을 실제로 흘려 런타임 동작을 보므로, 필요하면 빌드된 플레이어에서도 돌 수 있어야 한다 — 그래서 Any Platform으로 열어 둔다. UNITY_INCLUDE_TESTS 제약이 평소 빌드에서는 테스트를 빼 주므로 열어 둬도 안전하다.

5단계 — 테스트 스크립트 배치

각 테스트 폴더에 스크립트를 만든다(우클릭 → Create → Scripting에서 C# 스크립트). 배치가 핵심이다 — 파일이 어느 asmdef 폴더 안에 있느냐로 EditMode/PlayMode가 갈린다. 아래가 완성본의 5개 테스트 구성이며, 각 메서드의 본문은 완성본 .cs를 대조해 옮긴다(로직은 P5 코어에 대한 검증이라 여기서 다시 유도하지 않는다).

EditMode 파일: namespace FabSim.Tests.EditMode · using NUnit.Framework; + using FabSim.Sim;(둘 다) + using DesCore;(Dispatch만) · 메서드마다 [Test], 반환형 void.

6단계 — Test Runner로 green 만들기

  1. Window → General → Test Runner를 연다.
  2. EditMode 탭에서 트리를 펼치면 FabSim.Tests.EditMode 아래 RailGraphTests(3) · DispatchSelectionTests(1)가 보인다. Run All → 4/4 green.
  3. PlayMode 탭으로 바꾼다. FabSim.Tests.PlayMode 아래 TimeScaleBridgeTests(1). Run All → 에디터가 잠깐 Play로 들어갔다 나오며 1/1 green.
  4. 다섯 개가 전부 초록이면 완성본과 같은 상태다. 하나라도 빨가면 아래 함정을 순서대로 짚는다.
빨간 테스트, 순서대로 의심

① 트리에 어셈블리가 안 보임 → 컴파일 에러가 남아 있다(Console부터 클린으로). ② [Test] · Assert가 빨감 → nunit이 Assembly References에 없다(3단계). ③ FabModel · RailGraph · FabSimulation이 "찾을 수 없음" → FabSim.Runtime 참조 누락. ④ PlayMode 테스트가 안 돎/멈춤 → 반환형이 IEnumerator가 아니거나 [UnityTest]가 아니다. ⑤ EditMode 탭에 PlayMode 테스트가 섞여 보임 → 파일을 엉뚱한 폴더에 뒀다(파일 위치 = 어셈블리).

이 실습이 "끝났다"고 말할 수 있는 완료 기준을 미리 못 박는다. 넷이 모두 참이면 Phase 6을 통과한 것이다.

  • DesCore.Tests.Practice가 6/6 통과 — 스텁 5개를 모두 채웠다.
  • fleet-dispatch가 CI·대응 비교를 출력 — 정책별 평균 ± CI와 유의 판정이 찍힌다.

핵심 확인 질문

FabSim에 asmdef를 넣을 때 DesCore/는 어느 어셈블리에 들어가야 하며, 왜인가?

Runtime 어셈블리에, Scripts/와 함께 들어가야 한다. 테스트가 코어(DesCore)를 참조해 검증해야 하므로 코어가 Runtime에 있어야 하고, DesCore/와 Scripts/가 형제 폴더라 공통 상위인 Assets 루트에 Runtime asmdef를 두면 둘이 한 어셈블리로 묶인다.

스켈레톤 테스트가 처음에 5개나 실패하도록 만든 이유는?

Assert.Fail("TODO") 스텁이 들어 있기 때문이다. 이는 의도된 설계다 — 실행하면 "5 실패 / 1 통과"가 나와 미완성이 수치로 드러난다. 채워 나갈수록 통과 수가 늘어 진척이 눈에 보이고, 다 채우면 6/6이 된다.

관측 CI가 이론 Wq를 감싸지 못하면 무엇을 먼저 의심하는가?

① warm-up 미제거(초기 과도가 평균을 끌어내림), ② 반복 수 부족(CI가 불안정), ③ ρ 과대(수렴이 느려 손님 수가 모자람), ④ 시드 문제 순으로 본다. 대개는 손님 수·반복 수를 늘리거나 warm-up 절단을 제대로 하면 이론값이 CI 안으로 들어온다.

warm-up on/off로 평균대기 수치가 달라지는 이유는?

시뮬레이션이 빈 큐에서 출발하므로 초기 구간의 대기가 정상 부하보다 낮다. 이 구간을 포함하면 평균이 정상 상태보다 낮게 편향된다. 다만 8시간 교대처럼 종료형이면 초기 과도가 전체에서 차지하는 비중이 작아 차이가 미미하고, 끝없이 도는 M/M/1(정상 상태형)에서는 앞 5,000명을 버리지 않으면 편향이 크게 나타난다.

대응(paired) 비교가 독립 비교보다 적은 반복으로 유의성을 얻는 이유는?

같은 명령 스트림(공통 난수)을 두 정책에 흘리면 우연이 상쇄되고, 반복별 차이 dr의 분산이 크게 줄어든다. 분산이 작을수록 같은 반복 수로도 t 값이 커져 유의성을 더 쉽게 얻는다. 독립 비교는 두 실행의 우연이 섞여 분산이 커서 더 많은 반복이 필요하다.
