PHASE 11 — OPENUSD

Phase 11 · Station 06

고급 — 대규모 씬 성능·페이로드/인스턴싱·툴체인

팹 씬은 결국 수천 대로 자란다. 그 규모에서 USD를 여는 시간, 도는 메모리, 순회 비용이 병목이 된다. 이 스테이션은 그 규모를 견디게 하는 도구들 — 페이로드로 지연 로드, 인스턴싱 두 갈래, 플랫튼의 트레이드오프, 드로우 모드, 그리고 순회 최적화와 커맨드라인 툴체인을 다룬다. Phase 9의 대규모 성능을 씬 데이터 층에서 되짚는 셈이다.

페이로드 — 무거운 것을 나중에 켠다

03에서 references와 payloads를 나란히 봤다. 규모가 커지면 이 둘의 차이가 성능을 가른다. reference는 항상 합성·로드되지만, payload는 "로드 경계"다 — 스테이지를 열 때 켜지 않고 미뤄 둘 수 있다. 팹 전체를 열되 벤더 OHT의 무거운 하이폴리 메시는 페이로드로 걸어 두면, 필요한 구역만 Load()하고 나머지는 Unload()로 꺼 둔 채 구조만 순회한다.

페이로드와 active·visibility를 헷갈리지 않는 것이 중요하다. 셋 다 "안 보이게/안 돌게" 만들 수 있지만 비용을 아끼는 지점이 다르다. visibility는 렌더에서만 빼되 데이터는 이미 메모리에 올라와 있고, active=false는 합성 결과에서 가지를 접지만 역시 씬을 여는 시점엔 파일을 읽는다. 오직 payload만이 "디스크에서 읽는 것 자체를 미룬다" — 그래서 열기 시간과 메모리를 실제로 아끼는 유일한 손잡이다. 팹이 수천 대로 자랄 때 오픈 시간이 폭발하는 병목의 첫 처방이 "무거운 지오를 payload로 감싸고 LoadNone으로 열기"인 이유가 여기 있다. 

from pxr import Usd

# 무거운 씬은 페이로드를 끈 채로 연다 — 구조만, 메시는 나중에
stage = Usd.Stage.Open("fab_full.usda", load=Usd.Stage.LoadNone)

bay_a = stage.GetPrimAtPath("/World/Bay_A")
bay_a.Load()          # 이 구역만 켠다(Unload()로 다시 끈다)

인스턴싱 두 갈래 — PointInstancer vs instanceable

"같은 것 수천 개"를 USD로 가볍게 두는 길은 둘이고, 목적이 다르다.

축PointInstancerinstanceable + prototypes
방식인스턴서 한 Prim이 위치/인덱스 배열로 뿌림Prim에 instanceable=true → 공유 프로토타입(네이티브 인스턴싱)
인스턴스 정체개별 Prim 없음(배열 원소)각 인스턴스가 Prim 경로를 가짐(인스턴스 프록시)
알맞은 것동일 OHT 수백 대처럼 점군형 대량 배치구조를 가진 자산(로드포트+장비)을 여러 번 참조해 배치
개별 편집어려움(배열이라)프로토타입은 공유·불변, 오버라이드는 인스턴스 밖에서

instanceable은 reference 등으로 같은 내용을 여러 번 들여온 Prim을 공유 프로토타입 하나로 접어, 씬 그래프는 여러 개로 보이되 메모리·순회 비용은 한 벌만 치르게 한다. 규모 전략의 실전은 "완전 동일한 대량 → PointInstancer, 구조 있는 반복 자산 → instanceable"로 나눠 쓰는 것이다.

핵심 차이를 한 문장으로 못 박으면 — PointInstancer는 인스턴스가 "데이터"(위치 배열의 원소)이고, instanceable은 인스턴스가 "구조"(각자 Path를 가진 인스턴스 프록시)다. 그래서 관제 화면에서 "이 장비 클릭 → 경로 하이라이트"처럼 개별 선택·상호작용이 필요하면 각 인스턴스가 Path로 잡히는 instanceable이 유리하고, "레일 위 수백 캐리어를 그냥 빽빽이 보여만 주면 되는" 순수 대량 표시에는 PointInstancer가 가볍다. 둘 다 프로토타입은 한 벌만 메모리에 상주시키므로 "같은 것 수천 개"의 메모리 폭발을 막는다는 목적은 공유한다 — 다만 인스턴스에 정체성을 얼마나 줄지가 갈림길이다. Phase 4의 GPU Instancing과 Phase 9의 대량 Transform 갱신이 런타임에서 하던 절약을, 여기서는 씬 데이터 모델이 미리 구조로 담아 두는 셈이다.

플랫튼 — 합성을 접어 하나로

03의 컴포지션은 열 때마다 여러 레이어를 규칙(LIVRPS)대로 합성한다. 이 합성은 편집 유연성을 주지만 여는 비용이 든다. UsdStage.Flatten()(또는 usdcat --flatten)은 합성 결과를 단일 레이어로 구워 낸다 — 여는 속도는 빨라지고 배포는 단순해지지만, 레이어 구조와 편집 가능성을 잃고 파일이 커진다. 최종 납품·렌더 팜 전달처럼 "더 안 고칠 스냅샷"에는 유리하고, 작업 원본에는 불리하다. Phase 3의 라이트 베이킹과 정확히 같은 트레이드오프다 — 미리 구우면 빠르지만 다시 못 바꾼다.

드로우 모드 · 카드 — 먼 것은 판때기로

Phase 4의 LOD를 씬 데이터 층에서 하는 장치가 draw mode다. UsdGeomModelAPI를 적용한 모델 Prim에 model:drawMode를 주면, 먼 거리에서는 지오메트리 전체 대신 바운딩 박스(bounds)나 텍스처를 입힌 판때기 (cards)로 대체해 그린다. 수천 대의 원거리 OHT를 풀 메시로 그리지 않고 카드로 눌러, 씬을 열지 않고도(페이로드 언로드 상태) 위치감을 유지한다.

DIGITAL TWIN 관점

관제 탑다운 뷰(직교 투영)에서 팹 전체를 한눈에 볼 때, 개별 장비의 하이폴리 형상은 대부분 불필요하다. 페이로드 언로드 + 카드 드로우 모드를 조합하면 수천 대 규모의 트윈을 가볍게 조망하다가, 특정 구역을 클릭해 다가가면 그 구역만 Load()해 실체를 드러낸다 — Phase 8 대시보드의 "전체 조망 ↔ 상세 진입" 흐름을 USD가 데이터로 뒷받침한다.

순회 최적화 — Usd.PrimRange와 프레디킷

수천 Prim을 매번 전부 도는 것은 낭비다. Usd.PrimRange(파이썬 Stage.Traverse())에 프레디킷을 걸어 필요한 것만 훑는다 — 정의된 것만(UsdPrimIsDefined), 활성만(UsdPrimIsActive), 로드된 것만(UsdPrimIsLoaded), 추상 제외 (!UsdPrimIsAbstract). 기본 순회(Stage.Traverse())가 이미 이 "기본 프레디킷"을 쓴다. 인스턴스 내부까지 보려면 인스턴스 프록시 순회를 따로 켠다. 

from pxr import Usd

# 기본 순회는 정의·활성·로드·비추상만 훑는다(가장 흔한 필요)
for prim in stage.Traverse():
    ...

# 프레디킷을 명시해 좁히기 — 로드된 정의 Prim만
pred = Usd.PrimIsDefined & Usd.PrimIsLoaded
for prim in Usd.PrimRange.Stage(stage, pred):
    ...

USD 툴체인 — 커맨드라인 다섯 자루

규모가 커질수록 GUI보다 커맨드라인 툴이 실무의 손발이 된다. 이 다섯 자루는 USD 설치(옵션 2의 소스 빌드나 NVIDIA 배포)에 함께 딸려 오며 — usd-core(pip)에는 들어 있지 않다 — CI 파이프라인에 그대로 걸어 자동 검증·패키징·썸네일 생성을 무인으로 돌릴 수 있다.

툴하는 일대표 용도
usdcatUSD를 텍스트로 출력·포맷 변환usdcat --flatten a.usd > flat.usda로 합성 결과 확인
usdchecker규칙·usdz 호환성 검사납품 전 유효성 검증(깨진 참조·잘못된 스키마)
usdzip자산을 .usdz 패키지로 묶기텍스처까지 한 파일로 배포
usdrecordHydra로 스테이지를 이미지 렌더썸네일·리포트용 오프라인 캡처
usdview대화형 뷰어(순회·컴포지션·인스턴싱 확인)05 실습의 육안 검증 도구
 
# 합성을 평탄화해 텍스트로 — 무엇이 최종 값이 됐는지 눈으로
usdcat --flatten fab_full.usda -o fab_flat.usda

# 납품 전 유효성 검사
usdchecker fab_flat.usda

# 텍스처까지 한 파일로 패키징
usdzip fab_delivery.usdz fab_flat.usda textures/

툴·API의 정확한 옵션은 OpenUSD 공식 문서에서 확인한다 — USD Toolset, Usd.PrimRange.

핵심 확인 질문

reference와 payload는 규모에서 왜 성능이 갈리나?

reference는 스테이지를 열 때 항상 합성·로드되지만, payload는 로드 경계라 열 때 미뤄 둘 수 있다. 무거운 메시를 payload로 걸면 전체 구조는 알되 데이터는 안 읽으므로, 필요한 구역만 Load()해 메모리·오픈 시간을 아낀다.

열기 시간·메모리를 실제로 아끼려면 payload·visibility·active 중 무엇을 쓰나?

payload다. visibility는 렌더에서만 빼고 데이터는 이미 메모리에 올라와 있고, active=false도 씬을 열 때 파일은 읽는다. payload만이 디스크에서 읽는 것 자체를 미루므로, 무거운 지오를 payload로 감싸고 LoadNone으로 열면 오픈 시간·메모리를 진짜로 절약한다.

PointInstancer와 instanceable을 각각 언제 쓰나?

완전히 동일한 것을 대량으로 점처럼 뿌릴 때(OHT 수백 대)는 PointInstancer가, 구조를 가진 자산을 여러 번 반복해 배치하되 각 인스턴스가 Prim 경로를 갖길 원할 때(장비+로드포트 세트를 곳곳에)는 instanceable 네이티브 인스턴싱이 맞다.

Flatten의 이득과 손실은 무엇이며 무엇에 비유되나?

이득은 여는 속도·배포 단순화, 손실은 레이어 구조·편집 가능성과 커진 파일이다. Phase 3의 라이트 베이킹과 같다 — 미리 구우면 런타임은 빠르지만 다시 못 바꾼다. 그래서 작업 원본이 아니라 최종 스냅샷에 쓴다.

draw mode의 cards가 Phase 4의 어떤 기법에 대응하나?

거리별로 표현을 낮추는 LOD다. 먼 모델을 풀 메시 대신 바운딩 박스나 텍스처 카드로 대체해, 수천 대의 원거리 장비에 폴리곤을 낭비하지 않는다. 페이로드 언로드와 겹쳐 쓰면 "안 읽고도 위치는 보이는" 조망이 된다.

Stage.Traverse()에 프레디킷을 거는 것이 왜 순회 최적화인가?

수천 Prim을 전부 도는 대신 필요한 부분집합만 훑기 때문이다. 정의·활성·로드· 비추상 같은 조건(UsdPrimIsDefined 등)으로 가지치기하면 언로드·비활성 가지를 건너뛴다. 기본 Traverse()가 이미 합리적 기본 프레디킷을 쓴다.
