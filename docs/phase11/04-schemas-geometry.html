PHASE 11 — OPENUSD

Phase 11 · Station 04

스키마·지오메트리·머티리얼 — UsdGeom·PointInstancer·UsdShade

02~03에서 배운 것은 씬의 골격이었다 — Prim은 이름 붙은 그릇이고, Property는 그 안의 값이며, 컴포지션은 그릇들을 합성한다. 이 스테이션은 그 그릇에 의미를 주는 층을 다룬다. "이 Prim은 메시다", "이 표면은 강철이다", "동일한 OHT 300대다" — 이 모든 약속을 지탱하는 것이 스키마다.

스키마 — Prim에 타입과 뜻을 주는 계약

빈 Prim은 typeName이 없는 over거나 이름만 있는 그릇이다. 여기에 "이건 메시다"라는 타입을 붙이고, "여기엔 points·faceVertexIndices 같은 속성이 있다"는 약속을 강제하는 것이 스키마다. USD 스키마는 두 종류로 갈린다 — Prim의 정체(무엇인가)를 정하는 것과, 정체 위에 능력(무엇을 더 할 수 있나)을 덧대는 것.

스키마를 프로그래밍의 타입 시스템에 빗대면 감이 온다. IsA 스키마는 클래스 상속(class Mesh : Boundable, Imageable)처럼 "이 객체는 무엇인가"를 한 줄기로 정하고, API 스키마는 인터페이스·믹스인(implements MaterialBinding)처럼 "이 객체에 어떤 능력을 더 섞을까"를 여러 개 얹는다. 타입이 있어야 도구가 Prim을 열고 UsdGeom.Mesh(prim) 처럼 스키마 클래스로 감싸(cast) 정해진 속성을 안전하게 읽는다. 스키마가 없으면 points가 있는지, 이름이 맞는지 매번 손으로 확인해야 한다 — 스키마는 그 약속을 코드가 아니라 데이터에 새겨 도구 사이에서 흔들리지 않게 한다.

IsA 스키마 vs API 스키마

IsA 스키마(타입 스키마)는 Prim의 typeName이 된다. 한 Prim은 IsA 스키마를 정확히 하나 갖는다 — Xform이거나 Mesh거나 Scope이거나. 이 중 실제로 씬에 찍을 수 있는 것이 concrete(예: Mesh)이고, 직접 못 찍고 공통 인터페이스로만 존재하는 것이 abstract(예: Imageable, Boundable)다.

API 스키마는 타입을 바꾸지 않는다. 대신 Prim에 속성 묶음과 동작을 덧댄다. Prim 메타데이터 apiSchemas에 이름이 쌓인다. 세 결이 있다 — 코드 편의만 주고 씬에 흔적을 안 남기는 non-applied, 한 번만 붙는 single-apply(예: MaterialBindingAPI, UsdGeomModelAPI), 그리고 인스턴스 이름을 붙여 여러 번 붙는 multiple-apply(예: CollectionAPI:zoneA, CollectionAPI:zoneB).

왜 이렇게 갈라 두는가. 한 Prim은 "OHT이면서 동시에 벽"일 수 없으니 정체(IsA)는 하나여야 하지만, "재질도 바인딩되고, 존에도 속하고, 모델 정보도 갖는" 능력은 얼마든지 겹칠 수 있기 때문이다. 팹으로 옮기면 이렇다 — OHT Prim의 IsA는 Xform(또는 Mesh) 하나로 고정하고, 그 위에 "A 구역 소속"(CollectionAPI:zoneA), "강철 재질"(MaterialBindingAPI), "운영 상태 커스텀 속성"을 API로 겹겹이 얹는다. multiple-apply에 인스턴스 이름이 필요한 이유도 분명해진다 — 한 Prim이 zoneA와 zoneB 두 컬렉션에 동시에 속하려면 이름이 없으면 둘이 충돌한다. 흔한 함정 — API 스키마는 Apply()로 등록해야 apiSchemas 메타데이터에 남는다. 속성만 손으로 만들고 Apply()를 빼먹으면 도구가 "이 Prim이 그 능력을 갖췄다"는 것을 알지 못한다.

DIGITAL TWIN 관점

Phase 8의 소스 추상화가 떠오른다면 정확하다. IsA는 "이 노드가 무엇인가"(OHT냐 벽이냐), API는 "이 노드에 무슨 데이터를 더 실을까"(존 소속·재질·상태)다. 트윈에서 실물 상태를 얹을 때, 지오메트리 타입은 그대로 두고 API 스키마로 커스텀 속성을 덧붙이는 것이 USD다운 방식이다 — 원본 자산을 건드리지 않고 상태만 오버레이한다.

UsdGeom — 지오메트리와 계층의 스키마들

씬 그래프의 뼈대는 UsdGeom 스키마 무리다. 자주 만지는 것들:

스키마결역할
Xformconcrete변환(위치·회전·스케일)을 갖는 그룹 노드. 팹 레이아웃의 뼈대.
Scopeconcrete변환이 없는 순수 정리용 그룹. /World/Looks처럼 폴더 역할만.
Meshconcrete폴리곤 메시. points·faceVertexCounts·faceVertexIndices로 형상을 담는다.
Boundableabstract바운딩 박스(extent)를 갖는다는 약속. Mesh가 이를 상속한다.
Imageableabstract"렌더 가능"이라는 공통 인터페이스. visibility·purpose를 준다.
PointInstancerconcrete동일 프로토타입을 수많은 위치에 인스턴스로 뿌리는 특수 노드(아래).

Xform과 Scope의 차이가 실전에서 중요하다. 둘 다 자식을 묶지만, Scope는 변환 능력이 없어 "여기부터 재질 모음", "여기부터 차량들" 같은 순수 조직화에 쓴다. 좌표를 옮길 일이 없는 그룹을 Xform으로 만들면 불필요한 변환 계산이 붙는다. 

from pxr import Usd, UsdGeom, Sdf, Gf

stage = Usd.Stage.CreateNew("fab_layout.usda")
UsdGeom.SetStageUpAxis(stage, UsdGeom.Tokens.y)

world = UsdGeom.Xform.Define(stage, "/World")          # 변환 있는 루트
bay   = UsdGeom.Scope.Define(stage, "/World/Bay_A")    # 정리용 그룹(변환 없음)
oht   = UsdGeom.Mesh.Define(stage, "/World/Bay_A/OHT_01")

oht.CreatePointsAttr([Gf.Vec3f(-1,0,-1), Gf.Vec3f(1,0,-1),
                      Gf.Vec3f(1,0,1),  Gf.Vec3f(-1,0,1)])
oht.CreateFaceVertexCountsAttr([4])
oht.CreateFaceVertexIndicesAttr([0, 1, 2, 3])
stage.GetRootLayer().Save()

UsdGeomPointInstancer — 같은 OHT 수백 대를 하나의 Prim으로

팹에는 똑같은 OHT가 수백 대 달린다. 이를 각각 Mesh Prim으로 두면 씬 그래프가 수백 노드로 부풀고, 순회·편집·전송이 모두 무거워진다. PointInstancer는 이 문제를 정면으로 푼다 — 프로토타입 하나를 정의하고, 나머지는 positions· orientations·scales·protoIndices 배열로만 표현한다. 300대가 노드 300개가 아니라 Prim 한 개 + 배열이 된다.

수치로 감을 잡아 보자. OHT를 개별 Mesh Prim 300개로 두면 씬 그래프에 노드 300개 + 각자의 변환· 바운딩·재질 바인딩이 붙어 순회·직렬화가 300배로 든다. PointInstancer로 두면 프로토타입 메시는 한 벌이고, 나머지는 길이 300짜리 positions·orientations· protoIndices 배열뿐이다 — 값이 구조가 아니라 배열 원소로 사니, 대량 데이터가 촘촘한 배열로 담겨 읽기도 빠르다. 대신 트레이드오프가 있다. 인스턴스 각각은 독립 Prim이 아니라 배열 인덱스라, "3번 인스턴스만 색을 바꾸기" 같은 개별 편집이 번거롭다 (그 인덱스의 배열 원소를 직접 건드려야 한다). 그래서 "완전히 동일하고 대량이면 PointInstancer, 구조를 갖고 개별성이 필요하면 다른 방식"이라는 갈림이 생기고, 그 다른 방식(instanceable 네이티브 인스턴싱)을 Station 06에서 나란히 비교한다. 

pi = UsdGeom.PointInstancer.Define(stage, "/World/OHTs")

# 프로토타입 후보들(여러 종을 두고 인덱스로 고를 수 있다)
pi.CreatePrototypesRel().AddTarget("/World/OHTs/Prototypes/OHT")

# 인스턴스별 데이터 — 세 대를 레일에 4m 간격으로
pi.CreateProtoIndicesAttr([0, 0, 0])
pi.CreatePositionsAttr([Gf.Vec3f(0, 3, 0),
                        Gf.Vec3f(4, 3, 0),
                        Gf.Vec3f(8, 3, 0)])

protoIndices가 각 인스턴스가 어느 프로토타입을 쓰는지 가리키므로, OHT·AGV 여러 종을 프로토타입으로 두고 한 인스턴서에서 섞어 뿌릴 수 있다. (Station 06에서 이것이 개별 Prim 방식, instanceable 네이티브 인스턴싱과 어떻게 갈리는지 비교한다.)

UsdShade — 재질과 셰이더의 그래프

표면의 색·금속성·거칠기는 UsdShade.Material과 UsdShade.Shader로 정의한다. Material은 바인딩의 접점이고(surface·displacement· volume 출력을 가짐), 실제 계산은 Shader 노드들이 inputs/outputs로 엮여 수행한다. 이식성을 위해 표준 셰이더 UsdPreviewSurface를 쓰면 뷰어·DCC·엔진이 같은 룩을 낸다 — Phase 3의 PBR 파라미터(알베도·metallic·roughness)가 그대로 입력으로 들어간다.

구조를 한 번 더 뜯으면, USD의 재질은 노드 그래프다. Material은 바깥과 닿는 단자함이고(지오메트리는 이 Material만 알면 된다), 그 안에서 여러 Shader 노드가 ConnectToSource로 입력·출력을 이어 최종 surface 출력을 만든다 — 텍스처 읽기 노드 → 색 보정 노드 → PreviewSurface로 흐르는 식이다. 이 간접층 덕에 지오메트리는 셰이더 내부를 몰라도 되고, 재질만 통째로 바꿔 끼울 수 있다. 왜 하필 UsdPreviewSurface를 표준으로 미나 하면, USD 자체는 렌더러가 아니기 때문이다. USD는 룩을 기술할 뿐 픽셀은 각 렌더러(Storm/Hydra, RTX 등)가 그린다. 모두가 알아듣는 공통 셰이더가 있어야 "한 번 정의한 룩을 어느 뷰어·엔진에서 열어도 비슷하게" 볼 수 있고, 그 공통어가 PreviewSurface다. 더 정교한 룩이 필요하면 아래 MaterialX로 넘어간다. 

from pxr import UsdShade

mat = UsdShade.Material.Define(stage, "/World/Looks/Steel")
shd = UsdShade.Shader.Define(stage, "/World/Looks/Steel/PBR")
shd.CreateIdAttr("UsdPreviewSurface")
shd.CreateInput("metallic",  Sdf.ValueTypeNames.Float).Set(1.0)
shd.CreateInput("roughness", Sdf.ValueTypeNames.Float).Set(0.35)

mat.CreateSurfaceOutput().ConnectToSource(shd.ConnectableAPI(), "surface")

# 메시에 재질을 바인딩 — MaterialBindingAPI(단일 적용 API 스키마)
UsdShade.MaterialBindingAPI.Apply(oht.GetPrim())
UsdShade.MaterialBindingAPI(oht).Bind(mat)

더 정교한 룩은 MaterialX로 표현한다. MaterialX는 노드 기반 셰이딩 그래프의 업계 표준 교환 포맷이고, USD는 UsdMtlx를 통해 이를 UsdShade 네트워크로 읽어들인다. "룩을 한 번 정의하고 여러 렌더러가 같게 그린다"는 USD의 이식성 철학이 재질까지 이어지는 지점이다.

UsdGeomSubset — 한 메시에 재질 여럿

벤더 CAD의 OHT 한 덩어리는 보통 여러 재질(강철 프레임·유리 커버·경고 마킹)을 쓴다. USD는 이를 UsdGeomSubset으로 표현한다 — 메시의 면 일부를 이름 붙은 부분집합으로 묶고, 각 부분집합에 다른 재질을 바인딩한다. Phase 10의 서브메시가 바로 이 개념의 엔진 쪽 대응물이다: 재질 수 = 부분집합 수 = 드로우콜 하한. 

from pxr import UsdGeom

# 3~5번 면만 "glass" 부분집합으로 — 여기에 유리 재질을 따로 바인딩
glass = UsdGeom.Subset.CreateGeomSubset(
    oht, "glass", UsdGeom.Tokens.face, [3, 4, 5])
UsdShade.MaterialBindingAPI(glass.GetPrim()).Bind(glass_mat)

이 모든 조각 — Xform/Scope 계층, Mesh, PointInstancer, Material, Subset — 을 하나의 스크립트로 엮어 팹 씬을 조립하는 것이 예제 examples/openusd-lab/build_fab_stage.py이고, 다음 스테이션(05 실습)에서 직접 만든다.

핵심 확인 질문

한 Prim이 가질 수 있는 IsA 스키마와 API 스키마의 개수는 각각 몇 개인가?

IsA 스키마(타입)는 정확히 하나다 — Prim의 typeName이 곧 그것이다. API 스키마는 여럿 붙을 수 있고 apiSchemas 메타데이터에 쌓인다. "무엇인가"는 하나, "무엇을 더 할 수 있나"는 여럿이라는 비대칭이 핵심이다.

변환이 필요 없는 순수 그룹을 만들 때 Xform 대신 Scope를 쓰는 이유는?

Scope는 변환 능력이 없는 정리용 그룹이라, 좌표를 옮길 일이 없는 폴더성 노드 (/World/Looks, 재질·프로토타입 모음 등)에 알맞다. Xform으로 만들면 불필요한 변환 속성과 계산이 따라붙는다. 의도(변환 없음)를 타입으로 못 박는 것이다.

동일한 OHT 300대를 PointInstancer로 두면 개별 Mesh 300개 대비 무엇이 달라지나?

씬 그래프가 노드 300개가 아니라 인스턴서 Prim 하나 + 위치/인덱스 배열이 된다. 프로토타입은 한 벌만 저장되고 인스턴스는 값 배열로만 존재하므로 순회·편집·전송·메모리가 모두 가벼워진다. Phase 4 GPU 인스턴싱을 데이터 모델 차원에서 하는 셈이다.

PointInstancer의 대가(트레이드오프)는 무엇인가?

가벼움을 얻는 대신 개별 편집이 번거롭다. 인스턴스는 독립 Prim이 아니라 배열 인덱스라, "3번만 색을 바꾸기" 같은 일은 그 인덱스의 배열 원소를 직접 손대야 한다. 그래서 완전히 동일한 대량 배치엔 알맞지만, 각 인스턴스에 개별성·구조가 필요하면 instanceable 네이티브 인스턴싱이 더 맞는다(Station 06 비교).

UsdShade에서 Material과 Shader의 역할은 어떻게 나뉘는가?

Material은 바인딩의 접점으로 surface 등 출력을 갖고 지오메트리에 붙는다. 실제 색·금속성·거칠기 계산은 Shader 노드들이 입력/출력으로 엮여 수행하며, 표준 UsdPreviewSurface를 쓰면 여러 렌더러가 같은 룩을 낸다.

UsdGeomSubset이 Phase 10의 "서브메시 = 드로우콜 하한"과 이어지는 지점은?

UsdGeomSubset은 한 메시의 면 일부를 부분집합으로 묶어 서로 다른 재질을 바인딩하는 장치다. 부분집합(재질) 수가 곧 엔진의 서브메시 수이고, 서브메시 하나가 최소 드로우콜 하나가 된다. USD 단계의 재질 분할이 그대로 런타임 드로우콜 예산으로 번역된다.
