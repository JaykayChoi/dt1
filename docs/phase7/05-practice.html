PHASE 7 — FLEET 심화

Phase 7 · Station 05 · 실습

실습 — FabSim에 교통 제어 넣기

01~04에서 교통 제어·데드락·라우팅·배차를 개념과 콘솔 랩으로 익혔다. 이제 그 넷을 자기 손으로 — 스켈레톤의 SegmentController를 채우고, 데드락을 직접 유발·탐지하고, 자신의 FabSim에 교통 제어를 이식한다. 읽는 페이지가 아니라 만드는 페이지다.

시작 상태 — 어디서 출발하나

  • Unity 실습은 사용자의 FabSim/에서 진행한다(Phase 5 결과 — 레일 루프 + FabModel 브리지 + 뷰/HUD가 올라간 프로젝트).
  • 폴백: FabSim에 Phase 5 결과가 없거나 어긋났다면 FabSim-P5-Complete를 참고해 "레일 위를 차량이 도는" 상태를 먼저 만든다.
  • 콘솔 실습은 examples/fleet-traffic-practice· fleet-routing-practice 스켈레톤에서 출발한다.

버튼은 어떤 방식으로 열었든 동작한다. 콘솔 실습은 프로젝트를 열 필요 없이 examples/fleet-traffic-practice에서 바로 dotnet run으로 한다.

실습 체크리스트 — 4단계

  1. 콘솔에서 교통 제어 뼈대 세우기 — examples/fleet-traffic-practice의 SegmentController TODO (Acquire/Release/HolderOf/DetectDeadlockCycle)를 채운다.
    완료 기준: dotnet run --project examples/fleet-traffic-practice -c Release의 실험 A ON에서 처리량이 12대에서 꺾인다(완성본과 같은 234건/h 부근).
    막히면: 완성본 examples/fleet-traffic/SegmentController.cs.
  2. 데드락을 직접 유발·탐지 — 채운 DetectDeadlockCycle()로 양방향 corridor의 순환 대기를 잡는다.
    완료 기준: 탐지 로그에 사이클·발생 시각·차량 id(t=4.0s [0, 1])가 찍히고, 단방향화·자원 순서화 구성에선 데드락이 0이 된다.
    막히면: 완성본의 실험 B 하네스와 DetectDeadlockCycle의 DFS.
  3. FabSim에 SegmentController 이식 — 채운 컨트롤러를 FabSim의 Sim/에 넣고, FabModel의 엣지 이동을 "다음 엣지 Acquire → 획득 시 이동 → 도착 시 이전 엣지 Release"로 바꾼다. 대기 상태색과 엣지 점유 히트맵을 붙인다.
    완료 기준: Play에서 차량이 앞차 뒤에 실제로 정지·대기하고, 정체 구간이 히트맵으로 물들며, 데드락 시 관련 차량이 하이라이트된다.
    막히면: 완성본 FabSim-P7-Complete/Assets/Scripts/Sim/SegmentController.cs와 FabModel의 Acquire/Release 통합부, 히트맵/대기색 컨트롤러.
  4. 배차 심화 확장 과제 — examples/fleet-dispatch에 WeightedScore 정책과 P95Wait 지표를 더해 가중치 k를 스윕한다.
    완료 기준: k 스윕 표가 나오고, 이 균형 팹에서는 k를 올릴수록 P95가 오히려 오르는 정직한 결과(기아가 없어 에이징이 역효과)를 재현한다.
    막히면: 완성본 fleet-dispatch/Program.cs의 SelectWeightedPair·RunWeightedSweep.
보고 베끼지 말 것

완성본은 대조하는 곳이지 베끼는 곳이 아니다. 스켈레톤 실행 출력이 완성본과 어긋나면(제어 ON이 안 꺾임·데드락 미검출·A* 무개선) 그것이 "미완성"의 신호다. 손으로 만든 뒤 비교해야 다음 페이즈에서 막히지 않는다.

완료 기준 (Definition of Done)

  • 콘솔 스켈레톤이 완성본과 같은 수치(처리량 꺾임 234·데드락 t=4.0s·A* 6.5확장)를 낸다.
  • FabSim에서 차량이 앞차 뒤에 정지하고, 점유 히트맵·데드락 하이라이트가 동작한다.
  • fleet-dispatch가 k 스윕을 출력하고, 에이징의 정직한 트레이드오프를 재현한다.

핵심 확인 질문

Acquire가 즉시 콜백과 대기 큐 삽입으로 갈리는 조건은?

구간이 비어 있으면(HolderOf == -1) 그 차량을 점유자로 기록하고 즉시 onGranted를 부른다. 이미 점유 중이면 대기 큐에 넣고 waitingEdge에 "이 차량이 이 구간을 기다린다"를 기록해, 나중에 승계받게 한다.

이식 후 FabSim의 일방통행 루프에서 데드락이 안 생기는 이유는?

단방향 루프는 항상 빈 구간이 있어 대기 관계가 원(순환)을 이루지 못한다. 순환 대기가 없으면 데드락 4조건이 완성되지 않는다. 데드락을 보려면 양방향 corridor를 일부러 넣어야 한다.

대기 상태를 VehicleAgent에 어떻게 표현하고, 뷰는 그것을 어떻게 색으로 옮기나?

VehicleAgent에 IsWaiting(또는 Waiting 단계)을 두고, Acquire가 대기 큐로 갈 때 참으로 만든다. 뷰(VehicleFleetView)는 매 프레임 그 플래그를 읽어 차량의 MaterialPropertyBlock으로 _BaseColor를 경고색으로 밀어(GC 없이) 대기 차량을 물들인다.

엣지 점유율을 0~1로 집계해 히트맵 _Value로 넘기는 흐름은?

각 구간의 최근 창 기준 점유 시간 비율을 0~1로 집계하는 컨트롤러가, 레일 박스에 붙은 HeatmapCell.Value에 그 값을 넣는다. HeatmapCell은 MaterialPropertyBlock으로 FabVizHeatmap 셰이더의 _Value를 갱신해 파랑→적 램프로 정체를 시각화한다.

스켈레톤 실행 출력이 "미완성"임을 무엇으로 아는가?

교통 제어 ON이 OFF와 똑같이 처리량이 단조 증가하고(꺾이지 않음), 데드락을 못 잡으며, A*의 확장 노드가 다익스트라와 같거나 많게 나온다. 이 세 신호가 완성본 수치와 어긋나는 것이 곧 채울 곳이 남았다는 뜻이다.
