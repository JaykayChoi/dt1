PHASE 4 — 성능 분석·최적화

Phase 4 · Station 02

Frame Debugger와 배칭

Profiler가 "시간이 어디에 쓰이는가"에 답한다면, Frame Debugger는 "이 화면 한 장이 어떤 순서로, 몇 번의 명령으로 그려지는가"에 답한다. 그리고 그 명령 횟수를 줄이는 기술이 배칭이다 — 장비 수백 대를 그리면서도 CPU가 태연할 수 있는 이유가 여기에 있다.

화면 한 장이 그려지는 과정을 한 장씩

Frame Debugger(Window > Analysis > Frame Debugger)는 한 프레임의 렌더링을 드로우콜 단위로 순서대로 재생하는 도구다. 창을 열고 Enable을 누르면 게임이 그 자리에 멈추고, 왼쪽에 그 프레임을 구성한 렌더링 이벤트의 목록이 나타난다. 목록의 항목을 위에서부터 하나씩 클릭하면 Game 뷰가 "여기까지 그린 화면"으로 바뀐다 — 빈 화면에서 시작해 바닥이 깔리고, 장비가 하나씩 올라오고, 마지막에 포스트 프로세싱이 입혀지는 과정을 슬라이드쇼처럼 넘겨볼 수 있다.

지금까지 렌더링은 "카메라가 있으면 알아서 그려지는" 블랙박스였다. Frame Debugger는 그 블랙박스를 한 장씩 넘겨볼 수 있는 책으로 바꾼다. 각 항목을 선택하면 오른쪽 패널에 그 드로우에 쓰인 셰이더, 머티리얼 속성, 텍스처, 그리고 "왜 이 드로우가 앞의 드로우와 묶이지 못했는가"라는 배칭 실패 사유까지 표시된다. 이 마지막 정보가 최적화 작업의 나침반이 된다.

드로우콜이 왜 비싼가

드로우콜(draw call)은 CPU가 GPU에게 "이 메시를 이 상태로 그려라"라고 내리는 명령 하나다. 여기서 비용의 본체는 GPU가 삼각형을 그리는 작업이 아니라, 명령을 내리기 전에 CPU가 해야 하는 준비다.

  • 상태 변경 — 셰이더 교체, 머티리얼 파라미터 설정, 텍스처·버퍼 바인딩. 렌더 상태가 바뀔 때마다 CPU와 그래픽스 드라이버가 이를 검증하고 GPU에 전달해야 한다. Stats 창의 SetPass calls가 이 상태 전환의 횟수다.
  • 명령 제출 — 오브젝트별 변환 행렬 등 데이터를 준비해 커맨드 버퍼에 쌓는 일. 드로우 하나하나는 싸 보여도 수천 번 반복되면 CPU 렌더 스레드의 시간을 통째로 삼킨다.

그래서 드로우콜이 많은 씬의 전형적인 증상은 "GPU는 놀고 있는데 프레임이 안 나오는" CPU 바운드 상태다 — Station 01의 Profiler에서 렌더링 관련 마커가 CPU 타임라인을 채우고 있다면 이 경우를 의심한다. 그리고 팹 씬은 이 문제의 모범적인 피해자가 되기 쉽다. 장비 500대를 순진하게 그리면 500번 이상의 드로우콜이고, 장비마다 상태등·배관 같은 부속이 붙으면 곱절로 불어난다.

배칭 3형제

배칭(batching)은 여러 오브젝트의 드로우를 묶어 CPU의 준비 비용을 나눠 갖게 하는 기법의 총칭이다. URP에서 실전에 쓰이는 것은 세 가지다.

SRP Batcher — URP의 기본기

SRP Batcher는 URP에서 기본으로 켜져 있는 배칭이다. 머티리얼 데이터를 GPU 메모리에 상주시켜 두고, 같은 셰이더 변형(shader variant)을 쓰는 연속된 드로우 사이에서는 상태 준비를 건너뛴다. 핵심은 조건이 느슨하다는 것 — 머티리얼이 서로 달라도 셰이더 변형만 같으면 묶인다. URP/Lit 하나로 색과 파라미터만 다른 머티리얼 여러 장을 쓰는 씬이라면 거의 전부가 한 묶음이 된다.

한 가지 정확히 해 둘 점: SRP Batcher는 드로우콜의 개수를 줄이지 않는다. 콜 사이의 준비 비용을 줄일 뿐이다. Frame Debugger 목록에는 SRP Batch라는 항목 아래 여러 드로우가 접혀 들어간 모습으로 나타난다. URP 에셋의 SRP Batcher 체크박스를 꺼 보면(버전에 따라 추가 속성 표시를 켜야 보인다) 같은 씬의 CPU 렌더 비용이 어떻게 달라지는지 직접 비교할 수 있다.

GPU 인스턴싱 — 같은 것 수백 개를 한 번에

GPU 인스턴싱(GPU instancing)은 같은 메시 + 같은 머티리얼의 조합이 여러 개일 때, GPU에게 "이 메시를 N개 그려라"라는 드로우콜 하나로 처리하는 기법이다. 오브젝트별로 다른 것(위치·회전 등)은 인스턴스 데이터 배열로 한꺼번에 넘긴다. 머티리얼 인스펙터의 Enable GPU Instancing 체크박스 하나로 켜지며, Frame Debugger에는 Draw Mesh (instanced)라는 항목으로 나타난다.

같은 비히클 메시 수백 대, 같은 상태등 수백 개 — 동일한 물건이 대량으로 반복되는 씬에 이상적이다. 다만 우선순위 규칙이 하나 있다: SRP Batcher와 호환되는 렌더러에는 SRP Batcher가 우선 적용되어, 머티리얼에 인스턴싱을 켜 두어도 일반 MeshRenderer는 SRP Batch 경로로 그려질 수 있다. (스크립트에서 Graphics.RenderMeshInstanced로 직접 그리는 경로는 예외다.) 어느 경로를 탔는지는 추측하지 말고 Frame Debugger로 확인하는 것이 정답이다.

정적 배칭 — 움직이지 않는 것들을 미리 합쳐 두기

정적 배칭(static batching)은 씬에서 절대 움직이지 않는 오브젝트들에 Static 플래그를 세워 두면, Unity가 빌드·씬 로드 시점에 그 메시들을 하나의 큰 정점 버퍼로 합쳐 두는 기법이다. 이후에는 오브젝트 사이의 상태 전환 없이 매우 싸게 그려진다. 대가는 메모리다 — 같은 메시 100개를 정적 배칭하면 100개 분량의 정점 데이터가 복제되어 저장되고, 당연히 런타임에 움직일 수도 없다. 바닥·벽·레일처럼 한번 놓이면 끝인 지오메트리가 대상이다.

기법 묶이는 조건 효과 비용 · 한계
SRP Batcher 같은 셰이더 변형 (머티리얼은 달라도 됨) 드로우콜 사이의 CPU 준비 비용 절감 — 콜 수 자체는 그대로 SRP 호환 셰이더 필요, MaterialPropertyBlock과 비호환
GPU 인스턴싱 같은 메시 + 같은 머티리얼 (Enable GPU Instancing) N개를 드로우콜 1번으로 — 콜 수 자체가 줄어듦 인스턴스별 차이는 제한적, 호환 렌더러엔 SRP Batcher가 우선
정적 배칭 Static 플래그 + 같은 머티리얼, 움직이지 않는 오브젝트 합쳐 둔 메시를 상태 전환 없이 그림 합친 만큼 메모리 증가, 런타임 이동 불가
배칭이 깨지는 흔한 이유

스크립트에서 renderer.material에 접근하는 순간 Unity는 그 렌더러만의 머티리얼 복사본을 조용히 만들어 낸다 — 이제 "같은 머티리얼" 조건이 깨져 그 오브젝트는 인스턴싱 묶음에서 이탈한다 (읽기만 할 때도 복제된다. 공유 상태를 건드리려면 renderer.sharedMaterial을 쓴다). 렌더러별 속성을 주는 MaterialPropertyBlock은 복제는 피하지만 SRP Batcher와 비호환이라 그 렌더러를 SRP Batch에서 빼낸다. 또한 셰이더 키워드가 달라 셰이더 변형이 갈리면 같은 셰이더라도 SRP Batch가 그 지점에서 쪼개진다. 어느 경우든 증상은 같다 — "분명히 같은 물건인데 배치 수가 늘어나 있다". 원인은 Frame Debugger의 배칭 실패 사유에서 찾는다.

실험: 완성본 실험장에서 확인

FabSim-P4-Complete 프로젝트의 StressField 씬 (Assets/Scenes/StressField.unity)은 이 이론을 눈으로 확인하기 위한 무대다. 6개 베이에 장비 박스 192대, 각 장비의 상태등 192개, 레일 6개, 바닥·천장, 그리고 순환 트랙을 도는 비히클 96대 — 합쳐서 렌더러 488개(빌드 로그 기준)로 구성된다. 씬을 만드는 StressSceneBuilder.cs는 모든 머티리얼에 GPU 인스턴싱을 켜 둔다. 

// Assets/Editor/StressSceneBuilder.cs — CreateLitMaterial()
var material = new Material(Shader.Find("Universal Render Pipeline/Lit"));
material.SetColor("_BaseColor", baseColor);
material.SetFloat("_Metallic", metallic);
material.SetFloat("_Smoothness", smoothness);

// GPU 인스턴싱 — 같은 메시+머티리얼을 드로우콜 하나로 묶는 열쇠.
material.enableInstancing = true;

움직이지 않는 지오메트리(바닥·천장·장비·레일·상태등)에는 Static 플래그가 일괄로 세워진다. 결국 이 씬에는 배칭 3형제가 모두 출동해 있는 셈이다 — 움직이는 비히클 96대는 Static이 될 수 없으니 인스턴싱이 맡고, 나머지는 정적 배칭과 SRP Batcher의 영역이다. 

// Assets/Editor/StressSceneBuilder.cs — MarkStaticRecursively()
foreach (Transform child in root.GetComponentsInChildren<Transform>(true))
{
    GameObjectUtility.SetStaticEditorFlags(child.gameObject, (StaticEditorFlags)~0);
}

관찰 절차는 다음과 같다.

  1. Unity로 FabSim-P4-Complete를 열고 StressField 씬을 로드한 뒤 Play — 비히클 무리가 팹 둘레를 돌기 시작한다.
  2. Game 뷰 우상단의 Stats를 켜고 Batches와 SetPass calls를 기록한다. 렌더러가 488개라는 사실과 나란히 두면 배칭의 효과가 숫자로 보인다.
  3. Window > Analysis > Frame Debugger를 열고 Enable. 게임이 멈추고 이 프레임의 이벤트 목록이 나타난다.
  4. 목록을 위에서 아래로 클릭하며 화면이 쌓이는 순서를 관찰한다 — 불투명 지오메트리가 먼저, 포스트 프로세싱(Bloom, Tonemapping)이 마지막이다.
  5. 목록에서 Draw Mesh (instanced) 항목을 찾는다 — 같은 메시 수백 개가 몇 번의 콜로 묶였는지 확인한다. 만약 대부분이 SRP Batch 항목으로만 보인다면, 그것대로 수확이다 — "호환 렌더러에는 SRP Batcher가 우선"이라는 규칙을 눈으로 확인한 것이다.
  6. URP 에셋(Assets/Settings/FabSimUrp.asset)에서 SRP Batcher를 꺼 보고 Batches 수와 이벤트 목록이 어떻게 변하는지 비교한다. 관찰이 끝나면 반드시 다시 켠다.
DIGITAL TWIN 관점

OHT 수천 대가 움직이는 팹 트윈이 한 화면에 그려질 수 있는 이유가 바로 인스턴싱이다. 팹의 반송 세계는 렌더링 관점에서 축복받은 구조다 — 비히클은 전부 같은 메시의 반복이고, 레일과 장비도 몇 종류의 모듈이 복제된 것이다. "종류는 적고 개수는 많다"는 팹의 성질이 그대로 "묶을 수 있는 드로우가 많다"로 번역된다. 트윈의 시각화 성능 설계는 이 성질을 얼마나 살리느냐의 게임이다.

핵심 확인 질문

Profiler와 Frame Debugger의 역할 차이는 무엇인가?

Profiler는 시간의 도구다 — 한 프레임의 CPU/GPU 시간이 어떤 작업에 얼마나 쓰였는지 알려준다. Frame Debugger는 순서와 횟수의 도구다 — 그 프레임의 화면이 어떤 드로우콜을 어떤 순서로 거쳐 완성됐고, 무엇이 묶이고 무엇이 묶이지 못했는지 보여준다. "렌더링이 느리다"는 진단은 Profiler에서, "왜 드로우가 많은가"의 답은 Frame Debugger에서 나온다.

드로우콜이 비싼 진짜 이유는 무엇인가?

GPU가 그리는 작업 자체보다 CPU 쪽의 준비 — 셰이더·텍스처·버퍼 같은 렌더 상태를 바꾸고 검증해 명령을 제출하는 비용 — 가 본체이기 때문이다. 그래서 드로우콜 과다의 전형적 증상은 GPU가 아니라 CPU가 바쁜 CPU 바운드다.

SRP Batcher와 GPU 인스턴싱은 각각 무엇을 줄이는가?

SRP Batcher는 콜 사이의 준비 비용을 줄인다 — 같은 셰이더 변형이면 머티리얼이 달라도 묶이지만, 드로우콜 개수 자체는 그대로다. GPU 인스턴싱은 콜 개수 자체를 줄인다 — 같은 메시+머티리얼 N개를 드로우콜 1번으로 그린다. 단 호환 렌더러에는 SRP Batcher가 우선 적용된다는 규칙이 있다.

스크립트에서 renderer.material 접근이 배칭을 깨는 이유는?

접근하는 순간 그 렌더러만의 머티리얼 복사본이 생성되어 "같은 머티리얼" 조건이 깨지기 때문이다. 공유 머티리얼을 쓰려면 renderer.sharedMaterial, 렌더러별 속성이 꼭 필요하면 MaterialPropertyBlock을 쓰되 후자는 SRP Batcher와 비호환이라는 트레이드오프를 감수해야 한다.

StressField 씬의 렌더러 488개가 훨씬 적은 배치 수로 그려지는 이유는?

씬이 배칭에 유리하게 설계되어 있기 때문이다 — 오브젝트 대부분이 몇 종류의 메시·머티리얼의 반복이고, 모든 머티리얼에 GPU 인스턴싱이 켜져 있으며(material.enableInstancing = true), 움직이지 않는 지오메트리에는 Static 플래그가 세워져 있다. 움직이는 비히클은 인스턴싱이, 고정 지오메트리는 정적 배칭과 SRP Batcher가 묶는다.
