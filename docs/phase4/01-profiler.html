PHASE 4 — 성능 분석·최적화

Phase 4 · Station 01

Profiler 읽는 법 — CPU 타임라인 · GC

"느린 것 같다"는 감각으로는 아무것도 고칠 수 없다. Unity Profiler는 프레임당 16.6ms의 예산이 정확히 어디에 쓰였는지 보여 주는 계측기다. 이 스테이션에서는 CPU 타임라인을 읽고, CPU 바운드인지 GPU 바운드인지 가려내고, 프레임을 훔쳐 가는 GC 할당을 추적하는 법을 배운다.

측정 없이는 최적화도 없다

Phase 4 전체를 지배하는 원칙은 하나다: 측정 → 병목 확인 → 수정 → 재측정. "오브젝트가 많아서 느린 것 같으니 줄이자", "그림자가 무거울 테니 끄자" 같은 추측 기반 최적화는 대개 엉뚱한 곳을 고친다. 실제 병목이 스크립트의 GC 할당인데 텍스처 해상도를 줄이고 있으면, 며칠을 써도 프레임은 그대로다. 최적화는 코드를 잘 쓰는 기술이기 이전에 어디가 느린지 찾아내는 기술이다.

기준선은 프레임 예산(frame budget)이다. 60fps로 돌리려면 한 프레임을 1000ms ÷ 60 ≈ 16.6ms 안에 끝내야 한다. 30fps라면 33.3ms. 이 시간 안에 스크립트 실행, 물리, 애니메이션, 컬링과 드로우콜 제출, GPU의 실제 그리기까지 전부 끝나야 한다. 즉 한 프레임은 마감이 정해진 파이프라인이다.

Phase 2의 큐잉 이론을 기억한다면 이 그림이 낯설지 않을 것이다. 생산 라인의 처리량을 병목 스테이션이 결정했듯, 프레임 타임은 파이프라인에서 가장 느린 구간이 결정한다. CPU가 5ms 만에 일을 끝내도 GPU가 25ms를 쓰면 그 프레임은 25ms짜리다. 반대로 GPU가 놀고 있어도 스크립트가 20ms를 먹으면 역시 60fps는 물 건너간다. 그래서 최적화의 첫 질문은 언제나 "지금 어느 구간이 병목인가"이고, 그 답을 주는 도구가 Profiler다.

Profiler 열기와 기본 구조

Profiler는 Window > Analysis > Profiler(단축키 Ctrl+7)로 연다. 에디터에서 Play를 누르면 기록이 시작되고, 창은 크게 두 부분으로 나뉜다.

  • 위쪽 — 모듈 트랙: CPU Usage, Rendering, Memory 등 모듈별 시간축 차트. 프레임 하나가 세로 한 칸이고, 차트가 솟은 곳이 비싼 프레임이다. 필요한 모듈은 창 좌상단의 모듈 목록에서 켜고 끌 수 있다.
  • 아래쪽 — 상세 뷰: 위에서 선택한 모듈과 프레임의 내부를 보여 준다. CPU Usage 모듈을 선택했을 때의 상세 뷰가 이 페이지의 주인공이다.

기본 사용법은 "솟은 프레임을 클릭해 멈춰 놓고 읽기"다. 차트에서 스파이크가 보이면 그 프레임을 클릭한다 — 아래 상세 뷰가 해당 프레임의 데이터로 고정되고, 이제 그 16.6ms(혹은 그걸 넘긴 시간)가 어디에 쓰였는지 항목별로 추적할 수 있다. 흘러가는 차트를 눈으로 쫓는 것이 아니라, 한 프레임을 박제해 놓고 해부하는 것이 Profiler의 기본 자세다.

에디터에서의 측정 함정

에디터에서 측정한 수치에는 에디터 자신의 비용이 섞여 있다. 인스펙터·씬 뷰·Profiler 창 자체가 CPU와 메모리를 쓰고, 에디터 전용 안전 검사가 끼어들며, 코드 최적화 수준도 실제 빌드와 다르다. 그래서 에디터의 절대 수치를 "이 게임은 프레임당 12ms짜리"라고 믿으면 안 된다. 에디터 측정은 상대 비교와 경향 파악용이다 — "수정 전 8ms였던 항목이 수정 후 2ms가 됐다"는 판단은 유효하다. 최종 판정은 실제 빌드를 대상 기기에서 돌리며 프로파일링해서 내린다.

CPU 타임라인 읽는 법

CPU Usage 모듈의 상세 뷰는 두 가지 방식으로 볼 수 있다. 상세 뷰 좌상단의 드롭다운에서 전환한다.

  • Hierarchy 뷰 — 함수 호출을 트리로 접어 놓은 표. 항목별로 소요 시간(ms), 호출 횟수(Calls), GC Alloc(그 프레임에 할당한 힙 바이트) 열이 나온다. "무엇이 비싼가"를 찾을 때 가장 빠르다. Time ms 열로 정렬해 놓고 읽는 것이 기본.
  • Timeline 뷰 — 스레드별 시간축 위에 각 작업이 실제로 언제, 얼마나 오래 실행됐는지를 띠로 보여 준다. 메인 스레드, 렌더 스레드, 잡 워커 스레드가 나란히 보이므로 "메인 스레드가 렌더 스레드를 기다리는가" 같은 스레드 사이의 관계는 여기서만 보인다.

읽는 순서

  1. 프레임 총 ms를 확인한다 — 예산(16.6ms) 대비 얼마나 초과했는가.
  2. 가장 큰 항목부터 연다 — 메인 스레드의 뿌리는 대부분 PlayerLoop이고, 그 아래에서 스크립트(Update 계열), 렌더링, 물리 중 어느 갈래가 시간을 먹는지부터 가른다.
  3. 범인 갈래를 끝까지 판다 — 스크립트라면 어느 컴포넌트의 Update인지, 렌더링이라면 컬링인지 드로우콜 제출인지까지 내려간다.

자주 만나는 항목

  • Update / LateUpdate / FixedUpdate 계열 — 내가 쓴 스크립트의 비용. 그 아래에 컴포넌트별 항목이 나오므로 범인 지목이 쉽다.
  • Camera.Render — 렌더링의 CPU 쪽 비용. 컬링, 드로우콜 준비·제출이 여기 들어간다. 이 항목이 크면 GPU가 아니라 "그리라고 지시하는 일" 자체가 비싼 것이고, 처방은 배칭·드로우콜 감소다 (Station 02의 주제).
  • Gfx.WaitForPresent 계열의 대기 항목 — CPU가 GPU를 기다리는 시간. CPU는 일을 다 끝냈는데 GPU가 아직 이전 프레임을 그리고 있어서 노는 상태다. 이 항목이 크면 병목은 CPU 코드가 아니라 GPU다.
  • WaitForTargetFPS — vsync나 목표 프레임레이트에 맞추기 위한 대기. 병목이 아니라 예산이 남는다는 뜻이므로 이걸 줄이려고 애쓸 필요는 없다.

CPU 바운드인가, GPU 바운드인가

이 판별이 이 페이지의 절반이다. 프레임이 예산을 넘겼을 때, 다음에 무엇을 최적화할지는 어느 쪽이 병목인지에 따라 완전히 달라진다. CPU 바운드인데 셰이더를 다듬거나, GPU 바운드인데 스크립트를 다듬으면 프레임은 1ms도 안 줄어든다.

Profiler에서 보이는 신호 진단 다음 행동
Update 계열이 크다 CPU 바운드 — 스크립트 해당 컴포넌트 코드 최적화, GC Alloc 확인, 관리 주체 통합 (Station 03)
Camera.Render가 크다 CPU 바운드 — 렌더 제출 드로우콜 줄이기: 배칭, 컬링 (Station 02)
Gfx.WaitForPresent가 크다 GPU 바운드 해상도·셰이더 비용·오버드로우·그림자 품질 조정
WaitForTargetFPS가 크다 병목 없음 — 예산 여유 최적화 불필요. 다른 일을 하자
현장 감각

Hierarchy에서 내 스크립트가 익명의 항목 속에 묻혀 잘 안 보일 때는, 의심 가는 코드 구간을 Profiler.BeginSample("구간 이름")과 Profiler.EndSample()로 감싸 보자. 그 이름이 Hierarchy에 하나의 항목으로 나타나 시간과 GC Alloc이 따로 집계된다. "이 함수의 이 절반이 문제인가, 저 절반이 문제인가"를 가를 때 특히 유용한, 비용이 거의 없는 습관이다.

GC Alloc — 조용한 프레임 도둑

C# 코드에서 new로 만드는 클래스 인스턴스, 문자열, 배열은 관리 힙(managed heap)에 할당된다. 이 할당 자체는 싸다. 문제는 청소다 — 힙이 차면 가비지 컬렉터(GC)가 돌며 더 이상 안 쓰는 메모리를 회수하는데, 이 작업이 하필 바쁜 프레임에 끼어들면 그 프레임만 예산을 훌쩍 넘기는 스파이크가 생긴다. 위 도식의 프레임 4가 바로 그 순간이다. 인크리멘털 GC가 이 비용을 여러 프레임에 잘게 나눠 주긴 하지만, 근본 해법은 하나뿐이다: 매 프레임 반복되는 할당을 없애는 것.

범인 찾기는 Hierarchy 뷰의 GC Alloc 열이 담당한다. 이 열로 정렬하면 그 프레임에 힙을 할당한 함수가 바이트 수와 함께 나온다. 한 번의 할당은 문제가 아니다 — 로딩이나 초기화 때의 할당은 정상이다. 찾아야 할 것은 매 프레임 꾸준히 0이 아닌 값을 찍는 함수다. 흔한 범인은 정해져 있다.

  • 매 프레임 문자열 생성 — "FPS: " + fps 같은 연결·보간은 호출마다 새 문자열을 만든다. HUD 갱신 코드의 단골 문제.
  • LINQ와 클로저 — Where, ToList 등은 내부적으로 반복자와 컬렉션을 할당한다. Update 안의 LINQ는 거의 항상 유죄.
  • 박싱 — 값 타입을 object나 인터페이스로 넘길 때 암묵적으로 힙 복사본이 생긴다.
  • new 배열·리스트 — 매 프레임 임시 배열을 만들거나, 호출할 때마다 배열을 반환하는 API를 쓰는 경우. 버퍼를 미리 만들어 재사용하는 것이 답.

목표는 명확하다: 정상 상태에서 GC Alloc 0 B/프레임. 이 규율이 실제 코드에서 어떤 모습인지, Phase 4 완성본 프로젝트(FabSim-P4-Complete)의 두 스크립트로 확인하자.

사례 1 — VehicleFleet: 96대를 매니저 하나가 움직인다

완성본 씬에서는 OHT 비히클 96대가 순환 트랙을 돈다. 비히클마다 Update를 가진 스크립트를 붙이는 대신, 매니저 하나가 배열을 순회하며 전부 이동시킨다. Unity가 스크립트의 Update를 호출하는 데는 건당 오버헤드가 있어서, 같은 일이라도 "Update 96번"보다 "Update 1번 + 루프 96회"가 훨씬 싸다. 

private void Update()
{
    // 문자열도, 새 배열도, 박싱도 없다 — 이 루프의 GC Alloc은 0이어야 한다.
    // Profiler의 GC Alloc 열에서 직접 확인해 볼 것.
    float travel = Time.time * speed;
    for (int i = 0; i < vehicles.Length; i++)
    {
        Transform vehicle = vehicles[i];
        float distance = Mathf.Repeat(travel + offsets[i], perimeter);
        Vector3 position = vehicle.localPosition;
        vehicle.localPosition = EvaluateTrack(distance, position.y);
    }
}

이 루프가 GC 0인 이유: 등장하는 타입이 전부 값 타입이다. float, int, Vector3는 스택에서 살다 사라지고, vehicles와 offsets 배열은 시작할 때 한 번 만들어 계속 재사용한다. 루프 안에 new도, 문자열도, LINQ도 없다. 96대가 매 프레임 움직여도 힙은 1바이트도 자라지 않는다.

사례 2 — FpsHud: 문자열은 필요할 때만 만든다

화면에 프레임 타임을 표시하는 HUD는 문자열 없이는 성립하지 않는다. 그렇다고 매 프레임 만들 이유는 없다 — 사람 눈은 초당 60번 갱신되는 숫자를 읽지 못한다. 완성본의 HUD는 0.25초에 한 번만 문자열을 만들고, 나머지 프레임에는 캐시된 문자열을 그리기만 한다. 

private void Update()
{
    accumulated += Time.unscaledDeltaTime;
    frames++;
    timer += Time.unscaledDeltaTime;
    if (timer < UpdateInterval)
    {
        return;
    }

    float averageMs = accumulated / frames * 1000f;
    float fps = frames / accumulated;
    builder.Length = 0;
    builder.Append("Frame ").Append(averageMs.ToString("F2")).Append(" ms  (")
        .Append(fps.ToString("F0")).Append(" FPS)");
    display = builder.ToString();
    accumulated = 0f;
    frames = 0;
    timer = 0f;
}

갱신 주기가 되기 전에는 return으로 즉시 빠져나가므로, 60fps 기준 대략 15프레임 중 14프레임은 덧셈 몇 번 외에 아무것도 하지 않는다. 문자열을 만드는 1프레임도 미리 만들어 둔 StringBuilder를 builder.Length = 0으로 비워 재사용하니 낭비가 최소다. 할당을 완전히 0으로 만들 수는 없는 코드지만(ToString은 결국 문자열을 만든다), "매 프레임"을 "4분의 1초마다"로 바꾸는 것만으로 할당 빈도가 15분의 1이 된다. 할당을 없앨 수 없다면 빈도를 낮추라는 규율의 전형이다.

DIGITAL TWIN 관점

관제실의 디지털 트윈은 게임처럼 몇 시간 플레이하고 끄는 앱이 아니라 24시간 내내 떠 있는 화면이다. 평균 fps가 아무리 좋아도 몇 분에 한 번씩 GC 스파이크로 화면이 멈칫하면, 운영자는 "장비가 순간이동했다"고 느끼고 시스템에 대한 신뢰를 잃는다. 장시간 운전에서는 작은 할당도 반드시 누적되어 GC를 깨우므로, 정상 상태 GC Alloc 0이라는 규율은 성능 취향이 아니라 제품 품질 그 자체다.

Memory Profiler — 존재만 알아 두기

내장 Profiler의 Memory 모듈은 "지금 총 메모리를 얼마나 쓰는가"의 실시간 추이를 보여 주는 수준이다. 더 깊은 분석 — 어떤 에셋이 메모리를 차지하는가, 같은 텍스처가 중복 로드되지는 않았는가, 씬을 오갈수록 메모리가 새는가 — 은 별도 패키지인 Memory Profiler가 담당한다. Package Manager에서 설치하며, 특정 시점의 메모리 스냅샷을 찍어 두 스냅샷을 나란히 비교하는 방식으로 쓴다. 이 로드맵의 실습 범위에서는 필요하지 않으니, "메모리 누수가 의심되면 꺼내는 전용 도구가 따로 있다"는 것만 기억하면 된다.

핵심 확인 질문

16.6ms라는 숫자는 어디서 나오며, 무엇을 의미하는가?

1000ms ÷ 60프레임 ≈ 16.6ms. 60fps를 유지하려면 스크립트·물리·렌더 준비·GPU 그리기까지 한 프레임의 모든 일을 이 시간 안에 끝내야 한다는 예산이다. 최적화란 이 예산 안에서 무엇이 초과 지출 중인지 찾아 줄이는 일이다.

CPU 바운드인지 GPU 바운드인지 Profiler에서 어떻게 가르는가?

CPU 타임라인에서 Gfx.WaitForPresent 같은 GPU 대기 항목이 크면 GPU 바운드다 — CPU는 일을 끝내고 GPU를 기다리는 중이다. 반대로 Update 계열이나 Camera.Render가 프레임을 채우고 있으면 CPU 바운드이며, 전자는 스크립트, 후자는 드로우콜 제출이 범인이다. 병목이 어느 쪽이냐에 따라 처방이 완전히 달라지므로 이 판별이 항상 먼저다.

정상 상태에서 GC Alloc이 0 B/프레임이어야 하는 이유는?

매 프레임의 작은 할당도 반드시 누적되어 언젠가 GC를 깨우고, GC는 하필 바쁜 프레임에 끼어들어 스파이크(프레임 튐)를 만든다. 평균 fps에는 거의 안 잡히지만 사용자 눈에는 명확히 보이는 문제이며, 24시간 도는 관제 트윈에서는 누적을 피할 방법이 없으므로 할당 자체를 없애는 것이 유일한 근본 해법이다.

에디터에서 잰 수치를 최종 성능 판정에 쓰면 안 되는 이유는?

에디터 자체의 비용(인스펙터, 씬 뷰, Profiler 창, 에디터 전용 검사)이 수치에 섞이고 코드 최적화 수준도 빌드와 다르기 때문이다. 에디터 측정은 수정 전후의 상대 비교와 경향 파악용으로 쓰고, 최종 판정은 실제 빌드를 대상 기기에서 프로파일링해서 내린다.

VehicleFleet은 왜 비히클 96대 각각에 Update를 두지 않고 매니저 하나로 움직이는가?

Unity가 스크립트의 Update를 호출하는 데는 건당 오버헤드가 있어서, 같은 작업이라도 "Update 96번"보다 "Update 1번 + 배열 루프 96회"가 훨씬 싸다. 루프 안은 값 타입 연산뿐이라 GC Alloc도 0이다. Station 03에서 다룰 "관리 주체를 하나로" 원칙의 최소 사례다.
