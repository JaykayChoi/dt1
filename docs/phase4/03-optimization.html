PHASE 4 — 성능 분석·최적화

Phase 4 · Station 03

최적화 기법과 실험장

Station 01에서 시간이 어디서 새는지 보는 법을, Station 02에서 드로우콜이 어떻게 묶이는지 보는 법을 배웠다. 이제 처방전을 편다 — 병목 유형별 대표 기법들을 익히고, 장비 192대와 비히클 96대가 준비된 실험장에서 가설을 세우고 직접 측정해 확인한다.

최적화의 순서: 측정 → 병목 → 처방

최적화는 기법 목록을 외워서 전부 적용하는 일이 아니다. 진단이 먼저다. 같은 "프레임이 느리다"라도 원인이 스크립트인지, 드로우콜 제출인지, GPU인지, 가비지 컬렉션인지에 따라 처방이 완전히 다르고, 틀린 유형에 쓴 처방은 효과가 0이다 — GPU가 병목인 씬에서 스크립트를 아무리 다듬어도 프레임은 1ms도 빨라지지 않는다. 그래서 순서는 언제나 같다: Station 01·02의 도구로 측정하고, 병목의 유형을 특정하고, 그 유형에 맞는 처방을 하나씩 적용하며 다시 측정한다.

병목 유형 대표 신호 (Station 01·02의 도구) 처방
CPU — 스크립트 Profiler CPU 타임라인에서 Update() 등 스크립트 마커가 지배적이고, 호출 수가 수백~수천에 달한다 호출 수 줄이기 · 매니저 통합 · 오브젝트 풀링
CPU — 렌더 준비 렌더 루프(드로우콜 제출·컬링) 비용이 지배적이고, Stats의 Batches가 수천 개다 배칭(SRP Batcher · GPU 인스턴싱) · 컬링(오클루전 · LOD)
GPU CPU는 한가한데 Gfx.WaitForPresent류의 GPU 대기가 길다. 해상도를 낮추면 즉시 빨라진다 해상도·포스트 프로세싱 비용 축소 · 오버드로 줄이기
GC — 메모리 GC Alloc이 매 프레임 0보다 크고, 주기적으로 GC.Collect 스파이크가 찍힌다 매 프레임 할당 제거 · 풀링 · 문자열/박싱/클로저 정리
조기 최적화 경계

측정 없는 최적화는 성능 대신 복잡도만 산다. "느낌상 느릴 것 같아서" 넣은 캐싱과 트릭은 대개 병목이 아닌 곳을 겨냥하고, 코드만 읽기 어렵게 만든다. 단, 예외가 하나 있다 — 아키텍처 결정이다. 매니저 통합처럼 코드의 구조 자체를 정하는 선택은 나중에 바꾸는 비용이 크므로, 검증된 패턴이라면 처음부터 그렇게 짓는다. 미세 최적화는 측정 후에, 구조 결정은 설계 때에.

기법 사전 — 병목 유형별 처방

위 표의 처방 중 이 프로젝트에서 실제로 쓰게 될 네 가지를 자세히 본다. 공통 원리는 하나다 — 일을 잘게 여러 번 하지 말고, 모아서 한 번에 하거나, 아예 하지 않는다.

매니저 통합 — Update 수백 개를 하나로

MonoBehaviour의 Update()는 공짜가 아니다. 엔진은 매 프레임 Update()를 가진 오브젝트를 전부 순회하며 하나씩 호출하는데, 이 호출마다 네이티브 코드와 매니지드(C#) 코드의 경계를 넘는 고정 오버헤드가 붙는다. 비히클 수백 대에 각각 이동 스크립트를 붙이면 이 오버헤드를 수백 번 지불한다. 처방은 관리 주체를 하나로 — 매니저 한 개가 Update()를 한 번만 받고, 배열을 순회하며 전부 움직인다. 실험장의 비히클 96대를 실제로 움직이는 코드가 정확히 이 패턴이다 (FabSim-P4-Complete/Assets/Scripts/VehicleFleet.cs): 

private void Update()
{
    // 문자열도, 새 배열도, 박싱도 없다 — 이 루프의 GC Alloc은 0이어야 한다.
    // Profiler의 GC Alloc 열에서 직접 확인해 볼 것.
    float travel = Time.time * speed;
    for (int i = 0; i < vehicles.Length; i++)
    {
        Transform vehicle = vehicles[i];
        float distance = Mathf.Repeat(travel + offsets[i], perimeter);
        Vector3 position = vehicle.localPosition;
        vehicle.localPosition = EvaluateTrack(distance, position.y);
    }
}

이 루프가 벌어들이는 것은 두 가지다. 첫째, 호출 오버헤드가 96번에서 1번으로 준다. 둘째, 루프 안에서 아무것도 할당하지 않으므로 GC Alloc이 0이다 — 새 배열도, 문자열도, 박싱도 없이 기존 배열을 읽고 값 타입만 계산한다. Profiler의 GC Alloc 열에서 이 주장을 직접 검증할 수 있다는 점이 이 코드의 교재로서의 가치다.

오브젝트 풀링 — 파괴 대신 반납

Instantiate()는 비싸고, Destroy()는 가비지를 만들어 나중에 GC 스파이크로 청구된다. 반송 시뮬레이터처럼 FOUP과 비히클이 끊임없이 생기고 사라지는 씬에서 매번 생성·파괴하면 프레임이 주기적으로 털린다. 처방은 풀(object pool) — 미리 만들어 둔 오브젝트를 빌려주고, 다 쓰면 파괴하는 대신 비활성화해서 돌려받는다. 

// 개념 스케치 — 생성·파괴 대신 빌려주고 돌려받는다.
private readonly Queue<GameObject> pool = new Queue<GameObject>();

public GameObject Get()
{
    GameObject instance = pool.Count > 0
        ? pool.Dequeue()
        : Object.Instantiate(prefab);
    instance.SetActive(true);
    return instance;
}

public void Return(GameObject instance)
{
    instance.SetActive(false);
    pool.Enqueue(instance);
}

생성은 풀이 비었을 때 한 번뿐이고, 이후에는 SetActive 토글만 반복된다 — 할당 0, 파괴 0. Phase 5에서 반송 명령마다 FOUP을 스폰할 때 이 패턴을 그대로 쓴다.

LOD — 멀리 있는 것은 대충 그린다

LOD(Level of Detail)는 카메라와의 거리에 따라 같은 오브젝트를 단계별로 단순한 메시로 바꿔 그리는 기법이다. Unity에서는 LOD Group 컴포넌트에 상세 메시(LOD0)부터 저폴리 메시(LOD1, LOD2…)까지 등록하면, 화면에서 차지하는 비율에 따라 자동으로 전환되고 마지막 단계 아래로는 아예 그리지 않는다(컬링). 지금의 실험장은 전부 큐브라 줄일 폴리곤이 없어 효과가 미미하지만, 나중에 실제 장비 에셋으로 형태를 채우는 순간 — 장비 한 대가 수만 폴리곤이 되는 순간 — 필수 기법이 된다.

오클루전 컬링 — 가려진 것은 아예 안 그린다

카메라 시야(절두체) 밖의 오브젝트를 걸러내는 프러스텀 컬링은 Unity가 항상 자동으로 해 준다. 오클루전 컬링(occlusion culling)은 한 걸음 더 나아가, 시야 안에 있지만 다른 물체에 가려진 오브젝트까지 걸러낸다. 미리 씬의 가림 관계를 계산해 두는 베이크가 필요하다 — Window > Rendering > Occlusion Culling 창에서 Bake를 누르면 Static 오브젝트 기준으로 가림 데이터가 구워진다. 벽으로 구획된 베이가 늘어선 팹 씬은 이 기법의 이상적인 고객이다: 통로에 서면 옆 베이의 장비 수십 대는 벽 뒤에 있고, 오클루전 컬링이 있어야 그들을 그리지 않을 수 있다.

다음 단계 — Jobs / Burst. 비히클이 수천 대가 되어 매니저 통합으로도 메인 스레드가 모자라면, 그다음 처방은 C# Job System으로 대량 Transform 갱신을 워커 스레드에 나눠 주고 Burst 컴파일러로 그 코드를 기계어 수준까지 최적화하는 것이다. 이 프로젝트의 범위 밖이므로 지금은 이름만 알아 둔다 — "매니저 통합 다음 단계가 있다"는 것만.

실험장: FabSim-P4-Complete

처방을 글로 배웠으니 이제 몸으로 확인할 차례다. C:\work\dt1\FabSim-P4-Complete에는 이 실험을 위해 만들어 둔 스트레스 씬 Assets/Scenes/StressField.unity가 들어 있다. 구성은:

  • 베이 6개 × 장비 192대 — 베이마다 좌우 16대씩 2열. 장비마다 발광 상태등이 하나씩 붙어 있다 (상태등도 192개).
  • 비히클 96대 — 팹 외곽을 도는 순환 트랙을 VehicleFleet 매니저 하나가 일괄 주행시킨다. 바닥·천장·레일까지 합치면 씬의 MeshRenderer는 488개.
  • 조명은 전부 Baked — 베이당 4개, 총 24개의 포인트 라이트가 라이트맵에 구워져 있어 런타임 조명 비용이 0이다 (Phase 3의 결론이 그대로 적용된 상태).
  • FpsHud — 화면 좌상단에 프레임 ms와 FPS를 표시한다. 문자열을 매 프레임 만들면 그 자체가 GC를 만들므로 0.25초에 한 번만 갱신한다 — HUD 코드 자체가 작은 교재다.
  • 모든 머티리얼에 GPU 인스턴싱이 켜져 있고, 움직이지 않는 지오메트리는 전부 Static이다.

아래 버튼으로 연다. 첫 번째는 Phase 3에서 직접 만든 실습 프로젝트(자기 씬을 직접 늘려 실험하고 싶을 때), 두 번째가 준비된 실험장이다.

버튼은 어떤 방식으로 사이트를 열었든 동작한다 — 파일로 직접(file://) 열었다면 브라우저가 처음 한 번 "열기 허용" 확인창을 띄우니 허용하면 된다. 프로젝트가 열리면 Assets/Scenes/StressField.unity를 연다.

실험 과제 — 가설 → 조작 → 측정

각 실험의 형식은 같다: 가설을 먼저 적고, 변수를 하나만 조작하고, 같은 지표(프레임 ms · Batches · GC Alloc)로 측정해 가설과 대조한다. 실험이 끝나면 조작한 것을 원상 복구하고 다음으로 넘어간다.

  1. 기준 측정 — 가설: "이 씬은 60fps(프레임당 16.6ms) 안에 든다." 조작: 없음. 측정: Play에 들어가 FpsHud의 프레임 ms, Game 뷰 Stats의 Batches, Profiler의 GC Alloc을 기록한다. 이 수치가 이후 모든 실험의 "전" 열이 된다 — 기준 없이 시작한 실험은 결과를 해석할 수 없다.
  2. 인스턴싱 끄기 — 가설: "인스턴싱을 끄면 드로우콜이 폭증하고 프레임이 느려진다." 조작: Assets/Materials의 머티리얼들을 선택해 Inspector에서 Enable GPU Instancing을 해제한다. 측정: Batches와 프레임 ms를 기준과 비교한다. 결과가 가설과 다를 수 있다 — URP에서는 SRP Batcher가 기본으로 켜져 있어 인스턴싱이 빠진 자리를 받아낼 수 있다(Station 02). 어느 배칭 경로가 실제로 쓰였는지 Frame Debugger로 확인해 "왜 이 수치가 나왔는가"까지 설명하는 것이 이 실험의 진짜 목표다. 끝나면 다시 켜 둔다.
  3. VehicleFleet를 일부러 나쁘게 — 가설: "비히클마다 개별 스크립트를 붙이면 매니저 통합보다 느리다." 조작: 사고 실험으로도 충분하다 — 비히클 96대에 각각 Update()를 가진 이동 스크립트가 붙어 있다면 엔진의 스크립트 호출은 프레임당 96번, 매니저 방식은 1번이다. 직접 확인하고 싶다면 같은 트랙 수식을 개별 MonoBehaviour로 옮긴 스크립트를 만들어 전 비히클에 붙이고 VehicleFleet를 비활성화한 뒤 비교한다. 측정: Profiler CPU 타임라인에서 스크립트 항목의 호출 수와 합계 시간. 96대에서는 차이가 작게 보일 수 있다 — 다음 실험과 결합해 384대로 늘리면 차이가 뚜렷해진다.
  4. 비히클 수 늘리기 — 가설: "비히클을 4배로 늘리면 ___가 가장 먼저 무너진다" (빈칸을 먼저 채우고 시작할 것). 조작: 정석은 Assets/Editor/StressSceneBuilder.cs의 VehicleCount 상수를 96에서 384로 바꿔 씬을 재빌드하는 것이지만, 더 간단한 방법이 있다 — 씬에서 Vehicles (Dynamic) 오브젝트를 복제(Ctrl+D)하면 96대씩 늘어난다. VehicleFleet 매니저도 함께 복제되므로 복제본도 스스로 달린다. 같은 트랙 위 같은 위치라 겹쳐 보이니 복제본의 Y를 0.5씩 올려 층을 나눈다. 측정: 96 → 192 → 384로 늘려 가며 프레임 ms, Batches, GC Alloc 중 어느 지표가 먼저 한계에 닿는지 기록한다. "장비를 늘리면 어디가 먼저 무너지는가"는 Phase 5 시뮬레이터의 규모를 정할 때 그대로 다시 묻게 될 질문이다.

60fps 체크리스트

Phase 4를 통과하는 씬의 조건을 요약한다. Phase 5의 시뮬레이터는 이 상태를 유지한 채로 그 위에 로직을 쌓는다.

  • Realtime 광원 0개 — 조명은 전부 Baked, 조명의 런타임 비용은 0 (Phase 3).
  • GC Alloc 0 B/frame — 정상 주행 중 매 프레임 할당이 없다. 스파이크는 스폰 같은 이벤트 순간에만.
  • 배칭 동작 확인 — GPU 인스턴싱(또는 SRP Batcher)이 실제로 일하고 있음을 Frame Debugger로 확인했다.
  • Static 지정 완료 — 움직이지 않는 모든 지오메트리가 Static이다 (배칭·라이트맵·오클루전의 전제).
  • 포스트 프로세싱 비용 인지 — Bloom·Tonemapping이 프레임에서 몇 ms를 쓰는지 알고 켜 두고 있다.
  • 모든 최적화에 병목 근거 보유 — 적용한 기법마다 "어느 측정이 이 처방을 시켰는가"를 기록으로 답할 수 있다.

Phase 5의 미니 팹 시뮬레이터는 이 기법들 위에서 돈다. DES 코어가 계산한 비히클 위치를 화면에 그리는 VehicleFleetView가 바로 VehicleFleet의 매니저 통합 패턴의 재사용이고, 반송 명령마다 생기는 FOUP은 풀에서 빌려 온다. 이번 스테이션에서 측정으로 검증한 패턴들이 다음 Phase에서는 검증 없이 믿고 쓰는 기반이 된다 — 그것이 실험을 먼저 하는 이유다.

핵심 확인 질문

최적화가 왜 "측정 → 병목 → 처방" 순서여야 하는가?

병목 유형(CPU 스크립트 / CPU 렌더 준비 / GPU / GC)마다 처방이 완전히 다르고, 틀린 유형에 쓴 처방은 효과가 0이기 때문이다. GPU가 병목인 씬에서 스크립트를 최적화하면 코드만 복잡해지고 프레임은 그대로다. 측정이 병목 유형을 특정해야 비로소 어떤 처방이 유효한지 알 수 있다.

매니저 통합이 오브젝트별 Update보다 빠른 이유는?

엔진이 MonoBehaviour의 Update()를 호출할 때마다 네이티브-매니지드 경계를 넘는 고정 오버헤드가 붙는데, 오브젝트 N개면 이 비용을 N번 지불한다. 매니저 통합은 호출을 1번으로 줄이고, 배열을 연속으로 순회하므로 캐시 효율도 좋다. 덤으로 VehicleFleet처럼 루프 안에서 아무것도 할당하지 않게 짜면 GC Alloc 0까지 함께 얻는다.

오브젝트 풀링은 어느 병목 유형의 처방이며, 왜 효과가 있는가?

주로 GC(메모리) 병목의 처방이다. Instantiate는 그 자체로 비싸고, Destroy는 가비지를 만들어 나중에 GC.Collect 스파이크로 청구된다. 풀은 생성·파괴를 SetActive 토글로 바꿔 반복 스폰 상황에서 할당과 파괴를 모두 0으로 만든다.

프러스텀 컬링과 오클루전 컬링의 차이는?

프러스텀 컬링은 카메라 시야 밖의 오브젝트를 걸러내며 항상 자동으로 동작한다. 오클루전 컬링은 시야 안이지만 다른 물체에 가려진 오브젝트까지 걸러내며, 미리 가림 데이터를 베이크해야 동작한다. 벽으로 구획된 베이가 늘어선 팹 씬은 가려지는 물체가 많아 오클루전 컬링의 이득이 특히 크다.

"조기 최적화는 피하라"면서 매니저 통합은 왜 처음부터 적용하는가?

미세 최적화와 아키텍처 결정을 구분하기 때문이다. 캐싱 같은 미세 최적화는 측정으로 병목이 확인된 뒤에 넣어도 늦지 않지만, "이동 로직을 매니저가 가질 것인가, 오브젝트마다 가질 것인가" 같은 구조 결정은 나중에 뒤집는 비용이 크다. 검증된 구조 패턴은 처음부터 채택하고, 나머지는 측정이 시킬 때만 손댄다.
