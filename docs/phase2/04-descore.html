PHASE 2 — 이산사건시뮬레이션(DES)

Phase 2 · Station 04

C# DES 엔진 설계 — DesCore

SimPy로 DES의 감을 잡았으니 이제 같은 것을 C#으로 직접 만든다. Phase 5의 미니 팹 시뮬레이터는 Unity 위에서 돌아야 하고, 파이썬인 SimPy는 거기까지 따라올 수 없기 때문이다. 다행히 DES 엔진의 본질은 의외로 작다 — 부품 네 개, 수백 줄이면 충분하다.

왜 엔진을 직접 만드나

Station 03에서 확인했듯 SimPy는 훌륭한 학습 도구지만, 결국 파이썬 라이브러리다. 우리의 종착지는 Unity(C#) 안에서 도는 팹 물류 시뮬레이터이므로, 시뮬레이션의 심장 — 클록, 사건 큐, 자원 — 은 C#으로 있어야 한다. 상용 DES 도구를 붙이는 길도 있지만, Station 01에서 본 것처럼 DES 엔진의 뼈대는 "가장 이른 사건을 꺼내 클록을 점프시키고 실행한다"는 루프 하나다. 이 정도면 사서 쓰는 것보다 만들어 이해하는 편이 남는 장사다.

엔진 이름은 DesCore. 부품은 넷이고, 전부 합쳐 300줄 남짓이다.

DesCore 부품 SimPy 대응물 역할
SimEvent 이벤트 객체 (내부) "이 시각에 이 동작이 일어난다" — 사건 하나
EventQueue Environment 내부의 이벤트 힙 FEL — 사건을 시각 순으로 꺼내는 우선순위 큐
Simulation simpy.Environment 시뮬레이션 클록 + 사건 실행 루프
SimResource simpy.Resource 용량 제한 자원 — 점유, 대기열, 가동률 통계

설계에는 세 가지 제약을 걸었다. 전부 "Unity에 그대로 들고 가기 위해서"다.

  • UnityEngine 무의존 — 순수 C# 클래스 라이브러리로 만든다. 덕분에 Unity 없이 콘솔에서 몇 초 만에 돌려 보며 로직을 검증할 수 있고, 검증이 끝난 파일을 Unity 프로젝트에 복사하면 끝이다.
  • C# 9 문법까지만 — Unity의 컴파일러가 지원하는 수준에 맞춘다. file-scoped namespace 같은 이후 문법은 쓰지 않는다.
  • BCL 최소 사용 — .NET 6부터 제공되는 PriorityQueue<TElement, TPriority>는 Unity의 .NET 프로파일에 없다. 그래서 우선순위 큐(이진 최소 힙)를 직접 구현한다.
DIGITAL TWIN 관점

이 페이지의 코드는 연습용 스케치가 아니다. DesCore 폴더의 파일들이 Phase 5에서 그대로 Unity 프로젝트에 복사되어 미니 팹 시뮬레이터의 코어가 된다. 오늘 쓴 코드가 최종 코어다 — 그래서 의존성 제약을 엄격하게 지킨 것이다.

부품 ① 사건과 FEL — SimEvent · EventQueue

SimEvent — 시각, 일련번호, 동작

사건은 세 가지만 담는다. 언제(시뮬레이션 시각), 몇 번째로 예약됐는지(일련번호), 무엇을 하는지(동작). 

/// <summary>
/// 시뮬레이션에서 특정 시각에 실행될 하나의 사건.
/// </summary>
public sealed class SimEvent
{
    /// <summary>사건이 실행될 시뮬레이션 시각.</summary>
    public double Time { get; }

    /// <summary>같은 시각에 예약된 사건들의 실행 순서를 보장하는 일련번호.</summary>
    public long Sequence { get; }

    /// <summary>사건이 실행할 동작.</summary>
    public Action Action { get; }

시각과 동작은 당연한데, 일련번호는 왜 필요할까? 같은 시각에 예약된 사건이 두 개 있다고 하자 — 예를 들어 t=100.0에 "로트 3 가공 완료"와 "로트 7 투입". 힙은 시각이 같은 원소들 사이의 순서를 보장하지 않으므로, 일련번호가 없으면 이 둘의 실행 순서가 자료구조 내부 사정에 따라 흔들린다. 실행 순서가 흔들리면 같은 난수 시드로 돌려도 결과가 달라질 수 있다 — 재현이 안 되는 시뮬레이터는 디버깅이 불가능하다. 그래서 순서 비교는 (시각, 일련번호)의 사전식 비교로 한다. 시각이 같으면 먼저 예약된 사건이 먼저 실행된다. 

/// <summary>이 사건이 other보다 먼저 실행되어야 하면 true를 반환한다.</summary>
public bool IsBefore(SimEvent other)
{
    if (Time != other.Time)
    {
        return Time < other.Time;
    }

    return Sequence < other.Sequence;
}

EventQueue — 이진 최소 힙으로 만든 FEL

FEL(Future Event List, 미래 사건 목록)에 필요한 연산은 둘뿐이다. "사건을 넣는다", "가장 이른 사건을 꺼낸다". 이것을 빠르게 해 주는 자료구조가 이진 최소 힙(binary min-heap) — 배열로 표현한 완전 이진 트리로, 불변식은 단 하나다: 부모는 항상 자식보다 이르다. 이 불변식만 지키면 배열의 첫 원소 heap[0]이 언제나 전체에서 가장 이른 사건이 된다. 

/// <summary>사건을 큐에 넣는다. O(log n).</summary>
public void Push(SimEvent item)
{
    heap.Add(item);
    SiftUp(heap.Count - 1);
}

Push는 새 사건을 배열 끝에 붙인 뒤, 부모보다 이르면 부모와 자리를 바꾸며 위로 올라간다(SiftUp). 부모보다 늦어지는 순간 멈춘다. 

private void SiftUp(int index)
{
    while (index > 0)
    {
        int parent = (index - 1) / 2;
        if (!heap[index].IsBefore(heap[parent]))
        {
            break;
        }

        Swap(index, parent);
        index = parent;
    }
}

Pop은 반대다. 루트(가장 이른 사건)를 꺼내고, 배열의 마지막 원소를 루트 자리에 올린 뒤, 두 자식 중 더 이른 쪽과 자리를 바꾸며 아래로 내려보낸다(SiftDown). 트리 높이만큼만 움직이므로 둘 다 O(log n) — 사건이 수만 개 쌓여도 넣고 꺼내는 데 십수 번의 비교면 끝난다.

부품 ② 엔진 루프 — Simulation

엔진의 심장인데, 파일 전체가 이게 전부다. 

using System;

namespace DesCore
{
    /// <summary>
    /// DES 엔진의 심장 — 시뮬레이션 클록과 사건 루프.
    /// "가장 이른 사건을 꺼내고, 클록을 그 시각으로 점프시키고, 실행한다"의
    /// 반복이 엔진의 전부다.
    /// </summary>
    public sealed class Simulation
    {
        private readonly EventQueue eventQueue = new EventQueue();
        private long nextSequence;

        /// <summary>현재 시뮬레이션 시각.</summary>
        public double Now { get; private set; }

        /// <summary>지금까지 실행한 사건 수.</summary>
        public long ExecutedCount { get; private set; }

        /// <summary>delay 시간 뒤에 action이 실행되도록 예약한다.</summary>
        public void Schedule(double delay, Action action)
        {
            if (delay < 0.0)
            {
                throw new ArgumentOutOfRangeException(
                    nameof(delay), "Cannot schedule an event in the past.");
            }

            eventQueue.Push(new SimEvent(Now + delay, nextSequence++, action));
        }

        /// <summary>until 시각까지의 사건을 모두 실행하고 클록을 until로 맞춘다.</summary>
        public void Run(double until)
        {
            while (!eventQueue.IsEmpty && eventQueue.PeekTime <= until)
            {
                SimEvent next = eventQueue.Pop();
                Now = next.Time;
                ExecutedCount++;
                next.Action();
            }

            Now = until;
        }
    }
}

Station 01에서 봤던 DES 루프의 의사코드 — "FEL에서 가장 이른 사건을 꺼내고, 클록을 그 시각으로 점프시키고, 사건을 실행한다" — 가 Run의 while 루프 몇 줄로 그대로 옮겨졌다. 클록이 사건과 사건 사이를 점프한다는 것, 사건이 없는 구간은 계산 자체를 하지 않는다는 것이 코드에서 명확히 보인다.

  • Schedule(delay, action) — 상대 지연을 절대 시각 Now + delay로 바꾸고 일련번호를 발급해 힙에 넣는다. 음수 지연(과거 예약)은 예외로 막는다.
  • Run(until) — until 이전의 사건만 실행하고, 사건이 남았든 없든 클록을 정확히 until에 맞춘다. "끝까지 한 번에"(콘솔 데모)도, "한 프레임어치만"(Unity)도 같은 메서드 하나로 처리된다 — 이 시그니처가 뒤에서 볼 Unity 연동의 열쇠다.

부품 ③ 자원 — 코루틴 대신 콜백

SimPy에서 가장 매끄러웠던 부분을 떠올려 보자. yield req 한 줄이면 프로세스 함수가 그 자리에서 멈췄다가, 자원을 얻는 순간 같은 자리에서 재개됐다. 파이썬 제너레이터(코루틴)가 "멈춤과 재개"를 언어 차원에서 지원하기 때문이다. C#에도 이터레이터나 async/await 같은 장치가 있지만, 이것들을 시뮬레이션 클록 위에서 돌게 만들려면 커스텀 스케줄러라는 별도의 기계장치가 필요하다. 부품 네 개짜리 엔진의 단순함을 지키기 위해 DesCore는 가장 원초적인 방식을 택했다 — 콜백(callback). "자원을 얻으면 이 함수를 불러 달라"고 함수 자체를 넘기는 것이다. 

/// <summary>
/// 자원을 요청한다. 여유가 있으면 onGranted가 즉시 호출되고,
/// 없으면 대기열에 들어가 차례가 왔을 때 호출된다.
/// </summary>
public void Request(Action onGranted)
{
    if (onGranted == null)
    {
        throw new ArgumentNullException(nameof(onGranted));
    }

    if (InUse < Capacity)
    {
        AccumulateBusyArea();
        InUse++;
        onGranted();
    }
    else
    {
        waiters.Enqueue(onGranted);
    }
}

/// <summary>
/// 자원을 반납한다. 대기자가 있으면 점유 수를 유지한 채
/// 다음 대기자에게 바로 넘겨준다.
/// </summary>
public void Release()
{
    if (InUse == 0)
    {
        throw new InvalidOperationException($"{Name}: released more than requested.");
    }

    if (waiters.Count > 0)
    {
        Action next = waiters.Dequeue();
        next();
    }
    else
    {
        AccumulateBusyArea();
        InUse--;
    }
}

Release의 분기를 눈여겨보자. 대기자가 있으면 InUse를 줄였다가 다시 늘리는 대신, 점유 수를 유지한 채 다음 대기자의 콜백을 바로 호출한다 — 자원의 핸드오프다. 덕분에 자원이 잠깐이라도 "비어 있는" 순간이 생기지 않아 가동률 통계가 왜곡되지 않는다.

그 가동률은 시간 적분으로 계산한다. 점유 수가 바뀌는 순간마다 "직전 점유 수 × 경과 시간"을 면적으로 누적해 두고, 마지막에 전체 시간 × 용량으로 나누면 평균 가동률이 된다. 

private void AccumulateBusyArea()
{
    // 가동률은 "사용 중 수 × 경과 시간"의 시간 적분으로 계산한다.
    busyArea += InUse * (simulation.Now - lastChangeTime);
    lastChangeTime = simulation.Now;
}

같은 흐름, 두 표현 — "스테이션에서 가공한다"

이제 두 엔진의 차이가 가장 선명하게 드러나는 지점이다. "큐에 서고 → 자원을 얻으면 가공을 시작하고 → 가공 시간이 지나면 자원을 반납한다"는 동일한 흐름을 SimPy와 DesCore로 나란히 놓아 보자. 먼저 SimPy — 함수 하나를 위에서 아래로 읽으면 끝이다. 

def process_at(env, lot_id, stations, name):
    """스테이션 name에서 큐 대기 후 가공한다."""
    queued = env.now
    with stations[name].request() as req:
        yield req
        station_waits[name].append(env.now - queued)
        log(env, lot_id, f"{name} 가공 시작 (대기 {env.now - queued:.1f}분)")
        service = random.expovariate(1 / SERVICE_MEAN[name])
        busy[name] += service
        yield env.timeout(service)
    log(env, lot_id, f"{name} 가공 완료")

같은 흐름이 DesCore에서는 메서드 세 개의 연쇄가 된다. "대기가 끝나면 BeginService를, 가공이 끝나면 FinishService를 불러 달라"고 다음 단계를 콜백으로 넘긴다. 

private static void EnterStation(Lot lot, int stage)
{
    lot.QueuedAt = simulation.Now;
    stations[stage].Request(() => BeginService(lot, stage));
}

private static void BeginService(Lot lot, int stage)
{
    double waited = simulation.Now - lot.QueuedAt;
    stationWaits[stage].Add(waited);
    Log(lot, $"{StageNames[stage]} 가공 시작 (대기 {waited:F1}분)");
    double service = random.NextExponential(ServiceMeans[stage]);
    simulation.Schedule(service, () => FinishService(lot, stage));
}

private static void FinishService(Lot lot, int stage)
{
    stations[stage].Release();
    Log(lot, $"{StageNames[stage]} 가공 완료");
    if (stage == StageNames.Length - 1)
    {
        CompleteLot(lot);
    }
    else
    {
        RequestTransport(lot, stage);
    }
}

SimPy의 yield req가 하던 "여기서 멈췄다 재개"를, DesCore에서는 Request(() => BeginService(...))가 한다. SimPy의 yield env.timeout(service)는 Schedule(service, () => FinishService(...))가 된다. 의미는 완전히 같고, 흐름을 잇는 주체가 언어(코루틴)에서 프로그래머(콜백)로 바뀌었을 뿐이다.

콜백 지옥 주의

이 방식의 비용도 정직하게 봐야 한다. SimPy에서는 로트의 전체 여정이 lot_flow 한 함수에 위에서 아래로 읽혔지만, DesCore에서는 투입 → 가공 → 반송 → 가공 → … 이 여섯 개 메서드에 흩어져 있고 흐름은 람다를 따라가야 보인다. 단계가 지금의 두세 배로 늘면 — 픽업, 이동, 드롭, 충전, 재배차 — 콜백 연쇄는 급격히 읽기 어려워진다. Phase 5에서 장비를 상태 기계(state machine)로 정리하는 것이 이 구조의 다음 진화다.

수치가 완전히 같지는 않다 — 파이썬과 .NET의 난수 생성기가 다르므로 같은 시드를 줘도 뽑히는 표본 경로가 다르고, 따라서 완전히 같을 수가 없다. 봐야 할 것은 통계적 결론의 일치다. 병목은 두 엔진 모두 스테이션 B(가동률 약 0.8, 큐 대기 13~14분)이고, 반송차 2대는 두 엔진 모두 여유(가동률 약 0.35, 배차 대기 거의 0)다. 처리량과 리드타임도 수 퍼센트 안에서 겹친다. 서로 독립적으로 구현된 두 모델이 같은 결론을 내면 로직이 맞다는 강한 증거가 된다 — 이런 교차 검증이 시뮬레이터를 만들 때마다 반복하게 될 기본기다.

출력의 첫 줄도 그냥 지나치지 말자. 사건 2,137개가 2,000분어치 시뮬레이션의 전부다. 로트 354개가 태어나고, 가공되고, 실려 가고, 완성되는 모든 일이 계산 2,137번으로 끝났다. 만약 0.1분 간격의 고정 스텝으로 돌렸다면 2만 번을 깨어나 대부분 "아무 일도 없음"만 확인했을 것이다 — 상태가 바뀌는 순간만 계산하는 사건 기반 방식의 효율이 숫자로 보인다.

Unity로 가는 길 — Phase 5 예고

이 엔진 설계의 숨은 핵심은 시뮬레이션 시간과 벽시계 시간의 분리다. Simulation은 "현실의 1초"를 전혀 모른다. Run(until)에 "어디까지 진행할지"만 알려주면 되고, 그 until을 누가 어떻게 주느냐는 전적으로 바깥의 사정이다.

  • 콘솔 데모 — simulation.Run(2000.0) 한 번. 2,000분어치가 몇 밀리초 만에 끝까지 달린다.
  • Unity — 매 프레임 조금씩 민다. 렌더링과 시뮬레이션이 나란히 흐르게 된다.
public sealed class SimulationRunner : MonoBehaviour
{
    private readonly Simulation simulation = new Simulation();

    [Tooltip("배속 — 60이면 현실 1초에 시뮬레이션 60초(1분)가 흐른다.")]
    [SerializeField]
    private float speed = 60f;

    private void Update()
    {
        simulation.Run(simulation.Now + Time.deltaTime * speed);
    }
}

이게 전부다. Time.deltaTime × 배속만큼 클록을 밀어 주면 그 구간의 사건들이 실행되고, OHT의 화면 위 위치는 사건과 사건 사이를 보간해 그리면 된다. speed를 키우면 빨리 감기, 0으로 만들면 일시 정지 — 엔진 코드는 한 줄도 바뀌지 않는다. Run(until) 시그니처로 설계한 이유가 바로 이것이다.

빌드와 실행

코드는 examples/phase2-descore/에 있고, 구조는 둘로 나뉜다.

  • DesCore/ — 엔진 라이브러리. 부품 네 개 (SimEvent.cs, EventQueue.cs, Simulation.cs, SimResource.cs)에 지수분포 샘플링 확장(RandomExtensions.cs — SimPy의 random.expovariate 대응)을 더한 다섯 파일. 외부 패키지 의존성은 0이다.
  • DesCore.Demo/ — 콘솔 데모. 미니 반송 라인 시나리오 전체가 Program.cs 한 파일이다.

.NET SDK만 있으면 바로 돈다.

cd examples/phase2-descore
dotnet run --project DesCore.Demo

앞쪽 로트 두 개의 사건 로그가 먼저 찍히고 — Station 03에서 본 SimPy 로그와 같은 형식이다 — 이어서 위에서 인용한 통계가 출력된다.

이 엔진의 응용 — examples/fleet-dispatch

같은 DesCore를 재사용하는 배차 정책 실험 랩이 저장소에 하나 더 있다. 엔진(클록·이벤트 루프)은 그대로 두고 그 위에 반송차·명령· 배차 규칙만 얹어, DES로 배차 정책을 측정한다 — Phase 1의 Fleet Management 개념과 이 페이지의 엔진이 만나는 자리다. 세 배차 규칙을 부하별로 비교한다: examples/fleet-dispatch.

핵심 확인 질문

SimEvent에 일련번호(Sequence)는 왜 필요한가?

같은 시각에 예약된 사건들의 실행 순서를 결정적으로 만들기 위해서다. 힙은 시각이 같은 원소들의 순서를 보장하지 않으므로, (시각, 일련번호)의 사전식 비교로 "같은 시각이면 먼저 예약된 것 먼저"를 강제한다. 이것이 없으면 같은 시드로 돌려도 결과가 달라질 수 있고, 재현이 안 되는 시뮬레이터는 디버깅할 수 없다.

.NET에 PriorityQueue가 있는데 왜 힙을 직접 구현했나?

PriorityQueue<TElement, TPriority>는 .NET 6부터 제공되는데, Unity의 .NET 프로파일에는 없다. DesCore는 Phase 5에서 Unity 프로젝트에 그대로 복사되어야 하므로, Unity가 지원하는 C# 9 + BCL 범위 안에서 이진 최소 힙을 직접 구현했다. 필요한 연산이 Push/Pop 두 개뿐이라 구현도 100줄 남짓이다.

SimPy의 yield와 DesCore의 콜백 — 무엇이 같고 무엇이 다른가?

의미는 같다: "자원을 얻을 때까지(또는 시간이 흐를 때까지) 기다렸다가 다음 단계를 계속한다". 다른 것은 표현이다. SimPy는 코루틴이라 흐름이 한 함수 안에 위에서 아래로 이어지고 멈춤·재개를 언어가 처리한다. DesCore는 "그다음 할 일"을 함수(콜백)로 넘겨 흐름을 프로그래머가 직접 잇는다. 단계가 많아지면 콜백 연쇄는 읽기 어려워지는 비용이 있다.

두 엔진의 수치가 완전히 일치하지 않는데 왜 "검증됐다"고 할 수 있나?

파이썬과 .NET의 난수 생성기가 다르므로 표본 경로가 다른 것이 정상이고, 수치의 완전 일치는 애초에 기대할 수 없다. 검증의 기준은 통계적 결론의 일치다 — 병목 위치(둘 다 스테이션 B), 가동률 수준 (B 약 0.8, 반송차 약 0.35), 처리량·리드타임의 근접(수 퍼센트 이내). 독립적으로 구현된 두 모델이 같은 결론을 내면 모델 로직이 맞다는 강한 증거다.

시뮬레이션 시간과 실시간(벽시계)은 어떻게 연결되는가?

연결하지 않고 분리하는 것이 원칙이다. 엔진은 Run(until)로 "어디까지 진행할지"만 받는다. 콘솔에서는 한 번에 끝까지 (Run(2000)), Unity에서는 매 프레임 Run(Now + Time.deltaTime × 배속)으로 조금씩 민다. 엔진은 그 차이를 모르므로 배속·일시정지·빨리 감기가 엔진 수정 없이 바깥에서 구현된다.
