SimPy 실습 — 미니 반송 라인

Station 01에서 사건 큐를, Station 02에서 대기행렬 공식을 배웠다. 이번에는 그 둘을 코드로 굴린다. SimPy로 단일 스테이션(M/M/1)을 만들어 이론값과 맞춰 보고, 스테이션 3개 + 반송차 2대짜리 미니 반송 라인에서 병목이 숫자로 드러나는 것을 직접 본다.

SimPy — 프로세스로 쓰는 DES

Station 01에서 본 DES의 심장은 사건 루프였다 — 미래 사건 목록(FEL)에서 가장 이른 사건을 꺼내고, 클록을 점프시키고, 사건을 처리한다. 이 루프를 매번 손으로 짜는 대신, SimPy는 프로세스 지향(process-oriented)이라는 더 편한 서술 방식을 제공한다. 로트든 반송차든, 개체 하나의 생애를 "절차"로 쓰면 엔진이 그 절차를 사건들로 바꿔 굴려 준다.

이를 가능하게 하는 것이 파이썬의 제너레이터(generator, 코루틴)다. 제너레이터 함수는 yield에서 실행을 멈췄다가 나중에 그 지점부터 재개할 수 있다. SimPy에서 yield는 곧 "시간이 흐르는 지점"이다 — 프로세스가 yield로 제어를 넘기면, 엔진은 FEL에서 다음 사건을 꺼내 클록을 점프시키고, 때가 된 프로세스를 깨워 이어서 실행한다. 우리가 절차문을 쓰는 동안, 밑에서는 Station 01의 사건 루프가 그대로 돌고 있는 것이다.

설치는 한 줄이면 된다. SimPy는 순수 파이썬 라이브러리라 다른 의존성이 없다. 

pip install simpy

핵심 API 세 가지

이 스테이션의 예제 전부가 다음 세 가지 API로만 만들어져 있다. 이 셋만 알면 SimPy 모델을 읽고 쓸 수 있다.

env.timeout(t) — 시간이 흐른다

"지금부터 t만큼 시뮬레이션 시간이 지난 뒤 나를 깨워라"라는 사건을 FEL에 등록한다. 가공, 이동, 도착 간격 — 시간이 걸리는 모든 일이 이걸로 표현된다. 

yield env.timeout(4.0)   # 시뮬레이션 시간 4.0분 경과 (실제 CPU는 기다리지 않는다)

Resource.request() — 자원을 점유한다

용량이 정해진 자원(장비, 반송차)의 점유를 요청한다. 자원이 차 있으면 요청은 큐에 줄을 서고, 프로세스는 yield req에서 멈춘다 — Station 01·02에서 배운 "큐 대기"가 바로 여기서 자동으로 생긴다. 

with machine.request() as req:
    yield req            # 장비가 빌 때까지 큐에서 대기
    ...                  # 점유 상태로 작업 — with 블록을 나가면 자동 반납

env.process(...) — 프로세스를 등록한다

제너레이터를 "살아 있는 프로세스"로 엔진에 등록한다. 로트 투입, 반송차 운행 등 시뮬레이션 속에서 스스로 진행되는 모든 주체가 이걸로 태어난다. 

env.process(lot(env, machine))   # 로트 하나의 생애가 시작된다

예제 ① — 단일 스테이션 (M/M/1)

첫 예제는 Station 02에서 공식으로만 다룬 M/M/1을 코드로 재현한다. 로트가 평균 1.25분 간격으로 랜덤하게 도착하고(λ = 0.8/분), 장비 1대가 평균 1.0분의 랜덤한 시간으로 처리한다(μ = 1.0/분). 모델의 전부가 함수 두 개다. 

def lot(env, machine):
    """로트 하나의 생애: 도착 → 큐 대기 → 가공 → 퇴장."""
    global busy_time
    arrived = env.now
    with machine.request() as req:
        yield req  # 장비가 빌 때까지 큐에서 대기
        wait_times.append(env.now - arrived)
        service = random.expovariate(1 / SERVICE_MEAN)
        busy_time += service
        yield env.timeout(service)  # 가공


def source(env, machine):
    """포아송 도착원: 지수분포 간격으로 로트를 계속 투입한다."""
    while True:
        yield env.timeout(random.expovariate(1 / ARRIVAL_MEAN))
        env.process(lot(env, machine))

줄 단위로 읽어 보면 이렇다.

  • arrived = env.now — env.now가 시뮬레이션 클록이다. 도착 시각을 기억해 둔다.
  • yield req — 장비가 차 있으면 이 줄에서 프로세스가 멈춘다. 얼마나 멈추는지가 곧 큐 대기시간이고, 깨어난 직후의 env.now - arrived가 그 측정값이다.
  • yield env.timeout(service) — 지수분포에서 뽑은 가공 시간만큼 시간이 흐른다. with 블록이 끝나면 장비는 자동 반납되고, 큐의 다음 로트가 깨어난다.
  • source — 무한 루프로 "지수분포 간격만큼 기다렸다가 로트 하나 투입"을 반복한다. 이것이 포아송 도착 과정이다.

조립은 네 줄이다: simpy.Environment()로 환경을 만들고, simpy.Resource(env, capacity=1)로 장비를 만들고, env.process(source(...))로 도착원을 등록한 뒤 env.run(until=SIM_TIME)으로 20,000분을 굴린다. 실행 결과는 이렇다. 

처리 로트 수          : 16,084
가동률 ρ              : 측정 0.802 | 이론 0.800
평균 큐 대기 Wq [분]  : 측정 4.39 | 이론 4.00

이 세 줄이 이 예제의 존재 이유다. "이 시뮬레이터를 믿을 수 있는가?"라는 질문에 대한 답이 바로 이 검증(validation)이기 때문이다. 가동률은 이론값 0.800에 거의 정확히 붙었고, 평균 큐 대기도 Station 02의 공식 Wq = ρ/(μ−λ) = 4.00분과 같은 자리에서 만난다(4.39분 — 남는 오차는 유한한 실행 길이와 하나의 난수 시드에서 오는 통계적 요동이다). 이론 해가 존재하는 단순한 경우로 먼저 시뮬레이터를 맞춰 보고, 그다음에 이론이 없는 복잡한 시스템으로 확장한다 — 시뮬레이션 실무의 기본 수순이다.

초기 과도 상태 주의

시뮬레이션은 텅 빈 시스템에서 시작한다. 초반의 로트들은 큐가 아직 자라기 전이라 비정상적으로 대기가 짧고, 이 워밍업 구간(warm-up transient)이 통계에 그대로 섞인다. 짧게 돌리면 이 왜곡이 평균을 지배해 이론값과 크게 어긋난다. 예제들이 20,000분, 200,000분씩 길게 굴리는 이유가 이것이다 — 정상 상태 표본을 충분히 쌓아 초반의 왜곡을 희석하는 것.

예제 ② — 미니 반송 라인

두 번째 예제는 팹 물류의 뼈대를 최소 구성으로 재현한다. 로트가 A → B → C 세 공정을 순서대로 거치고, 공정 사이 이동은 반송차 2대(OHT/AGV를 추상화한 리소스)가 맡는다. "가공하고, 기다리고, 실려 가는" 흐름 전체다. 파라미터는 파일 상단에 모여 있다. 

ARRIVAL_MEAN = 6.0            # 평균 도착 간격 [분]
SERVICE_MEAN = {"A": 4.0, "B": 5.0, "C": 3.0}  # 스테이션별 평균 가공 시간 [분]
TRAVEL_TIME = 2.0             # 스테이션 간 반송 소요 [분]
VEHICLE_COUNT = 2             # 반송차 대수

코드의 뼈대는 헬퍼 두 개 + 여정 하나다. process_at은 예제 ①의 lot과 같은 패턴으로 "스테이션에서 큐 대기 후 가공"을, transport는 "반송차를 배차받아 이동"을 맡는다. 스테이션도 반송차도 똑같은 simpy.Resource다 — 장비든 탈것이든, 자리를 두고 줄을 서게 만드는 것은 전부 리소스라는 것이 DES 모델링의 감각이다. 

def transport(env, lot_id, vehicles, hop):
    """반송차를 배차받아 다음 스테이션으로 이동한다."""
    queued = env.now
    with vehicles.request() as req:
        yield req
        transport_waits.append(env.now - queued)
        log(env, lot_id, f"{hop} 반송 시작 (배차 대기 {env.now - queued:.1f}분)")
        busy["vehicle"] += TRAVEL_TIME
        yield env.timeout(TRAVEL_TIME)


def lot_flow(env, lot_id, stations, vehicles):
    """로트 하나의 전체 여정: A 가공 → 반송 → B 가공 → 반송 → C 가공."""
    born = env.now
    log(env, lot_id, "투입")
    yield from process_at(env, lot_id, stations, "A")
    yield from transport(env, lot_id, vehicles, "A→B")
    yield from process_at(env, lot_id, stations, "B")
    yield from transport(env, lot_id, vehicles, "B→C")
    yield from process_at(env, lot_id, stations, "C")
    lead_times.append(env.now - born)
    log(env, lot_id, f"완성 (리드타임 {env.now - born:.1f}분)")

lot_flow가 로트 하나의 여정 전체를 위에서 아래로 읽히는 절차문으로 담고 있다는 점을 보라. yield from은 하위 절차(가공, 반송)를 호출해 그 안의 yield들을 그대로 이어받는 문법이다. 실행하면 앞쪽 로트 두 개의 이벤트 로그가 먼저 찍힌다. 

[t=    2.3] lot000 투입
[t=    2.3] lot000 A 가공 시작 (대기 0.0분)
[t=    3.3] lot001 투입
[t=    6.6] lot000 A 가공 완료
[t=    6.6] lot000 A→B 반송 시작 (배차 대기 0.0분)
[t=    6.6] lot001 A 가공 시작 (대기 3.2분)
...

이 로그의 한 줄 한 줄이 Station 01에서 배운 "사건"이다. 클록이 2.3에서 3.3으로, 다시 6.6으로 점프하는 것, 그리고 t=6.6에 사건 세 개가 연달아 처리되는 것(완료 → 반송 시작 → 다음 로트 가공 시작)까지 — 사건 루프의 동작이 그대로 보인다. 2,000분을 다 굴린 최종 통계는 이렇다. 

완성 로트 수        : 335
처리량              : 10.1 로트/시간
평균 리드타임       : 37.8 분
평균 반송 배차 대기 : 0.16 분
반송차 가동률       : 0.336

스테이션    가동률    평균 큐 대기 [분]
    A       0.638         4.9
    B       0.797        14.2
    C       0.502         2.7

숫자를 읽어 보자. 병목은 B다 — 가동률이 0.797로 가장 높고, 평균 큐 대기가 14.2분으로 A(4.9분)의 3배, C(2.7분)의 5배가 넘는다. 평균 가공 시간이 가장 긴(5.0분) 스테이션 앞에 줄이 쌓이는 것이다. 반면 반송차는 여유롭다 — 가동률 0.336에 배차 대기는 평균 0.16분, 사실상 부르면 바로 온다. 이 시스템에서 리드타임 37.8분의 대부분은 이동이 아니라 B 앞에서 기다리는 시간이 만든다. Phase 1에서 말로만 배운 병목과 가동률이 숫자로 보이는 순간이다.

DIGITAL TWIN 관점

같은 시나리오를 다음 스테이션에서 C#으로 다시 만든다(DesCore). 같은 모델을 서로 다른 두 구현으로 굴려 통계가 일치하는지 비교하는 것 — 이것이 직접 만든 엔진을 검증하는 가장 실전적인 방법이다. 여기서 얻은 처리량·리드타임·가동률 숫자들이 그때의 정답지가 된다.

직접 실험해 보기

파라미터 몇 줄을 바꾸면 시스템의 성격이 바뀐다. 각 항목마다 먼저 결과를 예측하고, 돌려서 확인하라 — 이 "가설 → 실험" 루프가 시뮬레이션을 도구로 만드는 습관이다.

  • 반송차를 1대로 — VEHICLE_COUNT = 1. 배차 대기 0.16분과 반송차 가동률 0.336은 어떻게 변하는가? 반송이 새로운 병목이 되는가, 아니면 여전히 B가 지배하는가?
  • B를 빠르게 — SERVICE_MEAN에서 B를 5.0 → 4.0으로. 병목은 어디로 이동하는가? 리드타임은 얼마나 줄어드는가? 가장 긴 큐가 어느 스테이션 앞으로 옮겨 가는지 확인하라.
  • 투입을 당기면 — ARRIVAL_MEAN을 6.0 → 5.5분으로. B의 가동률 ρ가 5.0/5.5 ≈ 0.91이 된다. Station 02의 ρ–Wq 곡선에서 0.91이 어느 지점이었는지 떠올리고, 리드타임이 어떻게 되는지 보라.
  • 예제 03 — 03_utilization_sweep.py는 λ를 0.5에서 0.95까지 올려 가며 M/M/1의 평균 대기를 측정하는 스크립트다. Station 02에서 이미 본 그 ρ–Wq 폭발 곡선이 바로 이 스크립트가 만든 숫자다.

핵심 확인 질문

SimPy 프로세스 안에서 yield는 무엇을 의미하는가?

"시간이 흐르는 지점"이다. 프로세스가 yield로 제어를 엔진에 넘기면, 엔진은 FEL에서 다음 사건을 꺼내 클록을 점프시키고 때가 된 프로세스를 재개한다. yield env.timeout(t)는 t 후에, yield req는 자원이 빌 때 깨어난다. yield가 없는 코드는 시뮬레이션 시간상 순간(0분)에 일어난다.

큐 대기는 코드의 어느 지점에서, 어떻게 생기는가?

Resource.request() 후의 yield req에서 생긴다. 자원의 용량이 차 있으면 요청은 자원의 큐에 줄을 서고 프로세스는 그 줄이 풀릴 때까지 멈춘다. 대기시간은 요청 직전의 env.now를 기억해 두었다가 깨어난 직후의 env.now와의 차로 측정한다 — 예제들의 env.now - queued 패턴이 그것이다.

예제 ②에서 B가 병목이라는 근거는 무엇인가?

두 지표가 같은 곳을 가리킨다. 가동률이 0.797로 세 스테이션 중 가장 높고, 평균 큐 대기가 14.2분으로 A(4.9분)·C(2.7분)의 3~5배다. 원인은 평균 가공 시간이 5.0분으로 가장 길어 상대적 여유(1 − ρ)가 가장 작기 때문이다. 리드타임 37.8분의 가장 큰 몫도 B 앞의 대기가 만든다.

"이 시뮬레이터를 믿을 수 있는가"에는 어떻게 답하는가?

검증(validation)으로 답한다. 방법은 두 갈래 — ① 이론 해가 존재하는 단순 케이스(M/M/1)를 돌려 공식과 맞춰 본다(예제 ①: ρ 0.802 vs 0.800, Wq 4.39 vs 4.00). ② 같은 모델을 독립된 두 구현으로 굴려 통계를 교차 비교한다 — 다음 스테이션에서 C# 엔진으로 예제 ②를 재현하는 것이 바로 이 방법이다.

예제들은 왜 SIM_TIME을 20,000분, 200,000분씩 길게 잡는가?

시뮬레이션이 텅 빈 시스템에서 시작하기 때문이다. 초반 워밍업 구간의 비정상적으로 짧은 대기가 통계에 섞이므로, 정상 상태 표본을 충분히 쌓아 이를 희석해야 한다. 또 ρ가 1에 가까울수록 평균이 수렴하는 데 오래 걸린다 — 예제 03이 200,000분씩 굴리는 이유다.
