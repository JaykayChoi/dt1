PHASE 5 — 미니 팹 시뮬레이터

Phase 5 · Station 01

아키텍처 — DES 코어 ↔ Unity 브리지

네 개의 페이즈가 여기서 하나로 합쳐진다. 이 스테이션에서는 코드를 새로 쓰기 전에 완성본이 어떻게 조립되어 있는지를 본다 — 순수 C# 시뮬레이션과 Unity 화면이 어디서 갈라지고, 시간과 공간이라는 두 개의 다리로 어떻게 다시 만나는지.

종점에 도착했다 — 무엇을 만드는가

지금까지의 여정은 전부 이 페이즈를 위한 부품 조달이었다. Phase 1에서 도메인을 배웠다 — 반송 명령, MCS, 배차 규칙, 포트, 그리고 레일 위를 달리는 OHT. Phase 2에서 그 세계를 계산하는 엔진을 만들었다 — 우선순위 큐 기반 이벤트 스케줄러 DesCore. Phase 3에서 무대를 세웠다 — PBR 머티리얼과 구운 조명의 팹 씬. Phase 4에서 규율을 얻었다 — 매니저 통합, GC Alloc 0, 측정하고 바꾸고 다시 측정하는 습관. Phase 5는 새로운 것을 배우는 페이즈가 아니라 이 부품들을 하나의 실행물로 조립하는 페이즈다.

조립의 결과가 미니 팹 시뮬레이터다. 완성본 C:\work\dt1\FabSim-P5-Complete에서는 일방통행 루프 레일 위에 포트 6개가 놓여 있고, OHT 4대가 무작위로 발생하는 반송 명령을 받아 픽업하고, 실어 나르고, 내려놓는다. 화면 구석의 HUD는 처리량 · 평균 반송 시간 · 함대 가동률을 실시간으로 갱신한다 — Phase 1에서 글로 배운 MCS의 세계가 그대로 화면 위에서 돌아가는 것이다.

핵심 설계 — 시뮬레이션과 뷰의 분리

이 프로젝트의 코드는 두 세계로 나뉜다. 폴더 경계가 곧 설계 경계다.

  • Sim 계층 — Assets/DesCore/와 Assets/Scripts/Sim/. 순수 C#이며 UnityEngine을 단 한 줄도 참조하지 않는다. 시뮬레이션 클록, 레일 그래프, 배차 로직, 통계가 전부 여기 산다.
  • View 계층 — Assets/Scripts/ 바로 아래의 MonoBehaviour들. 러너(FabSimulation), 시각화 (VehicleFleetView), HUD(StatsHud). Sim 계층의 상태를 읽어서 그리기만 한다.

왜 이렇게 자르는가. 분리가 주는 보상이 세 가지 있고, 셋 다 공짜가 아니라 이 경계를 지킨 대가로만 주어진다.

  • 헤드리스 테스트 — Sim 계층만 떼어 Unity 없이 돌릴 수 있다. 아래는 완성본의 스모크 테스트 출력이다: 시뮬레이션 1시간(3,600초), 비히클 4대, 평균 명령 간격 25초 조건에서 렌더링 한 프레임 없이 결과를 뽑았다.

브리지 ① 시간 — 클록을 프레임이 민다

Phase 2 Station 04의 마지막 절에서 SimulationRunner라는 스케치를 보여주며 "이게 전부다"라고 예고했었다. 그 스케치가 완성본에서 실물이 됐다. Assets/Scripts/FabSimulation.cs의 Update가 시간의 다리 전체다. 

private void Update()
{
    if (Input.GetKeyDown(KeyCode.Alpha0))
    {
        timeScale = 0f;
    }
    else if (Input.GetKeyDown(KeyCode.Alpha1))
    {
        timeScale = 1f;
    }
    else if (Input.GetKeyDown(KeyCode.Alpha2))
    {
        timeScale = 30f;
    }
    else if (Input.GetKeyDown(KeyCode.Alpha3))
    {
        timeScale = 120f;
    }

    // 시뮬레이션 시간과 벽시계 시간의 분리 — 엔진은 배속을 모른다.
    Model.Simulation.Run(Model.Simulation.Now + Time.deltaTime * timeScale);
}

마지막 한 줄이 핵심이다. 매 프레임, "현재 시뮬레이션 시각 + 이번 프레임의 실시간 × 배속"까지 클록을 밀어 달라고 엔진에 요청한다. 엔진의 Run(until)은 그 시각 이전의 사건을 전부 실행하고 클록을 정확히 until에 맞춘다 — Phase 2에서 이 시그니처로 설계한 이유가 바로 이 한 줄을 위해서였다.

  • 일시정지(배속 0) — Run(Now + 0)은 실행할 사건이 없으므로 아무 일도 하지 않는다. 정지 로직을 따로 만들 필요가 없다.
  • 120배속 — 60fps 기준 한 프레임에 시뮬레이션 2초어치를 민다. DES는 사건과 사건 사이를 점프하므로 비용은 "흘려보낸 시간"이 아니라 그 구간의 사건 수에 비례한다 — 1시간이 통째로 사건 1,893개였으니, 한 프레임에 실행되는 사건은 평균 1개 남짓이다.
  • 엔진 무수정 — 배속이 0이든 120이든 Simulation 코드는 한 줄도 바뀌지 않는다. 엔진은 "어디까지"만 들을 뿐, 그 값을 누가 어떤 배속으로 계산했는지 모른다.

브리지 ② 공간 — 사건 사이를 보간한다

시간의 다리만으로는 화면이 이상해진다. DES의 세계는 이산이다 — 모델은 "t=100에 노드 A를 출발했고, t=104에 노드 B에 도착한다"만 안다. 그 사이의 t=101.7에 비히클이 어디 있는지는 계산한 적도, 저장한 적도 없다. 반면 화면은 연속이다 — 매 프레임 비히클의 위치가 필요하다. 사건 시각만 알고 그대로 그리면 비히클은 노드에서 노드로 순간이동하는 것처럼 보일 것이다.

답은 보간이다. VehicleAgent가 현재 주행 중인 엣지의 출발/도착 노드와 시각(EdgeFromNode, EdgeToNode, EdgeDepartAt, EdgeArriveAt)을 노출하고, 뷰는 렌더 시점의 진행률 t = (now − depart) / (arrive − depart)를 계산해 두 노드 좌표를 선형 보간한다. Assets/Scripts/VehicleFleetView.cs의 공간 다리가 이것이다. 

private static Vector3 EvaluatePosition(VehicleAgent agent, RailGraph graph, double now)
{
    if (!agent.IsMoving)
    {
        return ToVector(graph.GetNode(agent.NodeId));
    }

    double span = agent.EdgeArriveAt - agent.EdgeDepartAt;
    float t = span <= 0.0
        ? 1f
        : Mathf.Clamp01((float)((now - agent.EdgeDepartAt) / span));
    Vector3 from = ToVector(graph.GetNode(agent.EdgeFromNode));
    Vector3 to = ToVector(graph.GetNode(agent.EdgeToNode));
    return Vector3.Lerp(from, to, t);
}

읽는 순서대로: 이동 중이 아니면 현재 노드 좌표를 그대로 쓴다. 이동 중이면 엣지 소요 시간 span 대비 경과 시간의 비율 t를 구하고 (Clamp01로 0~1 밖을 방어), 출발·도착 노드 좌표를 Lerp한다. 시뮬레이션은 위치를 매 프레임 갱신하지 않는다 — 위치는 뷰가 렌더 시점에 즉석에서 계산하는 파생값이다.

Phase 4의 규율도 이 파일에 그대로 살아 있다. 비히클마다 스크립트를 붙이는 대신 매니저 하나의 LateUpdate가 함대 전체를 순회하며 보간한다(매니저 통합 패턴). 그리고 FOUP 표시는 상태가 실제로 바뀐 프레임에만 갱신한다. 

// SetActive는 상태가 바뀔 때만 호출한다 (매 프레임 호출은 낭비).
bool isCarrying = agent.Phase == VehiclePhase.Carrying;
if (isCarrying != foupVisible[i])
{
    foups[i].SetActive(isCarrying);
    foupVisible[i] = isCarrying;
}
뷰는 읽기 전용이다

뷰에서 Sim 계층의 상태를 절대 변경하지 말 것. 흐름은 항상 한 방향 — 뷰는 클록을 밀고(Run), 상태를 읽는다. 그게 전부다. 뷰가 VehicleAgent의 좌표나 국면을 건드리는 순간, 같은 시드가 같은 결과를 내는 결정론이 깨지고, 헤드리스 테스트가 검증한 세계와 화면에 보이는 세계가 서로 다른 세계가 된다. 그때부터는 스모크 테스트가 통과해도 화면의 버그를 잡을 수 없다.

한 장의 도면 — 두 계층, 두 개의 다리

DIGITAL TWIN 관점

이 구조는 실물 관제 시스템과 동형이다. 지금 FabModel은 난수로 반송 명령을 만들어내지만, 그 자리를 실제 MCS의 이벤트 스트림 구독으로 바꾸면 — 명령 생성·배차·이동 사건이 난수 대신 현장에서 흘러들어오면 — 나머지 코드는 그대로인 채 이 시뮬레이터는 관제 트윈이 된다. Phase 1 Station 03에서 "시뮬레이션과 관제 트윈은 같은 뼈대의 양면"이라고 했던 말이, 여기서 코드 구조로 증명된 것이다.

완성본 열어 보기

도면을 봤으니 실물을 돌려 보자. 아래 버튼으로 완성본 프로젝트를 연다.

버튼은 어떤 방식으로 사이트를 열었든 동작한다 — 파일로 직접 열었다면 브라우저가 처음 한 번 "열기 허용" 확인창을 띄우니 허용하면 된다.

Play를 누르면 이 페이지에서 설명한 전부가 움직이기 시작한다.

  • OHT 4대가 루프 레일을 돌며 포트로 향한다. 픽업하는 순간 비히클 아래에 FOUP이 나타나고, 드롭하면 사라진다 — VehiclePhase.Carrying 상태가 그대로 보이는 것이다.
  • HUD가 처리량·평균 반송 시간·가동률을 갱신한다.
  • 키 0 / 1 / 2 / 3으로 배속을 0(일시정지) / 1 / 30 / 120배로 전환한다. 위에서 본 Update의 키 분기가 하는 일이다.

과제 하나. 배속 120으로 시뮬레이션 시각 1시간 지점까지 돌려 보고, HUD의 세 지표를 스모크 테스트 수치 — 시간당 160건, 평균 34.4초, 가동률 38% — 와 비교해 보라. 시드가 같으므로(기본 7) 거의 일치해야 정상이다. 헤드리스로 검증한 세계와 화면에 그려지는 세계가 같은 세계임을 눈으로 확인하는 것 — 그것이 이 아키텍처가 약속하는 전부다.

핵심 확인 질문

Sim 계층과 View 계층을 분리하면 얻는 보상 세 가지는?

① 헤드리스 테스트 — Sim만 떼어 렌더링 없이 시뮬레이션 1시간을 초 단위 만에 검증할 수 있다. ② 결정론적 재현 — 같은 시드면 배속·프레임레이트와 무관하게 같은 결과가 나와 버그 재현과 회귀 검증이 가능하다. ③ 재사용 — 순수 C#이므로 Phase 2의 DesCore를 폴더째, 무수정으로 가져올 수 있었다.

Run(Now + Time.deltaTime × 배속) 한 줄이 일시정지와 120배속을 엔진 수정 없이 지원하는 이유는?

엔진의 Run(until)은 "어디까지 진행할지"라는 절대 시각만 받고, 그 값이 어떻게 계산됐는지 모르기 때문이다. 배속 0이면 until = Now가 되어 실행할 사건이 없고(일시정지), 배속 120이면 한 프레임에 큰 구간을 밀 뿐이다. 배속이라는 개념은 전부 엔진 바깥(뷰의 timeScale)에 산다.

이동 중인 비히클의 화면 위치는 어떤 공식으로 구하는가?

진행률 t = (now − EdgeDepartAt) / (EdgeArriveAt − EdgeDepartAt)을 0~1로 클램프한 뒤, 출발 노드와 도착 노드 좌표를 Vector3.Lerp(from, to, t)로 선형 보간한다. 시뮬레이션은 엣지의 출발/도착 "사건 시각"만 기록하고, 그 사이의 위치는 뷰가 렌더 시점에 계산하는 파생값이다.

뷰가 Sim 상태를 변경하면 안 되는 이유는?

결정론이 깨지기 때문이다. Sim의 상태 전이가 사건 실행으로만 일어나야 "같은 시드 = 같은 결과"가 성립하는데, 뷰가 프레임 타이밍에 따라 상태를 건드리면 실행마다 결과가 달라지고, 헤드리스 테스트가 검증한 모델과 화면의 모델이 서로 다른 세계가 된다. 흐름은 항상 한 방향 — 뷰는 클록을 밀고 상태를 읽기만 한다.

배속 120으로 돌려도 프레임 비용이 폭발하지 않는 이유는?

DES의 비용은 흘려보낸 시뮬레이션 시간이 아니라 그 구간의 사건 수에 비례하기 때문이다. 이 모델은 1시간 전체가 사건 1,893개다 — 60fps에서 배속 120이면 한 프레임에 시뮬레이션 2초어치, 즉 사건 평균 1개 남짓만 실행된다. 사건이 없는 구간은 계산 자체를 하지 않는다.
