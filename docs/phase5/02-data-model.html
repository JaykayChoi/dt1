PHASE 5 — 미니 팹 시뮬레이터

Phase 5 · Station 02

데이터 모델 — 레일 그래프 · MCS 명령 흐름

미니 팹의 뼈대는 세 파일이다. 레일을 정의하는 FabLayout, 그 위에서 길을 찾는 RailGraph, 그리고 반송 명령을 만들어 배차하고 완주시키는 미니 MCS인 FabModel. Phase 1에서 말로 배운 레이아웃·명령 생애주기·배차 규칙이 이 페이지에서 전부 자료구조와 코드가 된다.

레이아웃은 곧 그래프다

Phase 1 Station 02는 "팹 레이아웃 도면은 그 자체로 그래프 정의서"라고 예고했다. 그 예고가 코드가 되는 곳이 여기다. 노드(node)는 비히클이 정지하거나 방향을 바꾸는 웨이포인트(waypoint) — 포트, 코너, 분기 지점이고, 엣지(edge)는 노드와 노드를 잇는 방향 있는 레일 구간이다. OHT 레일은 일방통행이 기본이므로 방향성은 장식이 아니라 본질이다.

이것을 담는 그릇이 RailGraph다. Phase 2의 DesCore와 같은 이유로 UnityEngine 무의존 순수 C#이라, Unity 없이 콘솔에서 돌려 보며 검증할 수 있다. 인터페이스는 세 개가 전부다.

  • AddNode(x, y, z) — 3D 좌표를 등록하고 노드 id(등록 순번)를 돌려준다.
  • AddEdge(from, to) — from → to 방향성 엣지를 추가한다. 가중치를 따로 저장하지 않는다 — 길이는 두 노드의 좌표에서 계산한다(GetDistance). 노드 좌표가 곧 유일한 진실이다.
  • FindPath(start, goal) — 다익스트라 최단 경로. 노드 시퀀스를 반환하고, 도달 불가능하면 null.

FindPath의 심장부를 보자. 

bestCost[start] = 0f;

// 노드 수가 작아(수십 개) 선형 탐색 다익스트라로 충분하다.
while (true)
{
    int current = -1;
    float currentCost = float.PositiveInfinity;
    for (int i = 0; i < nodes.Count; i++)
    {
        if (!visited[i] && bestCost[i] < currentCost)
        {
            current = i;
            currentCost = bestCost[i];
        }
    }

    if (current == -1)
    {
        return null;
    }

    if (current == goal)
    {
        break;
    }

    visited[current] = true;
    foreach (int next in outgoing[current])
    {
        float cost = currentCost + GetDistance(current, next);
        if (cost < bestCost[next])
        {
            bestCost[next] = cost;
            previous[next] = current;
        }
    }
}

교과서의 다익스트라는 "미방문 노드 중 비용 최소"를 우선순위 큐로 뽑지만, 여기서는 배열을 처음부터 끝까지 훑는 선형 탐색이다. Phase 2에서는 사건이 수천 개 쌓이는 FEL을 위해 이진 최소 힙을 직접 만들었는데, 왜 여기서는 안 만들었을까? 노드가 12개이기 때문이다. O(V²)라도 열두 개의 제곱은 아무것도 아니다. 자료구조는 교과서가 아니라 규모가 고른다 — 같은 프로젝트 안에서 힙을 만든 손과 선형 탐색을 고른 손이 같은 손이라는 것이 이 선택의 교훈이다.

도달 불가능하면 null을 돌려주는 것도 의미가 있다. 모든 엣지가 한 방향이므로 엣지 하나만 빠뜨려도 그래프가 끊긴다 — null은 레이아웃 정의 버그를 가장 시끄럽게 드러내는 신호다.

FabLayout — 한 곳에서 정의하는 진실

미니 팹의 레이아웃 전체가 파일 하나다. 상수 세 개와 메서드 하나 — 전문을 그대로 싣는다. 

namespace FabSim.Sim
{
    /// <summary>
    /// 미니 팹의 레일 레이아웃 정의 — 일방통행 직사각 루프 + 포트 노드.
    /// 씬 빌더(시각물)와 런타임(그래프)이 같은 정의를 공유해 어긋남을 막는다.
    /// 일방통행 루프는 Phase 1에서 배운 데드락 구조적 회피이기도 하다.
    /// </summary>
    public static class FabLayout
    {
        /// <summary>레일 높이 [m].</summary>
        public const float RailHeight = 4f;

        /// <summary>루프의 X 반폭 [m].</summary>
        public const float HalfWidth = 6f;

        /// <summary>루프의 Z 반길이 [m].</summary>
        public const float HalfLength = 16f;

        /// <summary>
        /// 일방통행 루프 그래프를 만든다. 반환된 portNodes는 픽업/드롭이
        /// 일어나는 노드 id들(긴 변의 비코너 노드 6개)이다.
        /// </summary>
        public static RailGraph Build(out int[] portNodes)
        {
            var graph = new RailGraph();
            float y = RailHeight;

            // 반시계 방향 일방통행 루프 — 12개 노드.
            int n0 = graph.AddNode(-HalfWidth, y, -HalfLength);
            int n1 = graph.AddNode(-HalfWidth, y, -8f);
            int n2 = graph.AddNode(-HalfWidth, y, 0f);
            int n3 = graph.AddNode(-HalfWidth, y, 8f);
            int n4 = graph.AddNode(-HalfWidth, y, HalfLength);
            int n5 = graph.AddNode(0f, y, HalfLength);
            int n6 = graph.AddNode(HalfWidth, y, HalfLength);
            int n7 = graph.AddNode(HalfWidth, y, 8f);
            int n8 = graph.AddNode(HalfWidth, y, 0f);
            int n9 = graph.AddNode(HalfWidth, y, -8f);
            int n10 = graph.AddNode(HalfWidth, y, -HalfLength);
            int n11 = graph.AddNode(0f, y, -HalfLength);

            int[] loop = { n0, n1, n2, n3, n4, n5, n6, n7, n8, n9, n10, n11 };
            for (int i = 0; i < loop.Length; i++)
            {
                graph.AddEdge(loop[i], loop[(i + 1) % loop.Length]);
            }

            // 긴 변의 중간 노드들이 장비 로드포트 위 정지 지점이다.
            portNodes = new[] { n1, n2, n3, n7, n8, n9 };
            return graph;
        }
    }
}

이 짧은 파일에 설계 결정이 세 개 들어 있다.

① 씬과 시뮬레이션이 같은 정의를 읽는다

레일 메시와 장비를 배치하는 씬 빌더도, 경로를 계산하는 런타임도 같은 Build()를 호출한다. 레이아웃 정의가 두 벌이면 언젠가 반드시 어긋난다 — 시각 레일과 시뮬레이션 그래프가 조금이라도 다르면 비히클이 레일 밖 허공을 달리는 화면이 나온다. 정의를 한 곳에 모아 두면 이 부류의 버그가 원천적으로 생길 수 없다. "고치려면 한 군데만 고친다"는 보너스는 덤이다.

② 일방통행 루프 — 레이아웃으로 사는 데드락 회피

(i + 1) % loop.Length — 엣지 12개가 전부 같은 방향이고, 역방향 엣지는 하나도 없다. Phase 1 Station 04에서 데드락의 성립 조건은 순환 대기, 즉 서로가 서로의 길을 막고 마주 보며 기다리는 고리라고 배웠다. 모든 비히클이 같은 방향으로 도는 루프에서는 마주 보는 대기 자체가 성립할 수 없다 — 앞차가 언젠가 움직이면 나도 움직인다. 데드락 방지를 감지·회피 알고리즘으로 사는 대신 레이아웃 선택으로 산 것이다. 물론 공짜는 아니다. 목적지가 바로 뒤 노드여도 루프를 한 바퀴 돌아야 한다 — 경로 자유도를 안전과 맞바꾼 거래이고, 실제 팹의 OHT 레일이 일방통행을 기본으로 까는 이유이기도 하다.

③ 포트 노드 — 긴 변의 비코너 6개

픽업·드롭이 일어나는 포트 노드는 긴 변(±X 두 변)의 중간 노드들 — n1, n2, n3과 n7, n8, n9 — 이다. 코너는 방향을 꺾는 구간이라 정지 지점으로 쓰지 않고, 긴 변의 중간 노드 바로 아래에 장비 로드포트가 놓인다. Phase 1에서 본 "OHT가 장비 위 레일에 정지해 호이스트를 내리는" 바로 그 지점이다.

DIGITAL TWIN 관점

Build()의 노드 좌표를 하드코딩 대신 실팹 CAD 도면·설비 DB에서 뽑은 좌표로 바꾸면, 이 페이지의 나머지 코드는 한 줄도 고치지 않고 실물 레이아웃 위에서 돈다. 그래프·배차·통계는 좌표가 어디서 왔는지 모르기 때문이다. 디지털 트윈에서 레이아웃 데이터의 원천은 도면과 설비 마스터이고, FabLayout 자리에 그 로더가 들어간다.

반송 명령의 생애 — 코드로 다시 보는 Station 03

Phase 1 Station 03에서 반송 명령의 생애주기를 배웠다 — 생성되고, 배차되고, 픽업·이동·드롭을 거쳐 완료된다. FabModel은 그 생애주기를 그대로 코드로 옮긴 미니 MCS다. 명령 하나는 TransportJob 객체다: 어디서 어디로(FromPort, ToPort)에 더해 타임스탬프 세 개 — CreatedAt(생성), AssignedAt(배차), CompletedAt(완료) — 를 지니고 다닌다. 이 세 시각이 뒤에서 볼 통계의 원료 전부다. 비히클들은 시작 시 루프 위에 고르게 흩어 놓는다.

생성 — 지수분포 간격으로 도착하는 명령

private void ScheduleNextJobArrival()
{
    Simulation.Schedule(random.NextExponential(jobIntervalMean), () =>
    {
        OnJobArrived();
        ScheduleNextJobArrival();
    });
}

private void OnJobArrived()
{
    int from = portNodes[random.Next(portNodes.Length)];
    int to = from;
    while (to == from)
    {
        to = portNodes[random.Next(portNodes.Length)];
    }

    pendingJobs.Enqueue(new TransportJob
    {
        Id = nextJobId++,
        FromPort = from,
        ToPort = to,
        CreatedAt = Simulation.Now,
    });
    TryDispatch();
}

Phase 2의 도착 프로세스가 그대로다. 지수분포 간격(NextExponential — 포아송 도착)으로 다음 도착을 예약하고, 도착 사건이 다음 도착 사건을 스스로 예약한다. 출발·도착 포트는 랜덤으로 뽑되 같으면 다시 뽑고, 큐에 넣자마자 TryDispatch()를 부른다 — 상위 시스템(MES)이 내려보낸 반송 요청이 MCS의 대기 큐에 꽂히는 순간이다.

배차 — 선입선출 × 최근접 유휴

// 배차 규칙: 대기 명령 선입선출 × 최근접 유휴 비히클 (Phase 1 Station 04).
private void TryDispatch()
{
    while (pendingJobs.Count > 0)
    {
        TransportJob job = pendingJobs.Peek();
        VehicleAgent best = null;
        float bestDistance = float.PositiveInfinity;
        foreach (VehicleAgent vehicle in vehicles)
        {
            if (vehicle.Phase != VehiclePhase.Idle)
            {
                continue;
            }

            List<int> path = graph.FindPath(vehicle.NodeId, job.FromPort);
            if (path == null)
            {
                continue;
            }

            float distance = graph.GetPathLength(path);
            if (distance < bestDistance)
            {
                bestDistance = distance;
                best = vehicle;
            }
        }

        if (best == null)
        {
            return;
        }

        pendingJobs.Dequeue();
        Assign(best, job);
    }
}

Phase 1 Station 04에서 본 가장 단순한 배차 규칙 — 선입선출 × 최근접 유휴 — 이 30줄로 끝난다. 뜯어볼 지점이 셋 있다.

  • Peek() — 큐의 머리만 본다. 머리 명령이 배차될 때까지 뒤 명령은 절대 먼저 나가지 못한다. 유휴 비히클이 없으면 best == null로 통째로 return하고, 명령은 큐에 남아 다음 기회 — 새 명령 도착 또는 작업 완료 — 를 기다린다. TryDispatch가 불리는 곳이 정확히 그 두 곳 (OnJobArrived 끝, CompleteJob 끝)이다. 상태가 바뀌는 순간에만 다시 계산한다 — DES의 문법 그대로다.
  • 최근접의 "거리"는 그래프 거리다. FindPath로 경로를 찾고 GetPathLength로 잰다. 일방통행 루프에서는 픽업 포트 바로 뒤 노드에 있는 비히클이 사실은 가장 먼 비히클이다 — 한 바퀴를 다 돌아야 하니까. 직선거리로 쟀다면 최악의 비히클을 뽑았을 것이다.
  • while — 한 번에 여러 건. 유휴 비히클이 여럿이면 대기 명령을 연달아 배차한다. 작업 완료 한 번이 밀린 큐를 한꺼번에 풀 수 있다.

배차 이후 — 픽업·dwell·드롭의 콜백 연쇄

private void Assign(VehicleAgent vehicle, TransportJob job)
{
    vehicle.Phase = VehiclePhase.ToPickup;
    vehicle.CurrentJob = job;
    vehicle.BusyStartedAt = Simulation.Now;
    job.AssignedAt = Simulation.Now;

    List<int> toPickup = graph.FindPath(vehicle.NodeId, job.FromPort);
    TraversePath(vehicle, toPickup, 1, () =>
    {
        // 픽업 핸드오프 (Phase 1의 E84 자리) — 고정 dwell로 근사.
        Simulation.Schedule(PickupDwell, () =>
        {
            vehicle.Phase = VehiclePhase.Carrying;
            List<int> toDrop = graph.FindPath(job.FromPort, job.ToPort);
            TraversePath(vehicle, toDrop, 1, () =>
            {
                Simulation.Schedule(DropDwell, () => CompleteJob(vehicle));
            });
        });
    });
}

람다 세 겹 — Phase 2 Station 04가 "콜백 연쇄는 단계가 늘면 급격히 읽기 어려워진다"고 경고했던 바로 그 구조다. 픽업 경로를 완주하면 4초 dwell(실팹이라면 SEMI E84 핸드오프 절차가 도는 자리 — 코드 주석이 그대로 말하고 있다), 상태가 Carrying으로 바뀌고, 드롭 경로 완주 후 다시 dwell, 그리고 CompleteJob. Station 03에서 그림으로 배운 생애주기의 각 화살표가 콜백 하나씩에 대응한다.

이동은 사건의 연쇄다

Assign이 두 번 부르는 TraversePath가 이동의 전부다. 경로를 통째로 "이동 완료" 사건 하나로 예약할 수도 있었지만, 이 코드는 엣지 하나를 완주할 때마다 사건 하나를 건다. 

// 경로를 엣지 단위 사건의 연쇄로 굴린다 — "사건이 사건을 낳는" Phase 2 패턴.
private void TraversePath(VehicleAgent vehicle, List<int> path, int nextIndex,
    Action onArrived)
{
    if (nextIndex >= path.Count)
    {
        vehicle.IsMoving = false;
        vehicle.NodeId = path[path.Count - 1];
        onArrived();
        return;
    }

    int from = path[nextIndex - 1];
    int to = path[nextIndex];
    double travel = graph.GetDistance(from, to) / vehicleSpeed;
    vehicle.IsMoving = true;
    vehicle.EdgeFromNode = from;
    vehicle.EdgeToNode = to;
    vehicle.EdgeDepartAt = Simulation.Now;
    vehicle.EdgeArriveAt = Simulation.Now + travel;

    Simulation.Schedule(travel, () =>
    {
        vehicle.NodeId = to;
        TraversePath(vehicle, path, nextIndex + 1, onArrived);
    });
}

구간 이동 시간을 계산해 "구간 완주" 사건을 예약하고, 그 사건이 실행되면 NodeId를 갱신하고 다음 구간에 대해 자기 자신을 다시 부른다 — 도착 사건이 출발 사건을 낳는 재귀다. Phase 2에서 배운 "사건이 사건을 낳는다"가 실전에서 어떻게 쓰이는지를 보여주는 가장 좋은 예제다. 왜 굳이 엣지 단위로 쪼갰을까.

  • 화면 보간은 직선 구간에서만 성립한다. 경로는 코너를 꺾는다 — "출발점에서 도착점까지 직선 보간"은 엣지 안에서만 참이다.
  • NodeId가 구간마다 갱신된다. 배차의 최근접 계산이 "마지막으로 지난 노드" 기준으로 정확해진다.
  • 교통 제어가 들어올 자리가 엣지 경계다. 나중에 구간 점유(한 구간에 한 대)를 넣는다면, 잡고 놓는 코드가 정확히 이 재귀의 머리와 꼬리에 들어간다.

그리고 이동 중 상태를 담는 VehicleAgent의 필드들 — 

/// <summary>정지 중이거나 마지막으로 도착한 노드.</summary>
public int NodeId;

/// <summary>이동 중 여부. true면 아래 엣지 필드들이 유효하다.</summary>
public bool IsMoving;

public int EdgeFromNode;
public int EdgeToNode;
public double EdgeDepartAt;
public double EdgeArriveAt;

이 네 필드 (어느 엣지를, 언제 출발해, 언제 도착하는가) 가 모델과 뷰 사이의 데이터 계약이다. 모델은 비히클의 위치를 매 프레임 갱신하지 않는다 — 사건 두 개 사이의 시각만 남긴다. Unity 쪽 뷰는 렌더 시점 t에 대해 (t − EdgeDepartAt) / (EdgeArriveAt − EdgeDepartAt) 비율로 두 노드 좌표를 보간해 그린다. 모델은 매끄러운 움직임을 모른 채 통계적으로 정확하고, 뷰는 시뮬레이션 로직을 모른 채 매끄럽다 — Station 01에서 세운 브리지 원칙이 이 필드 네 개로 구현된다.

이 모델이 생략한 것

교통 제어(구간 점유)와 충돌이다. 루프가 한 방향이라 정면충돌도 추월도 구조적으로 없지만, 같은 구간에 두 대가 동시에 있을 수는 있다 — 포트에서 dwell 중인 비히클을 뒤차가 유령처럼 통과한다. 시각적으로는 겹침으로 보이고, 통계에는 아무 영향이 없다(감속도 대기도 없으므로). 실팹 OHT는 선행 차량 추종·구간 점유 제어가 필수이고, 이를 넣는 것 — 엣지를 용량 1짜리 리소스로 만드는 것 — 이 Station 03에서 제안하는 확장 과제다.

한 장으로 보는 데이터 모델

위쪽은 FabLayout.Build()가 만드는 그래프를 실제 좌표 비례로, 아래쪽은 반송 명령 하나가 지나가는 상태 흐름을 담당 코드와 함께 그린 것이다.

통계는 어디서 나오나

시뮬레이터 HUD에 뜨는 숫자 네 개는 전부 이 페이지에서 본 타임스탬프와 카운터에서 나온다. Phase 1 Station 04의 지표 표와 1:1로 대응된다.

지표 Station 04 FabModel의 계산 원천 데이터
처리량 GetThroughputPerHour — 완료 수 ÷ 경과 시간 CompletedJobs 카운터
평균 반송 시간 GetAverageDeliveryTime — Σ(완료 − 생성) ÷ 완료 수 TransportJob의 CreatedAt·CompletedAt
가동률 GetFleetUtilization — busy 시간 적분 ÷ (경과 시간 × 대수) VehicleAgent의 BusyStartedAt·TotalBusyTime
대기 명령 수 PendingJobCount pendingJobs 큐 길이

두 가지가 눈에 띈다. 첫째, 평균 반송 시간의 기준은 AssignedAt이 아니라 CreatedAt이다 — 배차를 기다린 시간까지 포함한 "요청한 쪽 관점"의 리드타임이라는 뜻이다. AssignedAt을 쓰면 대기와 이동을 분해할 수 있다 — 타임스탬프를 세 개 다 남기는 이유다. 둘째, 가동률은 Phase 2의 SimResource와 같은 시간 적분 수법이다. 작업이 끝날 때마다 TotalBusyTime에 누적하고, 조회 시점에 진행 중인 작업의 now − BusyStartedAt을 더해 "지금 이 순간까지"의 값을 만든다.

스모크 런의 실측은 이렇다.

완료 반송      : 160 건/시간
평균 반송 시간 : 34.4 초 (명령 생성 → 드롭 완료)
함대 가동률    : 38 %
실행 사건 수   : 1,893

이 숫자들은 Phase 2 Station 02의 리틀의 법칙으로 검산할 수 있다. L = λW — 도착률 λ ≈ 160건/시간 = 160/3600 건/초, 체류 시간 W = 34.4초이므로 시스템 안(생성부터 완료 사이)에 떠 있는 명령 수는 평균 L ≈ 160/3600 × 34.4 ≈ 1.5건이어야 한다. HUD의 대기 명령 수에 이동 중인 명령을 더한 값의 시간 평균이 이 근처면 통계 로직이 맞다는 강한 증거다 — 독립적인 법칙으로 자기 코드를 검산하는 습관은 Phase 2의 교차 검증과 같은 기본기다. 가동률 38%는 함대에 여유가 많다는 뜻이고, 실제로 대기 큐는 거의 항상 비어 있다. 마지막 줄도 지나치지 말자 — 수백 건의 반송이 도착하고, 배차되고, 구간을 완주하고, dwell을 거쳐 완료되는 세계 전체가 사건 1,893개로 끝났다.

핵심 확인 질문

FindPath는 왜 우선순위 큐 없는 선형 탐색 다익스트라인가?

노드가 12개(확장해도 수십 개)라서다. O(V²)라도 이 규모에서는 비용이 무의미하고, 힙을 붙이는 복잡도가 오히려 손해다. 사건이 수천 개 쌓이는 Phase 2의 FEL에는 이진 힙을 직접 만들었다 — 자료구조 선택은 교과서가 아니라 데이터의 규모가 결정한다.

씬 빌더와 런타임이 FabLayout 하나를 공유하는 이유는?

시각물(레일 메시·장비 배치)과 시뮬레이션 그래프가 어긋나는 부류의 버그를 원천 차단하기 위해서다. 정의가 두 벌이면 언젠가 달라지고, 그러면 비히클이 화면의 레일 밖을 달린다. 단일 정의(single source of truth)를 양쪽이 읽으면 이 불일치가 구조적으로 불가능해진다.

일방통행 루프는 어떻게 데드락을 "구조적으로" 막는가?

데드락의 성립 조건인 순환 대기는 서로가 서로의 길을 막고 마주 보며 기다릴 때 생긴다. 모든 엣지가 같은 방향인 루프에서는 마주 보는 대기 자체가 성립할 수 없다 — 앞차가 움직이면 나도 움직인다. 감지·회피 알고리즘 없이 레이아웃 선택만으로 조건 하나를 제거한 것이고, 대가는 우회 거리(목적지가 바로 뒤여도 한 바퀴)다.

TraversePath는 왜 경로 전체가 아니라 엣지 단위로 사건을 예약하는가?

셋 때문이다. ① 뷰의 직선 보간은 엣지 안에서만 성립한다(경로는 코너를 꺾는다). ② 구간마다 NodeId가 갱신돼 배차의 최근접 계산이 정확해진다. ③ 나중에 구간 점유(교통 제어)를 넣을 자리가 정확히 엣지 경계다. 도착 사건이 다음 출발 사건을 예약하는 이 재귀가 Phase 2의 "사건이 사건을 낳는다" 패턴의 실전형이다.

평균 반송 시간은 어느 두 시각의 차이고, 배차 대기를 포함하는가?

CreatedAt(명령 생성)부터 CompletedAt(드롭 완료)까지 — 배차 대기를 포함한다. 요청한 쪽이 체감하는 리드타임이기 때문이다. AssignedAt을 함께 기록해 두었으므로 필요하면 대기 시간(생성→배차)과 수행 시간(배차→완료)으로 분해할 수 있고, 리틀의 법칙(L = λW)으로 평균 재공 명령 수를 검산하는 데도 이 시각들이 쓰인다.
