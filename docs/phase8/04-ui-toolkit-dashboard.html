PHASE 8 — 디지털 트윈 연결

Phase 8 · Station 04

UI Toolkit 운영자 대시보드

Phase 5의 StatsHud는 OnGUI로 그린 임시 HUD였다. 운영자 화면은 그렇게 못 만든다 — 팀 스택은 UI Toolkit이다. 구조·스타일·로직을 나누고, 수천 대 목록을 가상화로 다룬다.

UXML / USS / UI Builder

UI Toolkit은 웹을 빼닮았다. UXML이 구조(HTML), USS가 스타일(CSS), C#이 로직(JS)이다. UI Builder로 시각 편집도 한다. 이 분리가 협업의 핵심 — 디자이너가 USS를, 개발자가 로직을 따로 만질 수 있다.

BEM 네이밍 — 이름만 보고 구조를 읽는다

클래스 이름은 저장소 컨벤션 그대로 BEM: block__element--modifier kebab-case로, 의미 기반이다. 색이 아니라 상태를 이름에 담는다 — vehicle-list__row--blocked이지 --red가 아니다. 나중에 "막힘"의 색을 바꿔도 이름은 그대로다. 

<!-- Dashboard.uxml (구조) -->
<ui:VisualElement class="kpi-panel">
  <ui:VisualElement class="kpi-panel__tile">
    <ui:Label class="kpi-panel__value" />
    <ui:Label class="kpi-panel__label" text="처리량" />
  </ui:VisualElement>
  <ui:VisualElement class="kpi-panel__tile--alert" />  <!-- 임계 초과 시 -->
</ui:VisualElement>
<ui:ListView class="vehicle-list" />

런타임 API와 갱신 주기

UIDocument 컴포넌트의 rootVisualElement에서 Q<T>(className)로 요소를 찾고, clicked·RegisterCallback로 이벤트를 건다. 여기서 StatsHud의 교훈("문자열을 매 프레임 만들지 말라 — GC")이 바인딩 주기로 되돌아온다: KPI는 schedule.Execute(UpdateKpis).Every(250)로 0.25초마다만 갱신하고, ListView는 더 성긴 주기로 RefreshItems. 프레임마다 재바인딩하지 않는다.

ListView 가상화 — 수천 대를 십여 행으로

차량 수천 대 목록을 항목마다 GameObject로 만들면 무너진다. ListView 가상화는 화면에 실제로 보이는 몇 행만 생성(makeItem)하고 재활용 (bindItem)한다. 수천 개가 다 있는 듯 스크롤되지만 실제로 존재하는 것은 십여 개 행뿐 — uGUI로는 불가능한, UI Toolkit의 결정적 이점이다.

축UI ToolkituGUI
대량 목록ListView 가상화(수천 행)항목마다 GameObject → 무거움
스타일 재사용USS 클래스컴포넌트별 수동
월드 스페이스제한적강함(빌보드 라벨)
편집UI Builder + 텍스트씬 계층
이 페이즈 용도운영자 대시보드차량 위 3D 라벨

uGUI와 공존

UI Toolkit이 uGUI를 완전히 대체하지는 않는다. 차량 위에 떠다니는 FOUP ID 월드 스페이스 라벨(빌보드)은 여전히 uGUI/월드 스페이스의 강점 영역이다. 스크린 스페이스 운영자 대시보드는 UI Toolkit, 3D 공간에 붙는 라벨은 uGUI — 둘은 각자의 자리에서 공존한다.

핵심 확인 질문

수천 대 목록에 ListView 가상화가 필수인 이유는?

항목마다 GameObject/VisualElement를 만들면 수천 개의 객체가 메모리·레이아웃 비용을 잡아먹어 프레임이 무너진다. 가상화는 화면에 보이는 십여 개 행만 생성·재활용하므로, 목록이 아무리 길어도 비용이 일정하게 유지된다.

BEM에서 --red 대신 --blocked를 쓰는 이유는?

이름은 의미(상태)를 담아야지 표현(색)을 담으면 안 된다. "막힘"의 색을 빨강에서 주황으로 바꿔도 --blocked는 그대로다. --red였다면 이름과 실제 색이 어긋나 혼란스러워진다.

Every(250) 바인딩이 StatsHud 스로틀과 같은 문제를 푸는 방식은?

둘 다 "매 프레임 갱신은 낭비(특히 문자열 생성 GC)"라는 문제를 갱신 빈도를 낮춰 푼다. StatsHud는 0.25초에 한 번만 문자열을 만들었고, 대시보드는 schedule.Execute(...).Every(250)로 KPI 바인딩을 0.25초 주기로 돌린다. 사람 눈에 충분하고 GC는 줄인다.

월드 스페이스 라벨을 uGUI로 남기는 이유는?

3D 공간의 특정 오브젝트에 붙어 따라다니는 빌보드 라벨은 uGUI/월드 스페이스 캔버스가 강하고 자연스럽다. UI Toolkit은 스크린 스페이스 UI(대시보드)에 강하다. 각자의 강점 영역이 달라 공존시키는 것이 실용적이다.

Q<T>()가 하는 일은?

rootVisualElement 아래에서 이름이나 클래스로 VisualElement를 찾아 반환한다(웹의 querySelector에 해당). 찾은 요소에 값을 바인딩하거나 콜백을 걸어 로직과 UI를 잇는다.
