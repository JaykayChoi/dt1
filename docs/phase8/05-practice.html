PHASE 8 — 디지털 트윈 연결

Phase 8 · Station 05 · 실습

실습 — 내 시뮬레이터를 트윈으로

01~04에서 완성본으로 "두 소스가 한 뷰를 구동하는" 구조를 읽었다. 이번엔 그 구조를 내 FabSim에 직접 붙인다. 다섯 마일스톤을 지나면, 내가 만든 시뮬레이터가 외부 스트림으로도 돌아가는 미니 트윈이 된다.

시작 상태 — 어디서 출발하나

  • 무대: 사용자의 FabSim(미니 팹 시뮬레이터: FabModel· VehicleFleetView·StatsHud가 도는 상태). 폴백: FabSim-P5-Complete. 트윈 브리지는 교통 제어(P7)와 독립이라 P5 상태에서도 성립한다.
  • 콘솔 실습: examples/twin-bridge-practice 스켈레톤(직렬화·페이싱이 TODO), 정답지 examples/twin-bridge.

마일스톤 — 목표 · 할 일 · 완료 기준

  1. M1 — 레코더·리플레이 서버 스켈레톤 채우기(examples/twin-bridge-practice)
    목표: 헤드리스 레코더가 NDJSON을 굽고, 서버가 타임스탬프 간격대로 송출한다.
    할 일: EventRecorder의 각 이벤트 직렬화(필드 추가)와 NdjsonPacer.ComputeDelay((curT−prevT)/speed) TODO를 채운다.
    완료 기준: 내 로그가 정답지 sample.ndjson과 구조 일치(HELLO·주기 SNAPSHOT·단조 n·델타 필드 완비), 서버가 페이싱대로 라인을 흘린다.
  2. M2 — FabSim에 IFleetSource 분리 리팩터링
    목표: 뷰·HUD가 FabSimulation이 아니라 IFleetSource를 읽는다.
    할 일: IFleetSource 선언, SimFleetSource로 FabModel을 감싸 구현, 두 뷰의 Initialize가 소스를 받도록 수정.
    완료 기준: 리팩터 후에도 내장 Sim으로 이전과 똑같이 동작(회귀 없음). 이것이 "뷰가 소스를 모른다"의 첫 증거다.
    막히면: FabSim-P8-Complete/Assets/Scripts/Twin/IFleetSource.cs·SimFleetSource.cs.
  3. M3 — TwinFeedClient 수신·버퍼·보간 재생
    목표: 리플레이 서버에 붙어, 받은 이벤트로 같은 뷰가 움직인다.
    할 일: TCP 수신 스레드 → ConcurrentQueue → 메인스레드에서 0.5초 플레이아웃 버퍼 후 VehicleAgent 필드에 적용. 위치는 손대지 않은 EvaluatePosition이 그린다.
    완료 기준: Replay 소스로 차량 궤적이 내장 Sim과 육안·수치로 동일(처리량·가동률 MAPE ≈ 0).
    막히면: FabSim-P8-Complete/Assets/Scripts/Twin/TwinFeedClient.cs.
  4. M4 — 고장 주입 재현
    목표: 끊기고 유실돼도 화면이 스스로 복구된다.
    할 일: 서버를 --cut/--drop으로 흔들고, 수신 지연·n 갭을 HUD에 표시, SNAPSHOT 도착 시 전체 덮어쓰기로 재동기화.
    완료 기준: 서버를 껐다 켜도 다음 SNAPSHOT에서 화면이 실물과 다시 맞는다 — "sync 버그"를 스스로 만들고 스스로 고쳐 본 것이 핵심.
  5. M5 — UI Toolkit 대시보드 패널 1개 추가
    목표: OnGUI HUD를 UI Toolkit 패널로 한 조각 대체한다.
    할 일: 최소 KPI 패널(Dashboard.uxml/.uss/DashboardController.cs, BEM 클래스)을 만들어 처리량·가동률을 schedule.Execute(...).Every(250)로 바인딩.
    완료 기준: 화면에 UI Toolkit KPI 패널이 뜨고 지표가 0.25초 주기로 갱신된다.
DIGITAL TWIN 관점

완성본과 내 구현의 차이가 곧 학습의 증거다. 소스 어댑터를 직접 붙여 본 사람만 "데이터 소스를 하나 더 붙여 주세요"라는 첫 과제 앞에서 손이 움직인다. 프로토콜을 어떻게 정했는지, 버퍼를 몇 초로 뒀는지 — 그 결정을 말할 수 있는 코드가 포트폴리오가 된다.

핵심 확인 질문

M2 리팩터의 완료 기준이 "동작이 안 바뀐다"인 이유는?

M2는 회귀 없는 리팩터다 — 소스를 인터페이스 뒤로 옮기되 내장 Sim의 동작은 그대로여야 한다. 동작이 바뀌면 리팩터가 아니라 기능 변경이다. "똑같이 동작한다"가 곧 "뷰가 소스를 모르게 잘 분리됐다"의 증거다.

M3에서 플레이아웃 버퍼 0.5초를 두는 이유는?

네트워크 지연·지터를 흡수해 매끄럽게 재생하기 위해서다. 도착하자마자 그리면 흔들림이 그대로 보인다. 0.5초 늦게 재생하면 그 창 안의 변동이 평탄해진다 — 게임의 스냅샷 보간과 같은 트레이드오프다.

M4에서 재동기화에 델타가 아니라 SNAPSHOT이 필요한 이유는?

갭 이후의 델타는 잃어버린 이전 상태를 기준으로 삼아 어긋난다. 자기 완결적인 SNAPSHOT만이 이전 맥락 없이 전체 상태를 덮어써 복원한다 — 그래서 복구·재접속의 공통 해법이 스냅샷이다.

EvaluatePosition을 고치지 않고도 소스를 바꿀 수 있는 구조적 이유는?

EvaluatePosition은 VehicleAgent의 엣지 시각 필드로만 위치를 보간한다. 어느 소스든 그 필드를 채워 주기만 하면 되므로, 소스가 내장 Sim이든 스트림이든 보간 코드는 손댈 필요가 없다. 프로토콜을 그 필드에 1:1로 맞춘 것이 이 구조의 핵심이다.

이 실습이 신입 첫 과제(소스 어댑터·sync 버그)와 대응되는 지점은?

M2~M3(새 소스를 인터페이스로 붙이기)이 "데이터 소스 어댑터" 과제, M4(끊김·유실 복구)가 "화면이 실물과 안 맞는 sync 버그" 과제에 각각 대응한다. 통제된 실험실에서 미리 겪어 두면 현장에서 같은 문제를 만났을 때 손이 움직인다.
