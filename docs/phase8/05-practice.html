PHASE 8 — 디지털 트윈 연결

Phase 8 · Station 05 · 실습

실습 — 내 시뮬레이터를 트윈으로

01~04에서 완성본으로 "두 소스가 한 뷰를 구동하는" 구조를 읽었다. 이번엔 그 구조를 내 FabSim에 직접 붙인다. 다섯 마일스톤을 지나면, 내가 만든 시뮬레이터가 외부 스트림으로도 돌아가는 미니 트윈이 된다.

시작 상태 — 어디서 출발하나

  • 무대: 사용자의 FabSim(미니 팹 시뮬레이터: FabModel· VehicleFleetView·StatsHud가 도는 상태). 폴백: FabSim-P5-Complete. 트윈 브리지는 교통 제어(P7)와 독립이라 P5 상태에서도 성립한다.
  • 콘솔 실습: examples/twin-bridge-practice 스켈레톤(직렬화·페이싱이 TODO), 정답지 examples/twin-bridge.

마일스톤 — 목표 · 할 일 · 완료 기준

  1. M1 — 레코더·리플레이 서버 스켈레톤 채우기(examples/twin-bridge-practice)
    목표: 헤드리스 레코더가 NDJSON을 굽고, 서버가 타임스탬프 간격대로 송출한다.
    할 일: EventRecorder의 각 이벤트 직렬화(필드 추가)와 NdjsonPacer.ComputeDelay((curT−prevT)/speed) TODO를 채운다.
    완료 기준: 내 로그가 정답지 sample.ndjson과 구조 일치(HELLO·주기 SNAPSHOT·단조 n·델타 필드 완비), 서버가 페이싱대로 라인을 흘린다.
  2. M2 — FabSim에 IFleetSource 분리 리팩터링
    목표: 뷰·HUD가 FabSimulation이 아니라 IFleetSource를 읽는다.
    할 일: IFleetSource 선언, SimFleetSource로 FabModel을 감싸 구현, 두 뷰의 Initialize가 소스를 받도록 수정.
    완료 기준: 리팩터 후에도 내장 Sim으로 이전과 똑같이 동작(회귀 없음). 이것이 "뷰가 소스를 모른다"의 첫 증거다.
    막히면: FabSim-P8-Complete/Assets/Scripts/Twin/IFleetSource.cs·SimFleetSource.cs.
  3. M3 — TwinFeedClient 수신·버퍼·보간 재생
    목표: 리플레이 서버에 붙어, 받은 이벤트로 같은 뷰가 움직인다.
    할 일: TCP 수신 스레드 → ConcurrentQueue → 메인스레드에서 0.5초 플레이아웃 버퍼 후 VehicleAgent 필드에 적용. 위치는 손대지 않은 EvaluatePosition이 그린다.
    완료 기준: Replay 소스로 차량 궤적이 내장 Sim과 육안·수치로 동일(처리량·가동률 MAPE ≈ 0).
    막히면: FabSim-P8-Complete/Assets/Scripts/Twin/TwinFeedClient.cs.
  4. M4 — 고장 주입 재현
    목표: 끊기고 유실돼도 화면이 스스로 복구된다.
    할 일: 서버를 --cut/--drop으로 흔들고, 수신 지연·n 갭을 HUD에 표시, SNAPSHOT 도착 시 전체 덮어쓰기로 재동기화.
    완료 기준: 서버를 껐다 켜도 다음 SNAPSHOT에서 화면이 실물과 다시 맞는다 — "sync 버그"를 스스로 만들고 스스로 고쳐 본 것이 핵심.
  5. M5 — UI Toolkit 대시보드 패널 1개 추가
    목표: OnGUI HUD를 UI Toolkit 패널로 한 조각 대체한다.
    할 일: 최소 KPI 패널(Dashboard.uxml/.uss/DashboardController.cs, BEM 클래스)을 만들어 처리량·가동률을 schedule.Execute(...).Every(250)로 바인딩.
    완료 기준: 화면에 UI Toolkit KPI 패널이 뜨고 지표가 0.25초 주기로 갱신된다.

과정 상세 — 클릭 단위로 따라 만들기

위 다섯 마일스톤은 무엇을 하는지의 지도다. 여기서는 어떻게 하는지를 파일·클래스·클릭 단위로 편다. 먼저 스스로 붙여 보고, 막히거나 결과가 어긋날 때 이 절차를 펼쳐 대조하라 — 인터페이스·필드·값은 완성본 FabSim-P8-Complete에서 그대로 뽑은 재현값이라, 따라 하면 정답 구조가 다시 만들어진다. 콘솔 M1 (examples/twin-bridge-practice)은 그대로 둔다 — 직렬화·페이싱 TODO는 그 스켈레톤에서 채운다. 여기서는 Unity 쪽 M2~M5만 편다.

전제: 새 스크립트는 두 폴더로 나뉜다. 소스 추상화(IFleetSource·SimFleetSource)는 Assets/Scripts/Sim/(네임스페이스 FabSim.Sim), 트윈 배선 (FleetSourceHost·TwinFeedClient·TwinReplaySource)은 Assets/Scripts/Twin/(네임스페이스 FabSim.Twin)에 둔다.

M2 — IFleetSource로 소스 분리 (뷰·HUD가 소스를 모르게)

지금 VehicleFleetView·StatsHud는 simulation.Model을 직접 읽는다. 이 결합을 인터페이스 하나로 끊는다. 순서는 (1) 인터페이스 선언 → (2) 내장 Sim을 감싼 구현 → (3) 소스를 소유·구동하는 호스트 → (4) 뷰·HUD가 인터페이스만 읽도록 치환, 넷이다.

  1. Assets/Scripts/Sim/IFleetSource.cs를 만들고 뷰가 실제로 읽는 것만 표면으로 노출한다 — 그래프·비히클 목록·현재 시각·완료/대기 수·세 지표 메서드·Tick(dt). 멤버는 아래 멤버 표 그대로.
  2. Assets/Scripts/Sim/SimFleetSource.cs(sealed)로 기존 FabModel을 감싼다. 모든 getter는 model에 위임하고, Now는 model.Simulation.Now, Tick(dt)는 model.Simulation.Run(model.Simulation.Now + dt * TimeScale) — Phase 5가 FabSimulation.Update에 넣었던 "실시간 × 배속" 브리지가 여기로 옮겨 온다.
  3. Assets/Scripts/Twin/FleetSourceHost.cs(MonoBehaviour)를 만든다. 이 호스트가 Source를 소유하고 Update에서 매 프레임 Source.Tick(Time.deltaTime)한다 — FabSimulation이 하던 클록 구동 역할을 이 호스트가 넘겨받는다. BuildSource()는 FabLayout.Build(out portNodes) → new FabModel(...) → model.Start() → new SimFleetSource(model, timeScale) 순이다.
  4. VehicleFleetView.Initialize·StatsHud.Initialize의 인자를 FabSimulation에서 IFleetSource로 바꾸고, 본문의 simulation.Model.X 읽기를 전부 source.X로 치환한다(치환표 아래). 씬에서는 뷰·HUD의 Initialize에 host.Source를 넘긴다.

함정 — 인터페이스는 딱 뷰가 읽는 것만

IFleetSource에 Model이나 Segments(P7 교통)를 얹지 마라 — 그 순간 "뷰가 소스를 모른다"가 깨진다. P7까지 온 FabSim이라면 대기·데드락 색칠이 Segments를 쓰므로 인터페이스 밖이다: 그 색칠은 Sim 전용으로 남기거나 뺀다. 배속 표시(simulation.TimeScale)도 인터페이스에 없으니 호스트에서 읽어라. 완료 기준은 회귀 없음 — 내장 Sim으로 리팩터 전과 육안·수치가 같아야 한다.

M3 — TwinFeedClient·TwinReplaySource로 스트림 재생

같은 인터페이스를 두 번째 구현이 채운다. 헤드리스 재생 코어(TwinReplaySource, UnityEngine 무의존 — 그래서 테스트된다)와, 그것을 TCP로 먹여 살리는 MonoBehaviour (TwinFeedClient)로 나눈다. 위치를 그리는 EvaluatePosition은 손대지 않는다 — 스트림이 엣지 시각 필드를 채우면 보간 코드는 소스가 바뀌어도 그대로다.

  1. Assets/Scripts/Twin/TwinReplaySource.cs(sealed, IFleetSource). 생성자는 로그 라인들 + playoutDelay를 받아 첫 HELLO로 vehicleCount·nodeCount를 읽고(로컬 그래프와 대조), 비히클 목록을 세운다. AppendLine(line)은 라이브 라인을 (t, line)으로 버퍼에 쌓고, Tick(dt)는 now += dt 뒤 지평선 now − playoutDelay를 지난 이벤트만 Apply한다. Apply는 type으로 분기해 아래 매핑대로 VehicleAgent 필드를 채우고, SNAPSHOT이면 전체를 덮어쓴다.
  2. Assets/Scripts/Twin/TwinFeedClient.cs(MonoBehaviour, IFleetSource). 직렬화 필드 host 127.0.0.1 · port 5088 · playoutDelay 0.5. Connect()가 백그라운드 수신 스레드를 띄우고, 소켓에서 읽은 라인을 ConcurrentQueue에 넣는다. Tick(dt)는 메인 스레드에서만 큐를 비워 — 첫 HELLO 라인으로 TwinReplaySource를 만들고, 이후 라인은 source.AppendLine에 넘긴 뒤 — source.Tick(dt)한다. 모든 IFleetSource getter는 내부 source에 위임한다.
  3. FleetSourceHost에 [SerializeField] TwinFeedClient twinClient를 두고, BuildSource()에서 kind == Twin이면 twinClient.Connect() 후 twinClient를 반환한다. 씬에서 호스트의 kind를 Twin으로 두고, twinClient 슬롯에 같은(또는 다른) 오브젝트의 TwinFeedClient 컴포넌트를 드래그해 연결한다.
  4. 콘솔에서 리플레이 서버를 띄운다: dotnet run --project examples/twin-bridge/ReplayServer -c Release -- --file examples/twin-bridge/sample.ndjson --port 5088 --speed 30. 그다음 Unity에서 Play — 호스트가 Twin이면 HELLO가 즉시 오고 뷰·HUD가 스트림으로 돈다.

정답 값 — 수신·재생

TwinFeedClient 기본값: host 127.0.0.1 · port 5088 · playoutDelay 0.5초. 스레드 경계는 락 프리 큐 하나로 마샬링한다 — 수신·파싱은 백그라운드, 상태 적용은 메인 스레드(Tick). TwinReplaySource.Tick은 now − playoutDelay 지평선을 지난 이벤트만 적용하고, 처리량·가동률을 FabModel과 같은 식(completed / (now/3600), busy / (now × 대수))으로 계산한다 — 그래서 Replay 궤적이 내장 Sim과 MAPE ≈ 0이다.

M4 — 고장 주입: 끊김·유실을 스스로 복구

이제 스트림을 일부러 흔든다. 서버의 두 플래그가 현장의 두 고장을 재현한다 — 연결이 죽고(서버 다운), 라인이 새는(패킷 유실) 상황. 화면이 어긋나고, 다음 SNAPSHOT에서 스스로 맞는 것을 확인한다.

  1. 끊김: 서버를 --cut 20으로 띄운다(접속 20초 후 연결 강제 종료). Play 중 TwinFeedClient.ReceiveLoop가 예외를 삼키고 Thread.Sleep(500) 뒤 재접속한다 — 자동 복구 경로다. 끊긴 동안 뷰는 마지막 상태로 얼어 있다.
  2. 유실: 서버를 --drop 0.02로 띄운다(델타 라인 2% 확률 스킵). 수신 n에 갭이 생기고, TrackSequence가 n > lastSequence + 1을 세어 DroppedCount에 누적한다.
  3. 드러내기: TwinFeedClient.Connected·DroppedCount를 HUD(또는 M5 패널)에 표시해, 접속 상태와 누적 유실을 눈으로 본다.
  4. 재동기화: SNAPSHOT이 60초 주기로 오고, ApplySnapshot이 완료·대기 수와 각 비히클의 전체 상태를 덮어쓴다. 끊김·유실로 어긋났던 화면이 다음 스냅샷에서 실물과 다시 맞는다.
정답 값 — 고장 주입 · 복구

서버 플래그: --cut <초>(그 시각에 연결 종료) · --drop <확률>(델타 라인을 그 확률로 스킵). 복구 축: TwinFeedClient.ReceiveLoop는 끊기면 500 ms 간격으로 자동 재접속, TrackSequence는 n 갭을 DroppedCount로 집계. SNAPSHOT은 60초 주기이며 ApplySnapshot이 완료/대기 수 + 각 비히클 전체 상태(phase·busy·moving과 이동 중이면 from/to/depart/eta, 정지면 node)를 덮어쓴다.

함정 — 델타로는 못 맞춘다

갭 이후의 델타는 잃어버린 이전 상태를 기준으로 삼아 어긋난다. 자기 완결적인 SNAPSHOT만이 이전 맥락 없이 전체를 덮어써 복원한다 — 그래서 재접속·유실 복구의 공통 해법이 스냅샷이다. "sync 버그"를 스스로 만들고(--cut/--drop) 스스로 고쳐 본 것이 이 마일스톤의 핵심이다.

M5 — UI Toolkit KPI 패널 한 조각

근거 — 완성본에 대시보드 파일은 없다

완성본 FabSim-P8-Complete에는 대시보드 .uxml/.uss/컨트롤러 파일이 들어 있지 않다. 그래서 M5는 사양(BEM 클래스 · Every(250) · 처리량/가동률)만 주어지고, 최소 골격을 직접 짓는다. 아래에서 바인딩 대상 API(host.Source.*)와 갱신 주기만 확정값이고, UXML/USS 레이아웃 값은 자유다.

  1. UIDocument 준비: 빈 GameObject에 UIDocument 컴포넌트를 추가하고, Project에서 Create → UI Toolkit → Panel Settings Asset으로 PanelSettings를 하나 만들어 UIDocument의 Panel Settings 슬롯에 지정한다.
  2. Dashboard.uxml: kpi-panel 블록 안에 처리량·가동률 두 행을 둔다. BEM kebab-case로 kpi-panel__row(행), kpi-panel__label(이름), kpi-panel__value(값 Label)를 쓴다. UIDocument의 Source Asset에 이 UXML을 지정한다.
  3. Dashboard.uss: 패널 배경·여백·값 폰트만 최소로 준다. 임계치 강조는 모디파이어 kpi-panel__value--warn으로 분리한다(예: 가동률이 특정 값을 넘으면 이 클래스를 토글).
  4. DashboardController.cs: UIDocument.rootVisualElement에서 값 Label 두 개를 Q<Label>(...)로 잡고, FleetSourceHost 참조를 받는다. root.schedule.Execute(UpdateKpis).Every(250)로 갱신을 건다. UpdateKpis는 host.Source.Now를 now로 잡아 host.Source.GetThroughputPerHour(now)· host.Source.GetFleetUtilization(now)를 읽어 Label 텍스트를 갱신한다.
재현 기준 — 바인딩 (사양·API)

갱신 주기 250 ms(schedule.Execute(UpdateKpis).Every(250)). 처리량 = host.Source.GetThroughputPerHour(host.Source.Now), 가동률 = host.Source.GetFleetUtilization(host.Source.Now) × 100 %. BEM: 블록 kpi-panel, 엘리먼트 __row/__label/__value, 모디파이어 --warn. 소스가 Sim이든 Twin이든 패널 코드는 host.Source만 읽으므로 그대로 돈다 — M2 분리의 배당금이다.

DIGITAL TWIN 관점

완성본과 내 구현의 차이가 곧 학습의 증거다. 소스 어댑터를 직접 붙여 본 사람만 "데이터 소스를 하나 더 붙여 주세요"라는 첫 과제 앞에서 손이 움직인다. 프로토콜을 어떻게 정했는지, 버퍼를 몇 초로 뒀는지 — 그 결정을 말할 수 있는 코드가 포트폴리오가 된다.

핵심 확인 질문

M2 리팩터의 완료 기준이 "동작이 안 바뀐다"인 이유는?

M2는 회귀 없는 리팩터다 — 소스를 인터페이스 뒤로 옮기되 내장 Sim의 동작은 그대로여야 한다. 동작이 바뀌면 리팩터가 아니라 기능 변경이다. "똑같이 동작한다"가 곧 "뷰가 소스를 모르게 잘 분리됐다"의 증거다.

M3에서 플레이아웃 버퍼 0.5초를 두는 이유는?

네트워크 지연·지터를 흡수해 매끄럽게 재생하기 위해서다. 도착하자마자 그리면 흔들림이 그대로 보인다. 0.5초 늦게 재생하면 그 창 안의 변동이 평탄해진다 — 게임의 스냅샷 보간과 같은 트레이드오프다.

M4에서 재동기화에 델타가 아니라 SNAPSHOT이 필요한 이유는?

갭 이후의 델타는 잃어버린 이전 상태를 기준으로 삼아 어긋난다. 자기 완결적인 SNAPSHOT만이 이전 맥락 없이 전체 상태를 덮어써 복원한다 — 그래서 복구·재접속의 공통 해법이 스냅샷이다.

EvaluatePosition을 고치지 않고도 소스를 바꿀 수 있는 구조적 이유는?

EvaluatePosition은 VehicleAgent의 엣지 시각 필드로만 위치를 보간한다. 어느 소스든 그 필드를 채워 주기만 하면 되므로, 소스가 내장 Sim이든 스트림이든 보간 코드는 손댈 필요가 없다. 프로토콜을 그 필드에 1:1로 맞춘 것이 이 구조의 핵심이다.

이 실습이 신입 첫 과제(소스 어댑터·sync 버그)와 대응되는 지점은?

M2~M3(새 소스를 인터페이스로 붙이기)이 "데이터 소스 어댑터" 과제, M4(끊김·유실 복구)가 "화면이 실물과 안 맞는 sync 버그" 과제에 각각 대응한다. 통제된 실험실에서 미리 겪어 두면 현장에서 같은 문제를 만났을 때 손이 움직인다.
