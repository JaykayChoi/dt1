PHASE 9 — 대규모 성능 심화

Phase 9 · Station 02

메모리 프로파일링과 씬 스트리밍

CPU를 Jobs로 풀어도, 수만 대의 메시·텍스처·씬을 전부 메모리에 올리면 다른 벽에 부딪힌다. 이번에는 메모리의 윤곽을 읽고, "화면에 없는 것은 메모리에도 없게" 만든다.

Memory Profiler — 스냅샷을 찍는다

Memory Profiler 패키지는 실행 중 메모리 스냅샷을 찍어 managed(C# 힙)와 native(엔진), 텍스처·메시별 점유를 뜯어본다. 핵심 기능은 두 스냅샷 diff — "무엇이 늘고 무엇이 안 줄었나"를 본다.

누수 사냥 절차 — 안 줄어든 것이 누수

누수는 "로드 → 언로드 → 다시 스냅샷"에서 안 줄어든 것이다. 대표 원인은 static 참조(등록 후 미삭제), 이벤트 구독 해제 누락 (+=만 하고 -= 빠짐), 살아 있는 핸들. diff에서 retained 객체의 참조 경로를 따라가면 누가 붙잡고 있는지 나온다.

Additive 씬 스트리밍

애디티브 씬은 기존 씬을 유지한 채 위에 겹쳐 로드한다 (SceneManager.LoadSceneAsync(name, LoadSceneMode.Additive) / UnloadSceneAsync). 큰 맵을 구역으로 나눠 카메라 근처만 올린다. 라이팅 주의: additive 씬은 각자의 라이트맵·라이팅 설정을 들고 오므로, 베이크한 씬을 올릴 때 라이트맵 참조가 유지되는지, 언로드 시 함께 내려가는지 확인해야 한다.

Addressables 기본

Addressables는 자산에 주소(키)를 붙여 빌드에서 분리하고, 비동기로 LoadAssetAsync/Release하며 참조 카운트로 수명을 관리한다(0이 되면 해제). 여기서는 "지도"까지 — 랩은 씬 단위 스트리밍을 주로 실습한다.

핵심 확인 질문

Memory Profiler에서 "누수"는 어떤 절차로 판별하는가?

"로드 전 → 로드 후 → 언로드 후" 세 스냅샷을 찍고, "로드 후 → 언로드 후" diff에서 안 줄어든 것을 찾는다. 언로드했는데 여전히 retained면 누군가 붙잡고 있는 것 — 그 객체의 참조 경로를 따라가 범인을 특정한다.

managed 메모리와 native 메모리의 차이, 텍스처는 어느 쪽인가?

managed는 C# GC가 관리하는 힙(객체·배열), native는 엔진이 직접 잡는 메모리다. 텍스처·메시·오디오 같은 자산은 native 쪽에 있다 — GC로는 안 줄어들고 자산을 언로드해야 내려간다.

Additive 씬을 언로드했는데 메모리가 안 줄었다 — 흔한 원인 두 가지는?

① static 컬렉션·필드에 그 씬의 객체를 등록해 두고 삭제하지 않음. ② 이벤트를 +=로 구독만 하고 -=로 해제하지 않아 참조가 남음. 둘 다 GC가 회수하지 못하게 붙잡는다.

Addressables의 참조 카운트가 하는 일과, Release를 빼먹으면?

LoadAssetAsync 때 카운트가 오르고 Release 때 내려가, 0이 되면 자산을 메모리에서 내린다. Release를 빼먹으면 카운트가 0이 안 돼 자산이 영영 상주한다 — 전형적인 누수다.

베이크한 씬을 additive로 올릴 때 라이팅에서 주의할 점은?

additive 씬은 자기 라이트맵·라이팅 설정을 들고 오므로, 라이트맵 참조가 로드 시 제대로 연결되는지, 언로드 시 함께 내려가는지 확인해야 한다. 안 그러면 조명이 어긋나거나 라이트맵 메모리가 남는다.
