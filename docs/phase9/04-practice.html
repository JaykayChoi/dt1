측정 함정 회피

VSync를 끈다(안 그러면 프레임이 주사율에 캡되어 CPU 개선이 안 보인다). Burst는 첫 실행에 컴파일 스톨이 있으니 워밍업 프레임을 버린다. FPS가 아니라 Profiler의 Move 마커 ms로 읽는다. 가능하면 Development Build로, 에디터 값이면 "에디터"라 명시하고 방식 간 상대 비율로 결론낸다.

과정 상세 — 클릭 단위로 따라 만들기

위 마일스톤은 무엇을 하는지의 지도다. 여기서는 어떻게 하는지를 코드·클릭 단위로 편다. 먼저 스스로 스켈레톤의 TODO를 채우고 측정해 본 뒤, 막히거나 수치가 어긋날 때 이 절차를 펼쳐 대조하라 — 값과 코드는 완성본 FabSim-P9-Complete에서 그대로 뽑은 재현값이라, 따라 하면 정답 경로와 측정표가 다시 만들어진다. 근거 파일은 넷이다: Assets/Scripts/JobifiedVehicleFleet.cs(잡 경로), Assets/Scripts/TrackMath.cs(순수 수학), Assets/Scripts/VehicleFleet.cs(매니저 루프 원본), Assets/Editor/StressSceneBuilder.cs(씬·대수).

설계의 핵심 한 가지를 먼저 잡아 두면 측정이 단순해진다 — 세 방식(매니저/Job/Job+Burst)을 컴포넌트 하나로 잰다. JobifiedVehicleFleet.Update의 Move 마커는 잡 분기와 매니저 분기를 둘 다 감싸므로, useJobs 토글 하나로 같은 마커 아래 매니저 루프와 잡 경로를 번갈아 읽을 수 있다. useJobs=false 분기는 VehicleFleet.Update와 동일한 메인 스레드 for-루프다(수식이 같아 위치도 동일).

0단계 — 무대와 컴포넌트 확인

  1. 스트레스 씬(Assets/Scenes/StressField.unity)을 연다. Vehicles (Dynamic) 아래에 순환 주행 큐브가 붙어 있고, 이 오브젝트가 이동 컴포넌트를 지닌다.
  2. 완성본 씬은 기본으로 VehicleFleet(매니저 루프)를 얹는다. 세 방식을 한 마커로 재려면 이동 컴포넌트를 JobifiedVehicleFleet로 바꾼다 — 두 컴포넌트의 Initialize(Transform[], float, float) 시그니처가 같아, 씬 빌더의 CreateVehicles()에서 AddComponent<VehicleFleet>()를 AddComponent<JobifiedVehicleFleet>()로 한 줄만 바꾸면 배열·트랙값이 그대로 주입된다.
  3. 다시 빌드하면(4~6단계 참고) Vehicles (Dynamic)에 채워진 JobifiedVehicleFleet가 준비된다. 인스펙터에서 Use Jobs 체크가 보이면 정상이다.
정답 값 — 씬 트랙 파라미터 (씬 빌더가 Initialize로 주입)

기본 대수 96 · halfWidth 22.5 · halfLength 20 · perimeter = 4·(22.5+20) = 170 · 비히클 트랙 높이 y 3.6 · speed 6 m/s. (컴포넌트 필드 기본값 halfWidth 20·halfLength 30은 Initialize가 덮어쓴다 — Initialize 없이 인스펙터로 손수 얹으면 트랙이 어긋난다.)

1단계 — MoveJob.Execute: 트랙 수식 채우기

스켈레톤의 Execute는 인덱스 하나의 트랜스폼을 순환 트랙 위 한 점으로 옮긴다. 둘레 거리 distance를 Repeat로 접고(0~둘레), EvaluateTrack로 (x, y, z)를 얻어 localPosition에 넣는다. 매니저 분기와 완전히 같은 두 호출이라, 잡이 매니저와 동일한 궤도를 낸다(정확성 회귀 방지). 

public void Execute(int index, TransformAccess transform)
{
    float distance = TrackMath.Repeat(Travel + Offsets[index], Perimeter);
    TrackMath.EvaluateTrack(
        distance, transform.localPosition.y, HalfWidth, HalfLength,
        out float x, out float y, out float z);
    transform.localPosition = new Vector3(x, y, z);
}

2단계 — TrackMath: Repeat와 EvaluateTrack (순수 수학)

잡 본문이 부르는 두 함수는 UnityEngine 무의존 순수 수학이라 Burst 잡에서도 헤드리스 테스트에서도 같은 코드로 돈다. Repeat는 양수 나머지(0 이상 length 미만)를 낸다: 

public static float Repeat(float value, float length)
{
    float wrapped = value - Floor(value / length) * length;
    return wrapped < 0f ? wrapped + length : wrapped;
}

EvaluateTrack은 직사각 순환 트랙의 둘레 거리를 네 변으로 나눠 좌표를 낸다. 변의 길이는 w = halfWidth·2(가로변), l = halfLength·2(세로변)이고, 순서는 왼변 → 위변 → 오른변 → 아래변이다. y는 입력 height 그대로.

구간 변 (길이) 조건 (누적 소거 후 distance) x z
1왼변 (l)distance < l−halfWidth−halfLength + distance
2위변 (w)distance −= l; distance < w−halfWidth + distance+halfLength
3오른변 (l)distance −= w; distance < l+halfWidthhalfLength − distance
4아래변 (w)distance −= l; 나머지halfWidth − distance−halfLength

3단계 — Update·Start·OnDestroy: 잡 스케줄과 네이티브 배열

Update의 TODO는 MoveJob을 구성해 TransformAccessArray에 대해 스케줄하고 완료를 기다린다. Schedule(transformAccess)가 IJobParallelForTransform을 트랜스폼 배열에 걸어 워커 스레드로 쪼개고, Complete()로 이 프레임 안에 합류시킨다. 

var job = new MoveJob
{
    Offsets = offsets,
    Travel = Time.time * speed,
    Perimeter = perimeter,
    HalfWidth = halfWidth,
    HalfLength = halfLength,
};
JobHandle handle = job.Schedule(transformAccess);
handle.Complete();

잡이 읽을 배열은 Start에서 한 번 할당하고, 컴포넌트가 사라질 때 반드시 해제한다. offsets는 비히클을 둘레에 고르게 흩는 시작 오프셋이다. 

// Start()
perimeter = 4f * (halfWidth + halfLength);
offsets = new NativeArray<float>(vehicles.Length, Allocator.Persistent);
for (int i = 0; i < vehicles.Length; i++)
{
    offsets[i] = perimeter / vehicles.Length * i;
}
transformAccess = new TransformAccessArray(vehicles);

// OnDestroy()
if (offsets.IsCreated)
{
    offsets.Dispose();
}
if (transformAccess.isCreated)
{
    transformAccess.Dispose();
}
잡 함정 셋

① Offsets에 [ReadOnly]를 붙여야 여러 워커가 안전하게 공유한다 — 빼면 잡이 병렬 스케줄을 거부한다. ② Allocator.Persistent로 할당했으면 OnDestroy에서 Dispose()가 짝을 이뤄야 한다. 안 하면 네이티브 메모리가 새고 에디터가 미해제 경고를 띄운다. ③ Complete()를 잊으면 트랜스폼을 아직 잡이 쓰는 중에 메인 스레드가 건드려 안전 시스템이 예외를 던진다.

4단계 — M1 Baseline 측정 (VSync off · Profiler)

  1. Edit → Project Settings → Quality에서 현재 레벨의 VSync Count를 Don't Sync로 둔다. (안 그러면 프레임이 주사율에 캡되어 CPU 개선이 화면에 안 보인다.)
  2. Window → Analysis → Profiler(Ctrl+7)를 열고 CPU Usage 모듈을 켠다.
  3. 인스펙터에서 JobifiedVehicleFleet의 Use Jobs 체크를 끈다(매니저 루프).
  4. Play. Profiler를 잠시 흐르게 두고 일시정지한 뒤, CPU 프레임을 하나 선택한다. Hierarchy 뷰 검색창에 JobifiedVehicleFleet.Move를 넣어 그 마커의 Time ms(Self/Total)를 읽는다. FPS가 아니라 이 마커 ms가 측정값이다.
  5. 측정표 매니저 루프 열의 현재 대수(기본 96 → 스윕 전이라면 500 행부터는 6단계) 칸에 적는다. GC Alloc 열이 0에 가까운지도 함께 본다 — 이 루프엔 문자열·박싱·새 배열이 없다.

5단계 — M2·M3: Job과 Burst 측정

  1. Job (Burst off): 먼저 Jobs → Burst → Enable Compilation 체크를 끈다. 인스펙터에서 Use Jobs를 켠다. Play 후 Profiler Timeline 뷰를 보면 Move 구간이 여러 Worker 스레드로 분산된 게 보인다. 같은 JobifiedVehicleFleet.Move 마커 ms를 Job 열에 적는다.
  2. Job+Burst: Jobs → Burst → Enable Compilation을 다시 켠다. Burst는 첫 실행에 컴파일 스톨이 있으니 처음 몇 프레임을 버리고(워밍업) 안정된 뒤 읽는다. 같은 마커 ms를 Job+Burst 열에 적는다.
  3. SIMD를 눈으로 확인하려면 Jobs → Burst → Open Inspector…에서 MoveJob을 골라 어셈블리에 벡터 명령이 깔린 걸 본다.
정답 값 — 세 경로를 가르는 스위치

매니저 루프 = Use Jobs 끔 · Job = Use Jobs 켬 + Burst Enable Compilation 끔 · Job+Burst = Use Jobs 켬 + Burst Enable Compilation 켬. 세 값 모두 같은 JobifiedVehicleFleet.Move 마커에서 읽는다. 대수가 커질수록 Job<매니저, Job+Burst<Job으로 ms가 급감한다 — 절대값보다 방식 간 상대 비율로 결론낸다.

6단계 — M4: 대수 스윕으로 측정표 채우기

완성본은 대수를 StressSceneBuilder의 상수 private const int VehicleCount = 96;로 정한다(환경변수로 받지 않는다). 측정표의 각 행은 이 값을 500 · 2,000 · 5,000 · 10,000으로 바꿔 씬을 다시 빌드해 얻는다.

  1. StressSceneBuilder.cs의 VehicleCount를 목표 대수로 바꾼다.
  2. 빌더를 다시 실행한다. 완성본에는 메뉴 항목이 없으니, SetupAndBuild 위에 [MenuItem("FabSim/Rebuild Stress Scene")] 한 줄을 붙여 에디터 메뉴에서 클릭하거나, 배치 모드 Unity -batchmode -quit -projectPath <프로젝트> -executeMethod FabSim.EditorTools.StressSceneBuilder.SetupAndBuild로 재생성한다. (빌더가 라이트맵을 해상도 8로 다시 굽지만 금방 끝난다.)
  3. 0단계의 컴포넌트 스왑(JobifiedVehicleFleet)을 CreateVehicles()에 넣어 뒀다면 재빌드 때마다 유지된다. 새 씬에서 4·5단계를 반복해 한 행(매니저/Job/Job+Burst 3칸)을 채운다.
  4. 네 대수 × 세 방식 = 12칸을 채우고, 어느 대수에서 매니저 루프가 프레임 예산을 넘는지를 한 줄로 적는다. 500대에서는 Job이 매니저보다 느릴 수 있다 — 일이 적으면 스케줄·동기화 오버헤드가 계산 절감을 웃돌아 역전된다(핵심 확인 질문 참고).
측정 신뢰도

에디터에서 잰 값이면 표에 "에디터"라고 명시한다 — 에디터는 Deep Profile·도메인 오버헤드가 껴 실측보다 무겁다. 가능하면 Development Build로 재서 방식 간 상대 비율을 확정한다. 대수를 바꿔도 카메라 구도·베이크 조건은 그대로 둬야 방식만의 차이가 남는다.

7단계 — (선택) M5 스트리밍

완성본에는 구역 스트리밍 참조 구현(ZoneStreamer 등)이 없다 — 이 마일스톤은 정답지 없이 직접 설계하는 확장 과제다. 골자는 SceneManager.LoadSceneAsync(..., LoadSceneMode.Additive)와 UnloadSceneAsync로 구역을 얹고 내리고, Window → Analysis → Memory Profiler로 언로드 전·후 스냅샷을 diff해 메모리 회수를 확인하는 것이다. 값은 스스로 잰 것을 기록하고, 비교 대상이 없으므로 "완성본에서 확인"이 아니라 자기 측정으로 남긴다.

핵심 확인 질문

Baseline을 먼저 측정하지 않고 잡화부터 하면 무엇을 말할 수 없게 되는가?

"얼마나 좋아졌는가"를 말할 수 없다. 개선은 항상 기준 대비다 — Baseline이 없으면 잡화 후 수치가 좋은지 나쁜지 판단할 기준이 없다. Phase 4의 "측정 → 병목 → 처방"에서 측정이 먼저인 이유다.

잡화 후 차량 궤도가 기존과 같은지 확인해야 하는 이유는?

정확성 회귀를 막기 위해서다. 빨라졌어도 궤도가 틀리면 최적화가 아니라 버그다. float3로 옮긴 EvaluateTrack이 원본과 같은 위치를 내야 공정한 비교이자 올바른 구현이다.

500대에서 Job이 매니저 루프보다 느릴 수 있는 이유는?

잡의 스케줄·동기화 오버헤드 때문이다. 일이 적으면 워커에 나눠 주고 모으는 비용이 계산 절감보다 커서 역전된다. 이 역전 지점을 표로 드러내는 것이 실습의 핵심 관찰이다 — 최적화는 규모에 따라 다르다.

VSync를 켠 채 측정하면 왜 CPU 개선이 안 보이는가?

VSync는 프레임을 모니터 주사율(예: 60Hz)에 맞춰 캡한다. CPU가 더 빨라져도 프레임레이트는 60에 묶여 "여유"가 화면에 안 드러난다. Profiler의 실제 Move ms는 줄지만 FPS는 그대로라 착시가 생긴다 — 측정 시 VSync를 끈다.

NativeArray/TransformAccessArray를 Dispose하지 않으면?

네이티브 메모리가 해제되지 않아 누수가 나고, 에디터가 미해제 경고를 띄운다. 할당한 쪽(컴포넌트)이 OnDestroy에서 Dispose() 책임을 진다.
