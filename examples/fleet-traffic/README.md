# Fleet Traffic — 교통 제어와 데드락 실험 랩

Phase 5의 미니 팹은 차량이 **서로를 통과**한다. 이 랩은 처음으로 "한 구간엔 한 대"를
강제하고(**구간 점유**, zone control), 그 대가로 태어나는 **정체**와 **데드락**을 본다.
엔진은 Phase 2의 **DesCore**를, 통계 보고는 Phase 6의 방식(20회 반복, 평균 ± 95% CI)을
공용 헬퍼 `SimStats`로 재사용한다.

## 구조

```
fleet-traffic/
├── FleetTraffic.csproj    DesCore + SimStats 를 ProjectReference 로 참조
├── RailGraph.cs           순수 C# 그래프(FabSim.Sim.RailGraph 동일 스펙 복제) + A*/비용주입 다익스트라
├── SegmentController.cs    엣지를 용량 1 구간으로 — Acquire/Release + wait-for 사이클 탐지
├── TrafficModel.cs        단일 방향 루프 위 hold-and-wait 이동 시뮬레이션
└── Program.cs             실험 A(교통 제어 유/무 × 차량 수) · 실험 B(데드락 유발·탐지·해소)
```

## 실행

```
dotnet run --project examples/fleet-traffic -c Release
```

## 핵심 개념

- **구간 점유(hold-and-wait)** — 차량은 현재 구간을 쥔 채 다음 구간을 예약(`Acquire`)하고,
  획득되면 이동, 도착 노드에서 직전 구간을 반납(`Release`)한다. 앞 구간이 막히면 경계에서
  정지(**폐색**). 철도 폐색과 같은 원리.
- **wait-for 그래프** — "대기 차량 → 그 구간을 점유한 차량" 간선을 모은 방향 그래프.
  여기에 **사이클이 있으면 곧 데드락**이다. `SegmentController.DetectDeadlockCycle()`이
  DFS로 사이클을 찾아 낀 차량 id를 돌려준다.

## 출력 (검증된 실행 결과)

```
[교통 제어 ON — 한 구간에 한 대] 3600s × 20회 반복
  차량수   처리량(건/h)          평균반송(s)
     2     107.2 ± 1.7          66.3 ± 1.1
     4     211.8 ± 2.2          66.9 ± 0.8
     8     353.6 ± 4.0          79.6 ± 0.9
    12     234.1 ± 3.4          176.5 ± 2.5   ← 꺾임(처리량 붕괴 + 반송시간 폭증)

[교통 제어 OFF — 차량이 통과] 3600s × 20회 반복
     2     107.2 ± 1.7          66.4 ± 1.0
     4     216.2 ± 3.1          65.5 ± 1.0
     8     411.2 ± 4.1          66.3 ± 0.6
    12     434.9 ± 9.5          66.0 ± 0.6   ← 단조 증가, 반송시간 평평

[데드락 실험] 양방향 corridor(2구간) · 마주 오는 차량 2대
  구성                     데드락 발생   최초 탐지 시각   관련 차량
  양방향(무대책)           예            t=4.0 s          [0, 1]
  단방향화                 아니오        —                —
  자원 순서화              아니오        —                —
```

## 이 표에서 읽어야 할 것

- **밀도-흐름 붕괴** — 교통 제어 ON에서 차량을 2→4→8로 늘리면 처리량이 오르지만(107→212→354),
  **8에서 12로 가면 오히려 234로 꺾인다**. 구간이 막혀 서로를 기다리느라(폐색) 평균 반송시간이
  80초에서 **176초로 폭증**한다. Phase 2 큐잉의 "가동률↑ → 대기 폭증"이 공간에서 재현된 것 —
  교통공학의 밀도-흐름 기본 다이어그램이다.
- **제어의 대가와 값어치** — 교통 제어 OFF는 차량이 서로를 통과하므로 처리량이 435까지 단조
  증가하고 반송시간도 평평하다(비현실적이지만 상한선). ON은 그 상한을 못 내는 대신 **충돌을 막고
  현실을 재현**한다. 트윈의 충실도는 이 "막히고 멈추는" 재현에서 나온다.
- **정체 ≠ 데드락** — 실험 A의 꺾임은 *느려진 것*(정체)이고, 실험 B는 *멈춘 것*(데드락)이다.
  단일 방향 루프는 항상 빈 구간이 하나 이상 있어 순환 대기가 없다 → 아무리 막혀도 데드락은 없다.
  양방향 corridor에서 마주 오는 두 차량이 각자 한 구간을 쥔 채 상대 구간을 기다리면 순환 대기가
  완성돼 t=4.0s에 데드락. **단방향화**(같은 방향으로만)나 **자원 순서화**(구간 id 오름차순으로만
  획득)가 그 순환을 원천 차단한다.

## 실험해 볼 것

- `LoopLen`(구간 수)을 12로 줄이면 같은 차량 수에서 밀도가 높아져 꺾임이 더 일찍 온다.
- `JobIntervalMean`을 늘려(저부하) 병목이 교통인지 명령 공급인지 가른다.
- `SegmentController.DetectDeadlockCycle()`을 매 사건마다 호출해, 정체 중에도 사이클이 없음을
  (단방향 루프의 데드락 부재를) 직접 확인한다.
