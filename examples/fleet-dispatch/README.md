# Fleet Dispatch — 배차 정책 실험 랩

Phase 2에서 만든 **DesCore** 엔진을 그대로 재사용해, 팹 Fleet Management의 핵심
질문 하나만 파고든다 — **"어느 반송차에 어떤 반송 명령을 줄 것인가?"**

같은 반송 명령 스트림(도착 시각·픽업/드롭 위치가 동일)을 세 배차 정책에 똑같이
흘려보내고, 부하를 세 단계로 스윕하며 처리량·대기·공차주행·가동률을 비교한다.
엔진(클록·이벤트 루프·난수)은 Phase 2 코드를 링크로 참조하고, 이 프로젝트는
그 위에 **반송차·명령·배차 규칙**이라는 응용 계층만 얹는다.

## 구조

```
fleet-dispatch/
├── FleetDispatch.csproj   ../phase2-descore/DesCore 를 ProjectReference 로 참조
└── Program.cs             레이아웃 · 반송차 · 명령 · 세 배차 정책 · 부하 스윕
```

의존성은 DesCore 하나뿐이다(`Simulation`, `SimResource`는 미사용, `Random` 확장만
사용). Phase 2 프로젝트를 지우지 않는 한 추가 설치 없이 그대로 빌드된다.

## 실행

```
dotnet run --project examples/fleet-dispatch -c Release
```

## 세 정책

| 정책 | 규칙 | 성격 |
|------|------|------|
| **랜덤** | 가장 오래된 명령 → 유휴 차량 중 **아무나** | 기준선 |
| **최근접차량** | 가장 오래된 명령 → 픽업에 **가장 가까운** 유휴 차량 | FIFO 공정성 유지, 공차↓ |
| **최근접작업** | 유휴 차량 ↔ 대기 명령 쌍 중 **공차 최소** 조합 | 탐욕적, 큐 재정렬 |

**공차(deadhead)** = 배차받은 차량이 짐 없이 픽업 지점까지 가는 빈 주행. 실제
팹에서 공차는 전력·마모·교통량을 늘리는 순수 손실이라 배차의 1차 최적화 대상이다.

## 출력 (검증된 실행 결과)

각 부하·정책 조합을 시드 교체로 **20회 반복**(공통 난수)하고 평균에 **95% 신뢰구간**을
붙인다. 한 반복 안에서 세 정책은 같은 반송 명령 스트림을 처리하므로, 정책 쌍의
**대응(paired) t-검정**으로 우연을 상쇄한 순수 정책 차이를 판정한다.

```
[여유] 도착간격 평균 1.2분 (offered ≈ 400건/교대) · 20 반복
  정책          평균대기(95% CI)     처리량(95% CI)      평균공차(95% CI)
  ------------------------------------------------------------------
  랜덤          0.75 ± 0.02         50.06 ± 1.48        47.07 ± 0.69
  최근접차량       0.57 ± 0.02         50.09 ± 1.49        35.88 ± 0.85
  최근접작업       0.56 ± 0.02         50.09 ± 1.49        35.30 ± 0.78
  대응 비교(최근접작업 − 최근접차량, 평균대기): Δ=-0.01분, t=-3.62, 95% 유의: 예
  warm-up 효과(최근접작업 평균대기): 제거 전 0.558분 → 제거 후 0.560분

[보통] 도착간격 평균 0.8분 (offered ≈ 600건/교대) · 20 반복
  랜덤          0.98 ± 0.05         75.51 ± 1.58        47.20 ± 0.51
  최근접차량       0.84 ± 0.05         75.56 ± 1.57        40.71 ± 0.73
  최근접작업       0.74 ± 0.02         75.57 ± 1.58        38.24 ± 0.59
  대응 비교(최근접작업 − 최근접차량, 평균대기): Δ=-0.11분, t=-6.09, 95% 유의: 예
  warm-up 효과(최근접작업 평균대기): 제거 전 0.738분 → 제거 후 0.738분

[혼잡] 도착간격 평균 0.5분 (offered ≈ 960건/교대) · 20 반복
  랜덤          8.91 ± 2.46         117.61 ± 1.50       47.02 ± 0.50
  최근접차량       9.02 ± 2.54         117.54 ± 1.54       46.58 ± 0.57
  최근접작업       1.22 ± 0.04         120.13 ± 1.87       33.20 ± 0.67
  대응 비교(최근접작업 − 최근접차량, 평균대기): Δ=-7.80분, t=-6.49, 95% 유의: 예
  warm-up 효과(최근접작업 평균대기): 제거 전 1.222분 → 제거 후 1.225분
```

(중심값·CI 반폭·t는 시드 base 20260709 · 20반복에서 실측한 값이다. 코드를 그대로
돌리면 재현된다.)

## 이 표에서 읽어야 할 것

- **점 하나가 아니라 구간으로 읽는다** — `4.64` 같은 단일 숫자는 시드 하나의 우연일 수
  있다. `8.91 ± 2.46`처럼 구간으로 보면 그 불확실성이 드러난다. 혼잡 부하의 랜덤·최근접차량은
  CI가 넓어(±2.5분) 서로 겹치지만, 최근접작업(1.22 ± 0.04)은 두 정책의 구간에서 한참
  아래로 떨어져 있다 — 우연으로 설명되지 않는 차이다.

- **여유 부하** — 유휴 차량이 많아 고를 여지가 크다. 최근접차량이 공차를 47.1→35.9 m로
  줄이고 평균대기도 함께 내려간다. 대응 비교의 Δ=-0.01분은 **통계적으로는 유의(t=-3.62)하나
  실무적으로는 무시할 크기** — "유의성 ≠ 효과 크기"를 보여 주는 자리다.

- **혼잡 부하** — 유휴 차량이 거의 0~1대뿐이라 **최근접차량은 랜덤과 같아진다**(고를 차가
  없음, CI도 겹침). 대신 대기 명령이 쌓이므로 큐를 재정렬하는 **최근접작업이 평균대기를
  8.9→1.2분으로** 끌어내리고 공차까지 줄여 처리량을 더 낸다. 대응 비교 Δ=-7.80분(t=-6.49)은
  크기도 유의성도 뚜렷하다. 바쁠수록 **어느 차량이 아니라 어느 명령을 먼저 잡느냐**가 승부처.

- **warm-up 제거** — 이 실험은 8시간에 끝나는 **종료형(terminating)** 시뮬레이션이라 초기
  과도 구간이 짧고, 첫 30분을 버려도 평균대기가 거의 안 변한다(1.222→1.225분). 반대로
  DesCore 테스트의 M/M/1은 끝없이 도는 **정상 상태형(steady-state)**이라 앞 5,000명을
  버리지 않으면 편향이 크다 — warm-up 제거의 필요성은 무엇을 재느냐에 달렸다.

## 가중 점수 배차와 P95 (Phase 7)

탐욕적 최근접작업은 공차를 줄이지만 먼 명령을 뒤로 미룰 위험(기아)이 있다. `WeightedScore`
정책은 `score = 공차거리[m] − k × 대기시간[분]`을 최소화해, 오래 기다린 명령의 점수를 낮춰
(우선) 뽑는다(k=0이면 최근접작업으로 환원). 꼬리(최악 근처)를 드러내는 **P95 대기**로
공정성을 잰다 — 평균은 기아를 숨긴다.

```
[과부하] 가중 점수 배차 — 에이징 가중치 k 스윕 · 도착간격 0.42분 · 20 반복
  k(m/min)  완료    처리량    평균대기   P95대기   평균공차
      0      1145    143.1      1.72      4.51     26.4   ← P95 최저
     20      1144    143.0      2.11      5.24     26.9
     40      1140    142.5      4.34      9.46     29.2
     80      1101    137.6     12.99     22.13     34.2
    160      1047    130.9     24.15     41.10     39.2
```

**에이징은 공짜가 아니다** — 교과서는 "에이징이 기아를 없앤다"고 하지만, 이 작은 균형 팹은
최근접작업(k=0)이 이미 꼬리가 짧아(P95 ≈ 4.5분) **고칠 기아가 거의 없다**. 그래서 k를 올리면
먼 옛 명령을 좇느라 공차가 26→39 m로 늘고, 처리량이 143→131건/h로 떨어지며, 큐가 자라
**P95가 오히려 4.5→41분으로 악화**된다. 공정성 손잡이는 실제 기아가 있을 때만 값어치를 한다 —
"측정하기 전엔 켜지 마라"는 Phase 6의 교훈이 배차에서 되풀이된다. (기아가 큰 시스템 — 먼
스토커에 접근이 드문 실제 팹 — 에서는 에이징이 P95를 내린다. 이 랩은 그 전제 조건을 드러낸다.)

## 실험해 볼 것

- `VehicleCount`를 2↔4로 바꿔 병목이 차량인지 배차인지 가른다.
- `Scenarios`의 `ArrivalMean`을 더 촘촘히 스윕해 정책 우열이 뒤집히는 부하를 찾는다.
- 최대대기(공정성)에 상한을 두는 네 번째 정책 — 예: "일정 시간 넘게 기다린 명령은
  최우선" — 을 추가해 탐욕적 정책의 스타베이션을 막아 본다.
