# Fleet Dispatch — 배차 정책 실험 랩

Phase 2에서 만든 **DesCore** 엔진을 그대로 재사용해, 팹 Fleet Management의 핵심
질문 하나만 파고든다 — **"어느 반송차에 어떤 반송 명령을 줄 것인가?"**

같은 반송 명령 스트림(도착 시각·픽업/드롭 위치가 동일)을 세 배차 정책에 똑같이
흘려보내고, 부하를 세 단계로 스윕하며 처리량·대기·공차주행·가동률을 비교한다.
엔진(클록·이벤트 루프·난수)은 Phase 2 코드를 링크로 참조하고, 이 프로젝트는
그 위에 **반송차·명령·배차 규칙**이라는 응용 계층만 얹는다.

## 구조

```
fleet-dispatch/
├── FleetDispatch.csproj   ../phase2-descore/DesCore 를 ProjectReference 로 참조
└── Program.cs             레이아웃 · 반송차 · 명령 · 세 배차 정책 · 부하 스윕
```

의존성은 DesCore 하나뿐이다(`Simulation`, `SimResource`는 미사용, `Random` 확장만
사용). Phase 2 프로젝트를 지우지 않는 한 추가 설치 없이 그대로 빌드된다.

## 실행

```
dotnet run --project examples/fleet-dispatch -c Release
```

## 세 정책

| 정책 | 규칙 | 성격 |
|------|------|------|
| **랜덤** | 가장 오래된 명령 → 유휴 차량 중 **아무나** | 기준선 |
| **최근접차량** | 가장 오래된 명령 → 픽업에 **가장 가까운** 유휴 차량 | FIFO 공정성 유지, 공차↓ |
| **최근접작업** | 유휴 차량 ↔ 대기 명령 쌍 중 **공차 최소** 조합 | 탐욕적, 큐 재정렬 |

**공차(deadhead)** = 배차받은 차량이 짐 없이 픽업 지점까지 가는 빈 주행. 실제
팹에서 공차는 전력·마모·교통량을 늘리는 순수 손실이라 배차의 1차 최적화 대상이다.

## 출력 (검증된 실행 결과)

```
[여유] 도착간격 평균 1.2분 (offered ≈ 400건/교대)
  정책          완료   처리량   평균대기  최대대기  평균공차  가동률
  ------------------------------------------------------------
  랜덤            363     45.4      0.71       2.0      47.6    0.378
  최근접차량         363     45.4      0.49       2.3      33.3    0.327
  최근접작업         363     45.4      0.49       2.3      33.2    0.326

[보통] 도착간격 평균 0.8분 (offered ≈ 600건/교대)
  랜덤            559     69.9      0.89       3.4      49.0    0.588
  최근접차량         559     69.9      0.70       2.4      39.6    0.536
  최근접작업         559     69.9      0.67       2.6      38.8    0.531

[혼잡] 도착간격 평균 0.5분 (offered ≈ 960건/교대)
  랜덤            915    114.4      4.64      13.2      47.0    0.937
  최근접차량         915    114.4      4.99      14.6      47.1    0.937
  최근접작업         924    115.5      1.16       6.8      35.2    0.839
```

## 이 표에서 읽어야 할 것

- **여유 부하(가동률 ~0.38)** — 유휴 차량이 많아 고를 여지가 크다. 최근접차량이
  공차를 47.6→33.3 m(약 −30%)로 줄이고, 그 덕에 가동률·평균대기까지 함께 내려간다.
  "가까운 차를 보내라"만으로 빈 주행이 사라지는 구간.

- **혼잡 부하(가동률 ~0.94)** — 유휴 차량이 거의 0~1대뿐이라 **최근접차량은 랜덤과
  같아진다**(고를 차가 없음). 대신 대기 명령이 쌓이므로, 큐를 재정렬하는 **최근접작업이
  평균대기를 4.64→1.16분으로, 최대대기를 13.2→6.8분으로** 끌어내리고 공차까지 줄여
  처리량을 조금 더 낸다. 바쁠수록 **어느 차량이 아니라 어느 명령을 먼저 잡느냐**가 승부처.

- **공정성 대가** — 최근접작업은 먼 명령을 뒤로 미룰 수 있어(스타베이션) 최대대기가
  튈 위험이 있다. 이 시나리오에선 혼잡이 완화돼 오히려 최대대기가 낮지만, 부하와
  레이아웃에 따라 "효율 vs 공정성" 트레이드오프가 뒤집힐 수 있다. 시드·차량 수·도착
  간격을 바꿔가며 직접 확인해 볼 것.

## 실험해 볼 것

- `VehicleCount`를 2↔4로 바꿔 병목이 차량인지 배차인지 가른다.
- `Scenarios`의 `ArrivalMean`을 더 촘촘히 스윕해 정책 우열이 뒤집히는 부하를 찾는다.
- 최대대기(공정성)에 상한을 두는 네 번째 정책 — 예: "일정 시간 넘게 기다린 명령은
  최우선" — 을 추가해 탐욕적 정책의 스타베이션을 막아 본다.

## 배경 개념

배차·공차·교통제어·데드락은 `docs/phase1/04-fleet-management.html`,

만나는 지점 — DES로 배차 정책을 **측정**하는 자리 — 이다.
